# Spec-Driven Development (SDD)

> コードを書く前に仕様を先に作成し、その仕様を基準として実装・検証・反復を行うソフトウェア開発手法。

---

## 概要

Spec-Driven Development (SDD) は、ドキュメントとコードの従来の関係を逆転させます。従来の開発では、ドキュメントは後付けで作成されることが多く、実装されたものを説明するためのものでした。SDD では、仕様が最初の成果物であり、実装コードが一行も存在する前に作成されます。

SDD における仕様は静的な設計文書ではありません。理解が深まるにつれて進化する「生きた」成果物です。要件がより明確になり、明確化の過程でエッジケースが発見され、実装からのフィードバックが仕様に反映されます。この「生きた」特性により、開発ライフサイクル全体を通じて仕様とコードが同期し、古くなったドキュメントによる危険を排除します。

SDD は、AIが支援する開発環境において特に強力です。AIエージェントが曖昧な自然言語プロンプトの代わりに構造化された機械可読な仕様を受け取ると、出力の精度と一貫性が大幅に向上します。[[Kiro]] と [[Spec-Kit]] はどちらも、この洞察を中心に構築されています。AIエージェントには「望み」ではなく「仕様」を与える、というのがその哲学です。

---

## コア原則

### 1. 仕様優先 (Spec-First)

すべての機能は仕様から始まります。実装を開始する前に、仕様は次の3つの質問に答えなければなりません。何を作るのか？成功はどのように測定されるのか？明示的にスコープ外のものは何か？これらの質問が文書化されてから初めて実装が始まります。

仕様優先の原則は「何を作るか」と「どのように作るか」を分離し、チームが目指すべき安定した目標を設定します。

### 2. 生きた仕様 (Living Specification)

SDD の仕様は固定されません。実装によってギャップが明らかになると、仕様がまず更新され、実装がそれに続きます。この原則により仕様が常に最新の状態に保たれ、コードベースが文書化された意図から乖離することを防ぎます。

### 3. 反復的精練 (Iterative Refinement)

SDD は最初から完璧な仕様を要求しません。この手法は、仕様 → 実装 → 検証 → 仕様更新のサイクルを繰り返すことをサポートします。各イテレーションで仕様がより精確になり、実装がより正確になります。

### 4. 検証可能性 (Verifiability)

SDD 仕様のすべての要件は、検証可能な形式で記述されます。受け入れ基準は測定可能な条件を使用します。「レスポンスが速いこと」ではなく「レスポンスタイムが200ms以内であること」というように記述します。これにより、自動検証と、AIエージェントによる自己検証が可能になります。

## 実際の適用例

以下は、SDD を一般的なAPI機能開発に適用したシナリオです。

```
機能：注文ステータスエンドポイント

[ステップ1 — 仕様作成]
FR-001: システムはGET /orders/{id}/statusエンドポイントを公開する必要がある。
FR-002: エンドポイントはリクエストの95%において150ms以内にレスポンスを返す必要がある。
SC-001: 有効な注文IDはHTTP 200と"status"フィールドを含むJSONを返す。
SC-002: 無効または存在しない注文IDはHTTP 404と"not found"メッセージを返す。
SC-003: 認証されていないリクエストはHTTP 401を返す。

[ステップ2 — 明確化]
Q: 有効なステータス値は何ですか？
A: pending, processing, shipped, delivered, cancelled

[ステップ3 — 計画]
- JWTミドルウェアによる認証付きRESTエンドポイント
- インデックス付きorder_idカラムへのデータベースクエリ
- 30秒TTLのレスポンスキャッシュ

[ステップ4 — 実装]
- 計画に従いエンドポイント、ミドルウェア、キャッシュ層を実装

[ステップ5 — 検証]
- SC-001: PASS (HTTP 200とstatusフィールドを確認)
- SC-002: PASS (存在しないIDにHTTP 404を返すことを確認)
- SC-003: PASS (トークンなしでHTTP 401を返すことを確認)
- FR-002: PASS (p95レイテンシが87msと計測)
```

## 手法の比較

| 比較軸 | SDD | BDD (振る舞い駆動開発) | TDD (テスト駆動開発) | アドホック開発 |
|--------|-----|----------------------|---------------------|--------------|
| 開始点 | 構造化仕様 (spec.md) | ユーザー行動シナリオ (Gherkin) | 失敗するユニットテスト | 口頭要件またはチケット |
| 主な成果物 | 仕様文書 → 実装 → 検証レポート | フィーチャーファイル、ステップ定義、実装 | テストコード、実装コード | 実装コード（ドキュメントは任意） |
| AIエージェント適合性 | 非常に高い — 構造化仕様は自然な入力 | 中程度 — Gherkinの解析が必要 | 低い — テストが主な文脈 | 低い — 曖昧な指示 |
| ドキュメントとコードの同期 | 手法の設計により仕様が常に最新 | フィーチャーファイルが生きたドキュメントとして機能 | テストが唯一のドキュメント | ドキュメントとコードが容易に乖離 |

## 連携ツール & 実装

- [[Kiro]] — AWSのAI搭載IDE。スティアリングファイルと3つのドキュメント仕様システム（requirements.md、design.md、tasks.md）でSDDを実装。オートパイロットモードで仕様全体を自律的に実行します。
- [[Spec-Kit]] — Claude Codeワークフローツール。`specify → clarify → plan → tasks → implement → verify` パイプラインをスラッシュコマンドと拡張フックで実装します。
