# Spec-Driven Development (SDD)

> **Spec-Driven Development (SDD)** とは、コードを書く前に仕様書（specification）を先に作成し、その仕様書を基準として実装・検証・反復を行うソフトウェア開発手法です。

---

## 概要

多くの開発チームは同じように始めます。誰かが「この機能が必要だ」と言うと、開発者はすぐにエディタを開いてコードを書き始めます。ドキュメントは後回し、あるいは省略されます。この方法は短期的には速く感じますが、チームが大きくなりAIコーディングツールを使い始めると、そのコストが大きく膨らみます。

Spec-Driven Development はこの順序を逆にします。コードを一行も書く前に、**何を作るのか**を仕様書という構造化されたドキュメントで定義します。実装は仕様書に従い、その結果が仕様書と一致するかを検証します。AIコーディングエージェント（AIコーディングエージェント — 人間の指示を受けてコードを自動的に作成・修正するAIソフトウェアツール。例：Claude Code、GitHub Copilot）が明確で機械可読な仕様書をもとにコードを生成すると、曖昧な自然言語プロンプトを受け取る場合と比べて、出力の精度と一貫性が大幅に向上します。

この文書では、SDD とは何か、なぜ重要なのか、どのように動作するのか、そして他の開発手法とどう違うのかを、開発者でない読者にもわかるように説明します。

---

## 非エンジニア向けのたとえ話

SDD を理解する最も簡単な方法は、**建築設計図**で考えることです。

建物を建てる前に、建築家は設計図を描きます。部屋の寸法、ドアと窓の位置、電気配線の計画など、詳細が記載されています。施工チームはその設計図をもとに仕事します。現場で問題が起きたときは、まず設計図を修正してから施工が続きます。

設計図なしで建物を建てるチームを想像してみてください。それぞれが頭の中の計画で動き、配管が配線を塞ぎ、部屋のサイズが合わず、完成後に解体して作り直す事態になります。

ソフトウェア開発も同じです。仕様書は設計図です。SDD は「まず設計図を描いてから建てる」方法です。

## AIコーディングエージェントとは？

SDD の話をするとき頻繁に登場する **AIコーディングエージェント（AI coding agent）** を理解しておくと、SDD の必要性がより明確になります。

AIコーディングエージェントとは、開発者（またはプロダクトマネージャー）の指示を受けて、コードを自動的に作成・修正・実行するAIソフトウェアツールです。「ユーザーログイン機能を作って」と入力すると、AIが実際のコードを生成します。代表的なツールとして Claude Code、GitHub Copilot、Kiro などがあります。

AIコーディングエージェントの能力は2024年以降、急速に向上しています（Anthropic Claude Code ドキュメント、2025年8月時点）。しかしこれらのツールには共通の弱点があります。**指示が曖昧だと、結果も曖昧です。** 「良いログイン機能」がどういうものかをAIが自己判断することはできません。明確で構造化された基準がなければ、AIは自分の推測で実装します。仕様書がその基準になります。

---

## SDD ワークフロー — 6つのステージ

SDD は次の6つのステージを順番に進みます。各ステージは次のステージへの入力となります。

1. **原則文書 (Constitution)**: プロジェクト全体に適用されるルールとガイドラインを文書化します。「このプロジェクトではどんな原則で開発するか？」を定義するステージです。原則文書（constitution）はその後のすべての仕様書の基準となります。

2. **仕様作成 (Specify)**: 個々の機能に対する仕様書（specification）を作成します。「何を作るのか、成功はどう測るのか、スコープはどこまでか」に答えます。

3. **計画策定 (Plan)**: 仕様書をもとに技術的な実装計画を立てます。「どうやって作るか」を決めるステージです。

4. **タスク生成 (Tasks)**: 計画を独立して完了できる作業単位に分解します。各タスクは完了したかどうかを単独で確認できる粒度にします。

5. **実装 (Implement)**: タスクリストに従ってコードを書きます。AIコーディングエージェントがこのステージを自動化できます。仕様書から導出された構造化されたタスクリストは、AIエージェントへの理想的な入力です。

6. **レビュー (Review)**: 実装結果が仕様書の受け入れ基準を満たしているかを検証します。問題が見つかれば仕様書を先に修正し、次のサイクルを始めます。

この6つのステージは一度で終わりません。レビューの結果を仕様作成に反映しながら繰り返します。

---

## 仕様書（仕様書）とは？

**仕様書（specification）** は「何を作るか」を具体的に記述したドキュメントです。仕様書は次の質問に答えます。

- この機能はどのユーザーのためのものか？
- この機能が正しく動作していることをどうやって確認するか？（受け入れ基準）
- この機能のスコープ外は何か？
- 想定されるエッジケースや障害モードは何か？

SDD における仕様書は、一度書いてしまえばよいドキュメントではありません。開発が進む中で新しい情報が出てくれば、まず仕様書を更新し、実装がそれに続きます。これを **生きた仕様書（Living Specification）** と呼びます。

---

## SDD における原則文書（原則文書）とは？

SDD における **原則文書（constitution）** とは、プロジェクト全体に適用される開発原則と制約条件を定義したプロジェクト全体のドキュメントです。

仕様書が「この機能で何を作るか」に答えるとすれば、原則文書は「このプロジェクトで常に従うルールは何か」に答えます。すべての機能が受け継ぐ基本ルールと考えてください。

原則文書には例えば以下の内容が含まれます。

- チームが使用する技術スタック
- セキュリティ原則（例：ソースファイルに認証情報をハードコードしない）
- ドキュメント基準（例：すべてのAPIには実装開始前に仕様書が必要）
- レビューと承認プロセス

原則文書があれば、個々の仕様書で同じ基本的な制約を繰り返す必要がありません。すべての仕様書は原則文書を継承します。

## 手法の比較：SDD vs 従来型開発 vs TDD vs BDD vs バイブコーディング

| 比較項目 | 従来型開発 | TDD（テスト駆動開発, Test-Driven Development） | BDD（振る舞い駆動開発, Behavior-Driven Development） | SDD | バイブコーディング |
|---------|-----------|-----------------------------------------------|------------------------------------------------------|-----|-----------------|
| **開始点** | 口頭での要件またはチケット | 失敗するユニットテスト | ユーザー行動シナリオ（Gherkin） | 構造化された仕様書 | AIへの自然言語プロンプト |
| **主な成果物** | 実装コード（ドキュメントは後付け、またはなし） | テストコード＋実装コード | フィーチャーファイル＋実装コード | 仕様書＋実装コード＋検証レポート | 実装コード（仕様書なし） |
| **AIエージェント適合性** | 低い — 曖昧な指示 | 低い — テストが主なコンテキスト | 中程度 — Gherkinの解析が必要 | 非常に高い — 構造化された入力 | 中程度 — 速いが一貫性が低い |
| **ドキュメントとコードの同期** | ドキュメントがコードと容易に乖離 | テストが唯一のドキュメント | フィーチャーファイルが生きたドキュメントとして機能 | 仕様書が常に最新の状態に保たれる | ドキュメントなし |
| **適合するチーム規模** | 小規模〜大規模 | 小規模〜中規模 | 中規模〜大規模 | 小規模〜大規模 | 個人〜小規模 |
| **学習コスト** | 低い | 中程度（テスト作成スキルが必要） | 高い（Gherkin言語の習得が必要） | 中程度（仕様書作成の訓練が必要） | 非常に低い |

**バイブコーディング（vibe coding）** とは、形式的な仕様書や計画を持たず、AIコーディングツールに自由形式の自然言語で指示しながら開発するスタイルです。即興的（vibe）に感じのままにAIと会話しながら開発します。素早いプロトタイピングや個人的な探索には非常に効果的ですが、プロジェクトの複雑さが増すにつれて一貫性の問題が現れます。

SDD はバイブコーディングの対極ではありません。バイブコーディングが進化した先にあるものです。多くのチームがアイデアを素早く探索するためにバイブコーディングから始め、方向性が定まったらSDD に切り替えます。SDD はAIツールの速度を維持しながら、構造と一貫性を加える方法です。

**TDD（テスト駆動開発, Test-Driven Development）** と **BDD（振る舞い駆動開発, Behavior-Driven Development）** はいずれも、実装コードの前に何かを書くという手法です。TDD はテスト、BDD はユーザー行動シナリオを先に書きます。SDD はどちらよりも高い抽象レベルで動作し、実装前に機能全体の仕様を定義します。TDD や BDD との併用も可能です。

---

## SDD を使うべき場面

SDD が特に効果的な状況は以下の通りです。

- **AIコーディングエージェントを使用するチーム**: 構造化された仕様書はAIエージェントに明確な基準を与え、修正サイクルを減らします。
- **複数人で協業するプロジェクト**: 仕様書が技術チームと非技術チームの間の共通言語になります。
- **要件が頻繁に変わる環境**: まず仕様書を更新することで、コードに触れる前に変更の影響範囲を把握できます。
- **長期的なメンテナンスが必要なソフトウェア**: 生きた仕様書が、何年経っても開発当時の意図を説明します。
- **技術者以外の関係者と一緒に仕事するチーム**: 仕様書はコードを知らない人でも読んでフィードバックできるドキュメントです。

---

## SDD を使わなくてよい場面

すべての状況に SDD が合うわけではありません。以下の場合はより軽量なアプローチが良いかもしれません。

- **1〜2日で捨てるプロトタイプ**: 仕様書を書く時間が実装時間を超えてしまうことがあります。
- **完全に一人で行う小規模な実験**: 頭の中の計画で十分な場合。
- **要件自体がまだ不明確な初期探索フェーズ**: このフェーズではバイブコーディングで素早く探索し、方向性が定まったらSDD に切り替えるのが効果的です。

## よくある誤解

**誤解1：「SDD は開始前にたくさんのドキュメントを書かなければならない」**

仕様書は長い報告書ではありません。機能に関する特定の質問に答える、短くて構造化されたドキュメントです。慣れれば、ほとんどの機能で数分から1時間で書けます。そして間違ったものを作ってしまう手戻りを避けることで、その時間は回収できます。

**誤解2：「SDD は TDD や BDD を置き換える」**

SDD は TDD や BDD と競合しません。SDD はより高いレベルのフレームワークです。TDD のユニットテストや BDD の Gherkin シナリオは、どちらも SDD のワークフローの中で共存できます。

**誤解3：「優れた AI があれば仕様書は不要だ」**

AIコーディングエージェントはコードを生成するのが得意ですが、何を作りたいかを自分で判断することはできません。仕様書がその指示です。仕様書がなければAIは自分の推測で実装します。AIの能力が高いほど、明確な仕様書が重要になります。なぜなら、AIは持っているガイダンスのままに、自信を持って大規模に行動するからです。

**誤解4：「SDD はアジャイル開発と相容れない」**

SDD はアジャイル（Agile）手法と非常に相性が良いです。スプリントごとに仕様書を書き、実装し、レビューするプロセスは、アジャイルの反復サイクルと自然にフィットします。SDD はスピードを落とすものではなく、AI時代のアジャイルをより効果的にする方法です。

## 実践的な例 — 誰でも理解できるシナリオ

**状況**: ECサイトのプロダクトマネージャーのAlexさんが「注文状況通知機能」を追加したいと考えています。

**バイブコーディングの場合**: Alexさんが AIに「注文が発送されたらお客さんに通知を送って」とメッセージを送ります。AIはメール通知を作りましたが、Alexさんはプッシュ通知を想定していて、タイミングも発送時ではなく配達完了時を望んでいました。何度もやり直しが続きます。

**SDD の場合**: Alexさんはまず仕様書を書きます。
- **機能**: 注文状況通知
- **通知チャネル**: モバイルプッシュ通知（メールはスコープ外）
- **トリガーイベント**: 注文配達完了時（注文発送時はスコープ外）
- **受け入れ基準**: 配達確認から5分以内に、顧客のモバイルデバイスにプッシュ通知が届く。

AIコーディングエージェントがこの仕様書を受け取り、最初から正しく実装します。修正サイクルの数が大幅に減り、完成品が当初の意図と一致します。

## バイブコーディング（バイブコーディング）とは？

**バイブコーディング（vibe coding）** とは、形式的な仕様書や構造化された計画を持たずに、AIコーディングツールに自由形式の自然言語で指示しながら開発するアプローチです。「バイブ（vibe）」という言葉が表す即興的な感覚で、したいことを会話的に説明し、残りをAIに解釈させます。

バイブコーディングが本当に役立つのは以下の場合です。
- アイデアを素早く動くプロトタイプにするとき
- セッション間の一貫性が重要でない個人プロジェクト
- 機能の形が見えていない初期段階の探索

バイブコーディングの限界が現れるのは以下の場合です。
- 複数の人が同じコードベースで作業する必要があるとき
- 機能が複雑で AIに正確な基準が必要なとき
- プロジェクトを何ヶ月・何年にわたってメンテナンスするとき

SDD はバイブコーディングの精神を否定するものではありません。AIツールがもたらす速度に構造を加える方法です。

## SDD のメリット

- **手戻りの削減**: 実装前に要件を定義することで、間違ったものを作るコストが減ります。
- **AI出力の品質向上**: 構造化された仕様書はAIコーディングエージェントへのより良い入力です。AIはより正確で一貫した結果を出します。
- **チームコミュニケーションの改善**: 仕様書は技術者と非技術者の両方が読んでレビューできる共有ドキュメントです。
- **変更管理の容易さ**: 要件が変わるとき、まず仕様書を更新することでコードに触れる前に影響全体を把握できます。
- **知識の保存**: 開発者がチームを離れても、仕様書がシステムの意図を記録します。

---

## リスクと限界

- **初期投資**: 良い仕様書を書くには練習が必要です。SDD に慣れていないチームは習慣を身につけるまで時間がかかります。
- **過度な仕様化のリスク**: 小さなすべての決定を形式的なドキュメントにしようとすると、チームの速度が落ちます。仕様書は重要なこと — 要件と受け入れ基準 — に集中すべきです。
- **仕様書のドリフト**: 要件が変わったときに仕様書を更新する規律がないと、仕様書がコードとずれてしまいます。これは SDD が解決しようとしているドキュメント問題と同じであり、チームの規律が必要です。
- **短期プロジェクトへの過剰適用**: 数時間で終わるタスクに完全な SDD ワークフローを適用するのは過剰かもしれません。

## 主要用語集

| 用語 | 説明 |
|------|------|
| **仕様書 (Specification)** | 機能の要件、受け入れ基準、スコープを記述した文書。SDD の主要な成果物。 |
| **原則文書 (Constitution)** | すべての機能が従うべき開発原則と制約条件を定義した、プロジェクト全体のドキュメント。 |
| **AIコーディングエージェント (AI Coding Agent)** | 人間の指示に基づいてコードを自動的に作成・修正するソフトウェアツール。例：Claude Code、GitHub Copilot、Kiro。 |
| **バイブコーディング (Vibe Coding)** | 形式的な仕様書を持たず、AIコーディングツールに自由形式の自然言語で指示しながら開発するスタイル。 |
| **パイプライン (Pipeline)** | SDD の6段階のワークフロー：Constitution → Specify → Plan → Tasks → Implement → Review。 |
| **TDD（テスト駆動開発）** | 実装コードの前に失敗するユニットテストを書く開発手法。 |
| **BDD（振る舞い駆動開発）** | 実装コードの前にユーザー行動シナリオを書く開発手法。 |
| **生きた仕様書 (Living Specification)** | 開発が進む中で継続的に更新され、書かれた説明が実際の動作と同期し続ける仕様書。 |
