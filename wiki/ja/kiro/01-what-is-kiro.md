# Kiroとは何か？

> KiroはAWSの製品であるエージェンティック（agentic）IDE・CLI です。非公式なチャットプロンプトではなく、構造化された仕様書を使ってAIコーディングアシスタントを導きます。

## このページの対象読者

このページは、AIを使った開発ツールに興味があり、Kiroが何であるか、自分の仕事に関係するかを知りたいプロダクトマネージャー、スタートアップ創業者、その他すべての方を対象としています。技術的な背景は不要です。

## なぜこれが重要か

AIコーディングアシスタントを試したことのある人なら、同じ悩みを経験したことがあるはずです。AIに何かを作るよう頼んだら、意図と違うものができた。1回目は正しいものを作ったのに、似たような質問を次回すると全然違う結果が返ってきた。プロジェクトが複雑になればなるほど、この問題は深刻になっていきます。

Kiroはこの問題を解決するために存在します。AIの代わりになるのではなく、AIが作業する構造化された指示書を与えることで、結果をより予測可能で一貫したものにします。

AIコーディングツールがソフトウェア開発の標準になりつつある今、Kiroが何かを理解することは重要です。問題はもはやAIツールを使うかどうかではなく、どう上手く使うかです。Kiroはその答えのひとつです。

## わかりやすい説明

Kiroは Amazon Web Services (AWS) が作ったソフトウェアツールで、kiro.dev から入手できます。IDE（統合開発環境 — 開発者がコードを書いて実行するプログラム）とCLI（コマンドラインインターフェース — ターミナルウィンドウで使うテキストベースのツール）の2形態があります。

Kiroの核心にあるシンプルなアイデアは、「AIがコードを書く前に、あなたが仕様書を書く」というものです。仕様書とは、何を作りたいか、「完成」がどういう状態かを記述した短い構造化ドキュメントです。AIはその後、非公式な会話プロンプトではなく、この仕様書をもとに作業を進めます。

このアプローチを仕様駆動開発（SDD、Spec-Driven Development）と呼びます。Kiroはつまり、SDDを簡単かつ自動化されたものにするために構築されたIDEです。

2026-05-19時点、Kiroはプレビュー段階で、継続的に開発が進められていました。機能、料金、地域での提供状況はこの記事が書かれた後に変わっている可能性があります。最新情報は必ず公式サイト [kiro.dev](https://kiro.dev) でご確認ください。

## 核心概念

### エージェンティックIDE（Agentic IDE）

エージェンティックIDEとは、AIがコードを提案するだけでなく、ファイルを読み取り、コードを書き、コマンドを実行し、複数ステップのアクションをあなたの代わりに取れる開発環境です。KiroのIDEはVisual Studio Code (VS Code)をベースに構築されているため、開発者は既存の拡張機能や設定をそのまま使えます。

**なぜ重要か**: ほとんどのAIコーディングツールは「提案」ツールです — どのコードを入力するかを表示するだけで、入力するのはあなたです。エージェンティックツールは、あなたが結果を確認している間に完全な実装計画を実行できます。

**例**: ログイン機能の仕様書を書きます。Kiroはその仕様書を読み取り、コードを一行も入力することなく、ファイル、テスト、データベーススキーマを含む機能全体を実装します。

### CLI（コマンドラインインターフェース）

KiroはCLIとしても動作するため、フルIDEを開かずにターミナルウィンドウから呼び出せます。自動化されたワークフローを実行するチームや、Kiroを既存の開発パイプラインに統合したいチームに便利です。

**なぜ重要か**: すべてのワークフローがグラフィカルIDEから始まるわけではありません。CLIはKiroをサーバー環境、CI/CDパイプライン、ターミナルベースのツールを好む開発者にも使いやすくします。

**例**: チームメンバーがターミナルで `kiro implement --spec feature-001` を実行し、CIパイプライン内で実装タスクを開始します。

### バイブコーディング（vibe coding） — Kiroが解決する問題

バイブコーディング（vibe coding）とは、正式な計画なしにAIツールに自由形式の会話的な指示を与える開発アプローチです。この言葉は、即興的で直感に頼った性質を表しています。シンプルなタスクには速いですが、プロジェクトが成長するにつれて結果が一貫しなくなります。

**なぜ重要か**: Kiroはバイブコーディングへの構造化された代替手段として設計されています。バイブコーディングを理解することで、Kiroが何を解決しようとしているかがわかります。

**例**: バイブコーディング — 「ねえAI、ユーザーデータを保存するボタンを追加して。」Kiro — どのデータを保存するか、成功・失敗時に何が起きるか、受け入れ基準は何かを記述した仕様書を書く。AIはその仕様書に対して実装を行います。

## たとえ話

> キッチンのリフォームを業者に頼むとします。「いい感じにして」と言えば、業者は自分で判断することになります — 嫌いな色のキャビネット、不便なレイアウト、違う素材。キッチンはできますが、あなたが欲しいキッチンではないかもしれません。
>
> 今度は設計図を提供するとします。正確な寸法、指定された素材、納品物のリスト、受け入れチェックリスト。業者はあなたが本当に欲しいものを作るのに必要なものをすべて持っています。何か不明な点があれば、推測する代わりに設計図を参照します。

Kiroとは、あなたの曖昧なAIプロンプトを設計図に変換し、その設計図をAIエージェントに渡すシステムです。

## 実際の活用事例

**シナリオ1 — 新機能開発**: 開発者が既存のアプリケーションに新しい機能を追加したいと思っています。会話的にAIにプロンプトして一貫した結果を期待する代わりに、Kiroを使って最初に仕様書を作成します。KiroのAIはその仕様書に基づいて機能を実装し、一貫してレビュー可能な結果を出します。

**シナリオ2 — チームの一貫性**: 3人の開発者チームが共有ステアリングファイル（プロジェクト全体の指針ドキュメント）でKiroを使っています。Kiroは毎回実行時に共有指針を読み込むため、すべての開発者のAI生成コードが同じ規約に従います。

**シナリオ3 — 非開発者の参加**: プロダクトマネージャーが機能が何をすべきかを平易な言語で記述した仕様書を書きます。開発者はその仕様書をKiroで使って実装を導きます。仕様書が両方の役割が共同作業する共通ドキュメントになります。

## よくある誤解

1. **「KiroはただのAIチャットボットだ。」** — Kiroはチャットボットではありません。仕様駆動のエージェンティックなソフトウェア開発専用に設計されたIDEとCLIです。会話インターフェースはその動作方法のほんの一部です。
2. **「Kiroは開発者に取って代わる。」** — Kiroは繰り返しの実装作業を自動化することで開発者をサポートします。開発者は引き続き仕様書を定義し、出力をレビューし、何を作るかの意思決定を行います。Kiroは実行を担当します。
3. **「Kiroを使うには開発者である必要がある。」** — Kiroを理解するのに技術的な知識は不要です。技術者でないコントリビューター（プロダクトマネージャー、デザイナー）が平易な言語で仕様書を書き、開発者とKiroがそれをもとに作業できます。
