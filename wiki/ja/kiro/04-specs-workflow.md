# Kiro仕様書の使い方

> Kiroの仕様書（Spec）は3つの構造化ドキュメントのセットです。要件・設計・タスク — これらがAIを機能のアイデアからレビュー済みの実装まで導きます。

## このページの対象読者

このページは、Kiroの仕様書ライフサイクル全体を理解したい開発者とプロダクトマネージャーを対象としています。初期説明の記述から完成した実装のレビューまでをカバーします。ソフトウェア開発のコンセプトへの基本的な馴染みがあると役立ちますが、必須ではありません。

## なぜこれが重要か

仕様書はKiroのワークフローの中心的な成果物です。Kiroが行うすべてのことはここから流れ出します。仕様書がどのように機能するか — どのドキュメントが含まれるか、それらがどう関係しあうか、どう更新するか — を理解することが、Kiroと効果的に作業するための土台です。

仕様書のライフサイクルを理解せずに、チームはKiroをコードジェネレーターとして扱い、その核心的な価値を見逃してしまいます。それは、AIのアウトプットをチームの実際の意図と一致させる構造化されたプロセスです。

## わかりやすい説明

Kiroの仕様書は、作りたい機能のシンプルな説明から始まります。その説明から、KiroのAIが3つの構造化ドキュメントを生成します。それをレビューして改善します。その後、Kiroはそれらのドキュメントから機能を実装します。

3つのドキュメントは：

- **`requirements.md`** — ユーザーが何をできるべきか（ユーザーストーリーと受け入れ基準）
- **`design.md`** — 機能をどのように作るか（技術アーキテクチャ、コンポーネント、データ構造）
- **`tasks.md`** — 設計を実装するための順序付きステップリスト

これらのドキュメントはプロジェクトの`.kiro/specs/[機能名]/`以下に置かれます。コードと一緒にバージョン管理されるため、仕様書の歴史とコードの歴史が同期されます。

## 核心概念

### requirements.md

要件ドキュメントはユーザーストーリーをキャプチャします — ユーザーが何をしたいか、なぜかを説明する短い記述です。各ユーザーストーリーには受け入れ基準があります。要件が満たされたかどうかを判断する測定可能な条件です。

**なぜ重要か**: 平易な言語で書かれた要件（ユーザーストーリー）により、技術者でないステークホルダーがコードが書かれる前にいねいに何が作られるかをレビューし承認できます。受け入れ基準はAIに実装の具体的なターゲットを与えます。

**例**:
```
## ユーザーストーリー1: タスクの追加
ユーザーとして、タイトル付きのタスクを追加して、
やるべきことを管理できるようにしたい。
受け入れ基準：
- タスクは追加直後にタスクリストに表示される
- タイトルのないタスクは送信できない
- タスクは自動的に一意のIDが割り当てられる
```

### design.md

設計ドキュメントは要件を技術計画に変換します。どのコンポーネントが作られるか、どのデータ構造を使うか、どう相互作用するか。コードが書かれる前にアーキテクチャの決定がここで文書化されます。

**なぜ重要か**: 実装前に設計を書くことで、変更がコストのかかる実装後ではなく、変更が安価な段階で矛盾や不明確な要件が露わになります。

**例**: タスクトラッカー機能の設計ドキュメントでは、タスクをローカルJSONファイルに保存すること、UIがリストコンポーネントをレンダリングすること、各タスクオブジェクトが`id`、`title`、`status`、`created_at`フィールドを持つことを指定できます。

### tasks.md

タスクドキュメントは設計から導き出された順序付きチェックリストです。各タスクは独立して完了可能で、検証できるほど具体的で、単一セッションで実装できるほど小さいサイズです。

**なぜ重要か**: 仕様書から導き出されたタスクリストはAIに明確で境界のある作業単位を与えます。各タスクには定義されたスコープがあり、設計ドキュメントに対して検証できます。これによりAIが合意された計画の外に逸脱するのを防ぎます。

**例**:
```
- [ ] id、title、status、created_atフィールドを持つTaskデータモデルを作成
- [ ] タイトルを受け取りTaskを返すaddTask()関数を実装
- [ ] すべてのタスクを表示するrenderTaskList()コンポーネントを実装
- [ ] ユーザーストーリー1の受け入れテストを書く
```

## たとえ話

> 家を建てることを考えてみましょう。要件ドキュメントはホームオーナーが欲しい部屋と機能のリストです（「寝室3つ、ホームオフィス、ガレージ」）。設計ドキュメントは建築士の設計図 — 実際のレイアウト、寸法、仕様です。タスクドキュメントは工事請負業者の作業指示書です — 「1日目に基礎を打つ、2〜3日目に壁を立てる、4〜5日目に屋根を付ける」。

各ドキュメントは異なる質問に答えます。何が欲しいか、どう作るか、誰がいつ何をするか。建設が始まる前に3つすべてが必要です。

## 実際の活用事例

**ステップごとのワークフロー**:

1. **機能を説明する**: Kiroに作りたいものを平易な言語で伝えます。「タスクを追加・完了・フィルタリング・削除できる個人タスクトラッカーを作りたい。」

2. **requirements.mdをレビューする**: Kiroがユーザーストーリーと受け入れ基準を生成します。注意深く読んでください。意図と一致していますか？受け入れ基準は測定可能ですか？あなたが欲しいものを正確に表すまでドキュメントを編集します。

3. **design.mdをレビューする**: Kiroが技術設計を生成します。技術的な背景があれば、賛同できないアーキテクチャの選択を確認してください。技術者でなければ、設計が要件を満たしていることを確認します。

4. **tasks.mdをレビューする**: Kiroが実装タスクリストを生成します。タスクが論理的な順序になっていて、すべての要件をカバーしているか確認します。

5. **実装する**: Kiroにタスクを実装するよう伝えます。Kiroはタスクリストを処理し、各タスクを要件と設計ドキュメントに対して確認します。

6. **レビューして反復する**: requirements.mdの受け入れ基準に対して実装をレビューします。何かおかしければ、まず仕様書を更新し、その後影響を受けるタスクをKiroに再実装するよう求めます。

## よくある誤解

1. **「仕様書ドキュメントはコードコメントとドキュメントの代替だ。」** — 仕様書ドキュメントは実装前の意図を記述します。コードコメントは実装がどう機能するかを記述します。どちらも異なる目的を持ち、どちらも必要です。
2. **「仕様書を一度書いたら変更すべきでない。」** — 仕様書は生きたドキュメントです。要件が変わったり、実装が設計の欠陥を明らかにしたとき、まず仕様書を更新し、実装がそれに続きます。変わらない仕様書は現実との同期が取れていない仕様書です。
3. **「Kiroが自動的に仕様書を書く — 私は承認するだけだ。」** — Kiroはあなたの説明から仕様書の草稿を作ります。レビューして修正するのはあなたの責任です。実装の質は完全にあなたが承認する仕様書の質にかかっています。
