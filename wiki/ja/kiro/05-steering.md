# Kiroステアリングガイド

> ステアリングファイル（steering files）とは`.kiro/steering/`にあるMarkdownドキュメントで、AIエージェントにすべてのセッションとすべてのチームメンバーに適用される、プロジェクト全体の持続的なルールを与えます。

## このページの対象読者

このページは、プロジェクト全体に適用される一貫した信頼できる指示をKiroに与える方法を理解したい開発者とチームリードを対象としています。コーディング規約というコンセプトへの基本的な馴染みがあると役立ちます。

## なぜこれが重要か

AIコーディングツールへの最大の不満の一つは一貫性のなさです。月曜日はあるスタイルでコードを生成し、水曜日には別のスタイルで生成する。ある開発者のAI生成コードはcamelCaseを使い、別の開発者のはsnake_caseを使う。AIはあるファイルではHTTPリクエストに1つのライブラリを使い、別のファイルでは別のライブラリを使う。

ステアリングファイルはこの問題を解決します。チームの規約、アーキテクチャルール、プロジェクト固有の制約がAIへの持続的な入力となる仕組みです — 毎回、すべてのチームメンバーに自動的に適用されます。

良いステアリングファイルに時間を投資するチームは、AIのアウトプットの一貫性が大幅に向上すると報告しています。ステアリングファイルをスキップするチームは、AIの不一致を修正するためにより多くの時間を費やします。

## わかりやすい説明

ステアリングファイルはプロジェクトの`.kiro/steering/`に保存されるMarkdownファイルです。AIへの定常的な指示セットとして書きます。「常にこの命名規則を使う」「このライブラリは使わない」「APIを作るときは常にこのレスポンス構造に従う」といった内容です。

Kiroはすべての AIセッションの開始時にステアリングファイルを読み込みます。コードを生成し、質問に答え、タスクを実行するたびに、ステアリングファイルのルールを適用します。ステアリングファイルはプロジェクトのgitリポジトリにコミットされているため、すべてのチームメンバーで共有され、コードと並んでバージョン管理されます。

複数のステアリングファイルを持つことができます。よくあるアプローチは、コーディング規約用に1つ、アーキテクチャルール用に1つ、セキュリティ要件用に1つというように、関心事ごとに整理することです。

## 核心概念

### ステアリングファイルに何を書くか

ステアリングファイルが最も役立つのは次のようなルールです：
- プロジェクト全体に適用される（1つの機能だけでなく）
- すべての仕様書やプロンプトで繰り返すのが面倒になるもの
- すべてのチームメンバーが一貫して従うべきもの

**なぜ重要か**: ルールが1つの機能にのみ適用される場合、その機能の仕様書に書きます。ルールがどこにでも適用される場合、ステアリングファイルに書きます。

**ステアリングファイルの内容例**:
```markdown
# プロジェクト規約

## 命名
- 変数名：camelCase（JavaScript/TypeScript）
- ファイル名：kebab-case
- データベース列名：snake_case

## APIレスポンス
すべてのAPIレスポンスは以下のエンベロープを使用すること：
{
  "status": "success" | "error",
  "data": <ペイロード>,
  "error": null | { "code": string, "message": string }
}

## 依存関係
- HTTPリクエスト：`fetch` APIを使用 — axiosを追加しない
- 日付処理：date-fnsを使用 — moment.jsは使用しない
```

### KiroがステアリングファイルをどのようにA読むか

Kiroはすべてのセッションの開始時に`.kiro/steering/`のすべてのファイルを読み込みます。ルールはAIモデルの持続的なコンテキストとして適用されます — そのセッション中にAIが行うすべてのコード生成、レビュー、提案に情報を与えます。

**なぜ重要か**: 一度入力して忘れるプロンプトとは異なり、ステアリングファイルは常に存在します。繰り返す必要がありません。AIはそれをすべての作業のバックグラウンドコンテキストとして持ちます。

**例**: APIレスポンスエンベロープの規約を確立した機能とは全く別の機能に取り組んでいる開発者でも、AIはその規約に従います — ステアリングファイルは常に読み込まれるからです。

## たとえ話

> フランチャイズのレストランチェーンを考えてみましょう。すべての店舗が同じ運営マニュアルに従っています。各料理の作り方、使う食材、盛り付け方。マニュアルはシフトごとに書き直されるのではなく — 常にすべての人に適用される定常的な指示セットです。

Kiroのステアリングファイルはプロジェクトの運営マニュアルです。すべてのAIアクションがマニュアルに従います。新しいチームメンバー、新しい機能、新しいセッション — マニュアルはそれらすべてに適用されます。

## 実際の活用事例

**新しいプロジェクトへのステアリングのセットアップ**:

1. プロジェクトルートに`.kiro/steering/`を作成します。
2. `conventions.md`を作成 — 命名規則、コードスタイルの好み、ライブラリの選択を文書化します。
3. `architecture.md`を作成 — プロジェクトの構造を文書化します。何がどこに行くか、レイヤーがどのように通信するか、どのパターンに従うか。
4. `security.md`を作成（関連する場合） — セキュリティ要件を文書化します。資格情報のハードコード禁止、必須の入力バリデーション、認証アプローチ。
5. すべてのステアリングファイルをgitにコミットします。

**ステアリングファイルのメンテナンス**:
- チームが新しいアーキテクチャの決定を下したとき、ステアリングファイルを更新します。
- ステアリングファイルをコードと同じ注意で扱います。プルリクエストで変更をレビューします。
- AIが規約を破るアウトプットを出したとき、その規約がステアリングファイルに文書化されているか確認します。ない場合は追加します。

**ステアリングファイルのカバレッジ確認**:
AIが同じルールを繰り返し破っていることに気づいたら、そのルールはおそらくステアリングファイルに入れる必要があります。AIのエラーをステアリングドキュメントのギャップのシグナルとして使います。

## よくある誤解

1. **「ステアリングファイルはREADMEと同じだ。」** — READMEは人間の読者にプロジェクトを説明します。ステアリングファイルはAIエージェントに指示します。特定のAI動作を生み出すために書かれており、初日の新しいチームメンバーが読むためではありません（そのような目的を副次的に果たせますが）。
2. **「ステアリングが多ければ多いほどいい。」** — 過度に長かったり矛盾したりするステアリングファイルはAIのパフォーマンスを低下させます。各ルールを明確で曖昧さのないものにしてください。2つのルールが矛盾する場合、ドキュメント内でその矛盾を解決します。
3. **「ステアリングファイルは機能ごとの仕様書の代替だ。」** — ステアリングファイルと仕様書は異なる目的を果たします。ステアリングは常に真であることをカバーします。仕様書は1つの特定の機能に対して真であることをカバーします。どちらも必要です。
