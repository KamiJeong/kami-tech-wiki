# GitHub Spec Kitとは何か？

> GitHub Spec KitはSDDツールキットです。Claude Codeを構造化された仕様書優先ワークフローに沿って導く一連のコマンドで、一貫してレビュー可能なAI実装を生み出します。

## このページの対象読者

このページは、Spec Kitとは何か、何をするのか、単にClaudeにプロンプトするのとどう違うのかを理解したいClaude Codeを使う開発者を対象としています。Claude Codeへの基本的な馴染み以外の特定の技術的背景は不要です。

## なぜこれが重要か

Claude Codeは強力なAIコーディングアシスタントですが、構造なしでは、アウトプットの品質はプロンプトの品質に大きく依存します。明確で詳細なプロンプトは良い結果を出します。曖昧なプロンプトは曖昧な結果を出します。複雑な機能では、プロンプトとアウトプットの間の手戻り修正サイクルが高コストになり得ます。

Spec Kitは構造を導入します。1つのプロンプトが1つのアウトプットを生み出す代わりに、Spec Kitは定義されたワークフローを通じてあなたを導きます。仕様書を書き、曖昧さを明確にし、実装計画を作り、タスクリストを生成し、そして実装する。各ステップが次のステップへの入力となるドキュメントを生み出します。AIは1つの会話的なプロンプトからではなく、構造化された成果物から作業します。

結果は、より予測可能なアウトプット、意図と実装の間のより明確な追跡可能性、そして何を作りなぜ作ったかの文書化された記録です。

## わかりやすい説明

Spec KitはClaude Codeスキルのセットとして実装されています。ワークフローの各ステップを実行するためにClaude Codeで呼び出す専用コマンドです。別のアプリケーションでも、GitHubが販売する製品でもありません。Claude Codeで使うために設計された特定のリポジトリに組み込まれたツールキットです。

ツールキットのコマンドは特定の順序に従います。プロジェクト憲章（プロジェクトの管理ルール）から始め、機能の仕様書を書き、曖昧さを明確にし、実装計画を作り、計画をタスクに分解し、最後にタスクを実装します。

Spec Kitは各ステップでMarkdownファイルを生成します — `spec.md`、`plan.md`、`tasks.md` — これらはコードの隣にリポジトリに置かれます。これらのファイルは機能の開発中に下されたすべての決定の文書化された記録として機能します。

重要な区別：Spec KitはGitHubの公式製品ではありません。名前は、仕様書駆動ワークフローでGitHubホストのリポジトリで使うために設計されたツールキットであることを反映しています。内部ツールキットであり、商業的に配布されるツールではありません。

## 核心概念

### SDDツールキット

Spec KitはSDD（仕様駆動開発、Spec-Driven Development）の実装です — コードの前に仕様書を書くという方法論。プロセスを歩み抜けるための一連のコマンドを提供することで、SDDを実用化します。

**なぜ重要か**: 方法論としてのSDDは原則として明確ですが、実践するには規律が必要です。Spec Kitは各ステップを構造化する具体的なコマンドを提供することで、SDDを従いやすくします。良い仕様書の書き方を覚えておく必要はありません — ツールキットがあなたを導きます。

**例**: 空白のドキュメントを開いてゼロから仕様書を書く代わりに、`/speckit-specify`を実行して機能を平易な言語で説明します。Spec Kitが構造化された仕様書ドキュメントを生成します。あなたはそれをレビューして改善します。

### Claude Codeスキル

Spec KitのコマンドはClaude Codeスキルとして実装されています — Claude Codeに新しいコマンドを追加する拡張の一種です。Spec Kitのコマンドを呼び出すと、Claude CodeはスキルのロジックをSORTINGします。このロジックはAIに特定の構造化されたタスクを実行するよう指示します。

**なぜ重要か**: Spec KitはClaude Codeスキルとして実装されているため、すでに使っているClaude Code環境の中で実行されます。インストールや学習が必要な別のツールはありません。コマンドは他のClaude Codeコマンドと並んで表示されます。

**例**: Claude Codeで`/speckit-plan`を実行すると、計画生成スキルがトリガーされます。このスキルは既存の`spec.md`を読み込み、実装アプローチ、テックスタックの決定、ファイル構造を含む`plan.md`を生成します。

### パイプラインアーティファクト

Spec Kitワークフローの各ステップはパイプラインアーティファクトと呼ばれるMarkdownファイルを生み出します。これらのファイル（`spec.md`、`plan.md`、`tasks.md`）はリポジトリに置かれ、機能の開発の文書化された記録として機能します。

**なぜ重要か**: パイプラインアーティファクトは開発プロセスを監査可能にします。チームメンバー（またはAI）は仕様書を読んで何を作りなぜ作ったかを理解できます — コードを追跡したり元の作者に聞いたりする必要なく。

**例**: 機能が構築されてから6ヶ月後、新しい開発者が`spec.md`を読んで機能が何をすべきかを理解し、`tasks.md`を読んでどのように実装されたかを理解します。ドキュメントはリポジトリにあり、誰かの記憶の中にあるのではありません。

## たとえ話

> Spec KitはAI開発のためのプロジェクト管理テンプレートのようなものです。毎回ゼロからプロセスを考え出す代わりに、定義された一連のステップに従います。要件を集め、計画を作り、タスクに分解し、実行する。テンプレートは作業を代わりにやるのではなく — 作業がどのように行われるかを構造化します。

AIが作業者です。Spec Kitは作業が一貫した文書化されたプロセスに従うことを保証するテンプレートです。

## 実際の活用事例

**シナリオ1 — 明確な要件の新機能**: 開発者が検索機能を追加したいと思っています。Claudeに直接プロンプトする代わりに、`/speckit-specify`で構造化された仕様書を作成し、`/speckit-plan`で実装計画を作成し、`/speckit-tasks`でタスクリストを生成し、`/speckit-implement`でタスクを実行します。各ステップがレビュー可能なドキュメントを生み出します。

**シナリオ2 — 不明確な要件の機能**: 開発者が漠然とした機能のアイデアを持っています。`/speckit-specify`で初期仕様書を作成し、`/speckit-clarify`で計画が始まる前に曖昧さを特定して解決します。明確化のステップが早期に誤解を発見します。

**シナリオ3 — チームレビューワークフロー**: チームが実装が始まる前にSpec Kitが生成した`spec.md`と`plan.md`をレビューします。技術者でないチームメンバーが仕様書を読み、技術メンバーが計画をレビューします。AIがコードを書く前に両グループが何を作るかに合意します。

## よくある誤解

1. **「Spec KitはGitHubが作った。」** — Spec KitはGitHubの公式製品ではありません。特定のリポジトリに構築され、GitHubベースのワークフローで動作するよう設計されたツールキットです。名前はその起源ではなくコンテキストを反映しています。
2. **「Spec KitはClaude Codeに取って代わる。」** — Spec KitはClaude Codeの中で実行されます。Claude Codeに構造化されたワークフローコマンドを追加しますが、Claude Codeを置き換えたり競合したりしません。
3. **「すべての機能にすべてのSpec Kitコマンドを使う必要がある。」** — 完全なパイプライン（プロジェクト憲章 → 仕様書作成 → 明確化 → チェックリスト → 計画 → タスク → 分析 → 実装）は完全なプロセスが必要な機能に向けたものです。小さくリスクの低い変更には簡略化されたフロー（仕様書作成 → 計画 → タスク → 実装）が適切です。
