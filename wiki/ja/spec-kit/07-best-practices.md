# Spec Kitベストプラクティス

> チームがSpec Kitから信頼できる一貫した結果を得るための5つの実証済み方法 — 避けるべきよくある落とし穴とともに。

## このページの対象読者

このページは、Spec Kitを少なくとも一度使った経験があり、「動く」から「安定して動く」に移行したい開発者を対象としています。Spec Kitワークフローへの理解があることを前提としています。

## なぜこれが重要か

Spec Kitの成果物の品質は受け取る入力の品質に比例します。良い受け入れ基準、現在の状態のプロジェクト憲章、各ステップでの丁寧なレビューは信頼できる結果を生み出します。漠然とした基準、古いプロジェクト憲章、スキップされたレビューはSpec Kitが防ぐよう設計されたものと同じ場当たり的な結果を生み出します。

以下の方法は、Spec Kitへの最も一般的な不満の原因を解決し、習慣にしたときに最も一貫して影響を与えるものです。

## わかりやすい説明

Spec Kitから信頼できる結果を得ることは設定ではなく習慣です。以下の5つの方法は最初は意図的な努力が必要です。数十の機能を経た後は自動になります — そしてその報われ方は修正が少なくて済み、実際に作ったものを反映するドキュメントを生み出すAI実装です。

## 核心概念

### 方法1：仕様書を承認する前に受け入れ基準を書く

受け入れ基準は仕様書の最も重要なフィールドです。AIが実装する基準であり、品質検証が確認するものです。漠然とした基準（「ユーザーがタスクをフィルタリングできる」）は漠然とした実装を生み出します。具体的な基準（「「アクティブ」を選択すると、ステータスが「未完了」のタスクのみが表示され、件数が即座に更新される」）は具体的な実装を生み出します。

**なぜ重要か**: 受け入れ基準のすべての曖昧さは、あなたの入力なしにAIが下す決定です。その決定が気に入らないかもしれません。

**避けるべき落とし穴**: すべての受け入れ基準が測定可能で曖昧さのないものであることを確認する前に`/speckit-plan`を実行すること。

### 方法2：基本的な決定が変わったときにプロジェクト憲章を更新する

プロジェクト憲章はすべての機能仕様書を管理します。技術スタックが変わったとき、新しいセキュリティ要件が採用されたとき、アーキテクチャパターンが廃止されたとき、新しい機能が仕様書を作られる前に、プロジェクト憲章を現在の状態を反映するよう更新する必要があります。

**なぜ重要か**: 古いプロジェクト憲章は廃止された技術スタックを勧め、もう使われないパターンを実装する計画を生み出します。仕様書は正しくても、実装は間違ったルールに従います。

**避けるべき落とし穴**: プロジェクト憲章を一度きりのドキュメントとして扱うこと。プロジェクトの基本ルールが変わったときに更新される生きたガバナンスドキュメントです。

### 方法3：次のコマンドを実行する前に各成果物をレビューする

各コマンドの結果が次の入力です。中間成果物をレビューせずにコマンドの順序全体を実行することが悪い実装への最速の道です。`/speckit-plan`の前に`spec.md`をレビューします。`/speckit-tasks`の前に`plan.md`をレビューします。`/speckit-implement`の前に`tasks.md`をレビューします。

**なぜ重要か**: 仕様書で発見されたエラーは修正に数分かかります。実装後に発見されたエラーは数時間かかる場合があります。各ステップのレビューステップがそのレバレッジが適用される場所です。

**避けるべき落とし穴**: Spec Kitをワンクリックソリューションとして扱うこと。コマンドはプロセスを構造化しますが、人間のレビューがプロセスを信頼できるものにするのです。

### 方法4：機能の途中で要件が変わったときに仕様書を最新の状態に保つ

要件は変わります。変わったとき、コマンドを再実行したりAIに変更を求めたりする前に`spec.md`を更新します。これにより仕様書が真実の原源であり続け、変更がコード化されるだけでなく文書化されることを保証します。

**なぜ重要か**: 対応する仕様書の変更なしにコードが変わると、仕様書が実装と同期が取れなくなります。仕様書をもとにした将来のAI作業は実際のコードと矛盾する成果物を生み出します。

**避けるべき落とし穴**: 仕様書を更新せずにAIに「これだけ変えて」と頼むこと。小さな文書化されていない変更が仕様書とコードの間の大きなギャップとして積み重なります。

### 方法5：パッチではなく機能にSpec Kitを使う

Spec Kitは機能のために設計されています。要件、受け入れ基準、意味のある実装計画を持つ追加または変更。些細なパッチ（タイポ修正、定数の更新、変数名の変更）では、全体のSpec Kitパイプラインは相応する恩恵なしにオーバーヘッドを追加します。

**なぜ重要か**: 1行の変更に完全なワークフローを適用することは価値のないオーバーヘッドです。意味のある機能に完全なワークフローを適用することで価値が生まれます。判断力を使います。

**避けるべき落とし穴**: Spec Kitをすべてに適用すること、またはまったく何にも適用しないこと。正しいスコープは意味のある複雑さの機能です。

## たとえ話

> 慣れた飛行機と慣れたルートでも毎回フライト前チェックリストに従うパイロットは、急いでいるときにスキップするパイロットよりも事故が少ないです。チェックリストは長いフライトでパイロットを遅くしません。はるかに遅くなるだろう事故を防ぎます。

Spec Kitのレビューステップはフライト前チェックリストです。急いでいるときにスキップすることが正確に最も必要なときです。

## 実際の活用事例

**ソロ開発者の習慣**:
- 次のコマンドを実行する前にすべての生成された成果物をレビューします — 5分余分にかかっても
- 重要なアーキテクチャの決定後にプロジェクト憲章を更新します
- 仕様書を承認する前に最初の受け入れ基準を書きます — 「測定可能」が基準です

**チームの習慣**:
- プルリクエストで仕様書をレビューします — コードだけでなく`spec.md`と`plan.md`もレビュー
- プロジェクト憲章を維持する担当者を指定します — 全員でも誰でもなく
- 仕様書ドリフト（コードと同期していない仕様書）を許容可能な技術的負債ではなくバグとして扱います

**Spec Kitが一貫して悪い結果を出すとき**:
1. `spec.md`の受け入れ基準を読みます — 十分に具体的か？
2. プロジェクト憲章を読みます — 現在のプロジェクトルールを反映しているか？
3. 計画をレビューします — 意図しないアプローチを承認したか？
4. 再実装する前に上流のドキュメントを修正します。

## よくある誤解

1. **「ベストプラクティスはチームのためでソロ開発者のためではない。」** — ソロ開発者も明確な仕様書と現在のプロジェクト憲章からチームと同じくらい恩恵を受けます。規律は結果物を読む他の人がいるかどうかにかかわらず、思考の明確さを強制します。
2. **「AIが漠然とした受け入れ基準を発見してくれる。」** — AIは書かれた基準をもとに実装します。測定可能な基準を特定してそうでないものを拒否しません。それが人間のレビュアーの役割です。
3. **「すべてのコマンドを使えばSpec Kitを正しく使っている。」** — コマンドを実行することは必要ですが十分ではありません。Spec Kitを効果的にするのは実行されたコマンドの数ではなく、各ステップでのレビューの品質です。
