# Spec Kitのリスクと限界

> Spec Kitは構造化されたAI支援開発のための強力なツールですが、プロダクト思考の代替にはならず、実装の品質を保証できず、小規模または探索的な作業には意味のある負担を追加します。

## このページの対象読者

このページは、Spec Kitが何をしないか、そして価値を減らすよりも摩擦を生み出す可能性が高い状況を正直に理解したい開発者とチームリードを対象としています。Spec Kitのワークフローへの馴染みがあることを前提としています。

## なぜこれが重要か

すべてのツールには価値を加えるスコープと、相応の恩恵なしにオーバーヘッドだけを追加するスコープがあります。適していない状況を含めてすべての状況にSpec Kitを適用すると、フラストレーションが生まれ、時間が無駄になります。限界を理解することで、役立つ場所では活用し、役立たない場所では避けられるようになります。

## わかりやすい説明

Spec Kitは仕様書を書けるほど要件が十分に明確な機能のAI支援実装を構造化することが得意です。明確な要件を生み出す思考の代替、AIアウトプットの品質の保証、小規模または使い捨ての作業の高速化は得意ではありません。

以下のリスクと限界はSpec Kitを避けるべき理由ではありません。Spec Kitが平均より少ない価値を提供する、またはSpec Kitの誤用が状況を良くするよりも悪くする可能性がある条件です。

## 核心概念

### Spec Kitはプロダクト思考の代替にならない

Spec Kitは開発プロセスを構造化します。何を作るかは決定しません。何が欲しいかわからないチームは混乱を反映した仕様書を生み出し — それを正確に実装します。

**なぜ重要か**: Spec Kitの大きな手戻りの最も一般的な原因は、十分なプロダクト思考なしに承認された仕様書です。AIは仕様書に書かれたものを正確に実装します。仕様書が間違っていれば、実装も間違っています。

**例**: チームがどのイベントが通知をトリガーすべきか、通知の対象が誰か、プッシュ・メール・インアプのどれにすべきかで合意する前に通知機能の仕様書を書きます。結果の仕様書は内部的には一貫していますが、製品には合いません。Spec Kitはそれを忠実に実装して正しいが使えない機能を生み出します。

**緩和方法**: `/speckit-specify`を実行する前にプロダクトに関する質問を解決します。Spec Kitは開発ツールで、製品設計ツールではありません。

### 弱い仕様書は弱い実装を生み出す

実装の品質は仕様書の品質によって制限されます。Spec Kitは仕様書の品質を増幅します — 良い仕様書は良い実装を生み出し、曖昧な仕様書は曖昧な実装を生み出します。

**なぜ重要か**: 開発者はときどきSpec Kitが自分が曖昧にしておいた詳細を「うまくやってくれる」と信じます。そうではありません。曖昧な部分を含めて仕様書に書かれたものを実装します。

**例**: 「フィルターが正しく動作する」という受け入れ基準は、すべての意味のある詳細をAIの裁量に委ねます。AIはある定義では「動作する」フィルターを生み出します。望んだ方法で動作するかどうかは、その定義が意図とどれだけ一致するかによります。

**緩和方法**: 計画の前に`/speckit-clarify`を実行して曖昧さを表面化します。仕様書を承認する前にすべての受け入れ基準の測定可能性をレビューします。

### 小規模実験のプロセスオーバーヘッド

完全なSpec Kitパイプライン（8ステップ、3つのレビューされた成果物）はコードを書く前に相当な時間投資を必要とします。小規模実験、使い捨てプロトタイプ、または1日の寿命の機能では、このオーバーヘッドが提供する価値を超える可能性があります。

**なぜ重要か**: 2行の変更に完全なパイプラインを適用することは無駄です。このプロセスはパッチではなく機能のために設計されています。

**例**: 開発者がコミットする前にどちらがより良い感じかを試してみるために2つのUIアプローチを試したいと思っています。各アプローチに完全なSpec Kitパイプラインを実行することはやり過ぎです。探索段階では直接プロンプティング — または手動コーディング — の方が適切です。

**緩和方法**: ワークフローを機能に合わせます。リスクの低い機能には簡略化されたワークフロー（4ステップ）を使い、使い捨て実験には直接プロンプティングを検討します。

### AIエージェントの変動性

明確な仕様書と計画があっても、AIのアウトプットは決定論的ではありません。同じ仕様書でも、異なるセッション、異なるモデル、異なるClaude Codeバージョンで若干異なる実装を生み出す場合があります。

**なぜ重要か**: Spec KitはAIの変動性を減らしますが、なくしません。チームは同じ仕様書から異なる実行で同一のアウトプットを期待すべきではありません。

**例**: 同じ`tasks.md`で`/speckit-implement`を2回実行すると、すべての受け入れ基準を満たすが異なる変数名、異なる関数シグニチャ、異なるエラーハンドリングアプローチを使うコードが生み出される場合があります。

**緩和方法**: 期待される正確なアウトプットに対してではなく、受け入れ基準に対して実装のアウトプットをレビューします。仕様書はターゲットを定義します。ターゲットへの正確な道筋は異なる場合があります。

### 成果物の整合性維持には規律が必要

Spec Kitは生成時の機能を説明する3つのパイプラインアーティファクト（`spec.md`、`plan.md`、`tasks.md`）を生成します。実装が進み要件が進化するにつれて、これらの成果物は実際のコードと同期が取れなくなる場合があります。

**なぜ重要か**: 仕様書ドリフト（spec drift）— 仕様書が実際に実装された方法と異なる形で機能を説明する状態 — はSpec Kitが防ぐよう設計されたドキュメント-コード同期問題の特定の形態です。チームが要件が変わったときに仕様書を更新する規律を維持しなければ、成果物が役立つよりも誤解を招くものになります。

**緩和方法**: 仕様書の更新を開発プロセスの一部として扱います。要件が変わったとき、まず`spec.md`を更新し、必要に応じて計画とタスクを更新してから再実装します。

## たとえ話

> GPS ナビゲーションシステムは持っている情報をもとに最適なルートを教えてくれます。地図データが古ければ、自信を持って間違ったルートを案内します。ターンバイターンの案内を無視すれば、助けることができません。存在しない住所にナビゲーションを求めれば、正常に失敗します。

Spec KitがGPSです。仕様書が地図データです。プロダクト思考はどの目的地を入力するかを知ることです。ワークフローに従う規律が案内に従うことです。システムが機能するためには4つすべてが必要です。

## 実際の活用事例

**Spec Kitが適切なツールであるシグナル**：
- ユーザーストーリーと受け入れ基準を書けるほど要件が十分に明確な機能
- コードを書く前に複数の人が何を作るかで合意する必要がある場合
- 数週間以上維持される機能
- 実装後の修正が高コストな場合

**別のアプローチが良いシグナル**：
- 決定されたソリューションを実装するのではなく、どのソリューションを使うか探索している場合
- 変更が小さすぎて仕様書が必要ない場合（1時間以内の作業）
- 要件が不明確すぎて仕様書を書くことが推測になる場合
- 機能が使い捨ての場合（プロトタイプ、スパイク、実験）

## よくある誤解

1. **「Spec Kitが悪いアウトプットを出したらツールが壊れている。」** — 悪いアウトプットは通常、悪い仕様書、古いプロジェクト憲章、またはスキップされたレビューステップに起因します。ツールは設計通りに動いています。改善が必要なのは入力です。
2. **「Spec Kitは正しい実装を保証する。」** — Spec Kitは構造を提供し曖昧さを減らします。正確性は保証しません。受け入れ基準に対する実装の人間によるレビューは常に必要です。
3. **「Spec Kitコマンドを多く実行するほど、アウトプットが良くなる。」** — 多くのコマンドを実行しても品質の低い入力を補うことはできません。簡略化された4ステップワークフローの良い仕様書は、完全な8ステップワークフローの曖昧な仕様書より優れています。
