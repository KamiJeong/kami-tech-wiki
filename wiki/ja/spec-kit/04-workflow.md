# Spec Kitワークフロー

> Spec Kitの完全なワークフローは8つのコマンドを順番に実行します — プロジェクトルールの確立（プロジェクト憲章）から実装の検証まで。リスクの低い機能には4ステップの簡略化バージョンが使えます。

## このページの対象読者

このページは、完全なSpec Kitワークフロー、完全パイプラインと簡略化バージョンをいつ使うべきか、各ステップが何を生み出すかを理解したい開発者とチームリードを対象としています。このページを読む前に[Spec Kitコアコンセプト](03-core-concepts.md)に馴染んでおくことをお勧めします。

## なぜこれが重要か

ワークフローの順序を知ることが重要な理由は、各ステップが前のステップの上に構築されるからです。レビューされていない計画に対して実装するということは、レビューされていない設計に対して実装することを意味します。明確化されていない仕様書に対して計画するということは、曖昧な仕様書に対して計画することを意味します。順序は任意ではありません — 各ステップは後で修正が高コストなときではなく、安価なときに特定の種類の不確実性を解消するために存在します。

完全パイプラインと簡略化バージョンをいつ使うべきかを知ることも重要です。3行の変更に8ステップを適用するのは無駄です。6つのエッジケースがある機能に4ステップを適用するのはリスクがあります。どちらを選ぶかは各ステップが何をするかの理解が必要です。

## わかりやすい説明

Spec Kitには2つのワークフローオプションがあります：

**完全ワークフロー**（8ステップ）：意味のある複雑さの機能、横断的な関心事がある機能、または実装前にチームのレビューが必要な機能。

```
プロジェクト憲章 → 仕様書作成 → 明確化 → チェックリスト → 計画 → タスク → 分析 → 実装
```

**簡略化ワークフロー**（4ステップ）：明確な要件とシンプルな実装を持つリスクの低い機能。

```
仕様書作成 → 計画 → タスク → 実装
```

実装後、検証ステップ（`/speckit-verify-run`）が実装が仕様書と一致しているかを確認します。このステップは両方のワークフローに適用されます。

## 核心概念

### 完全ワークフロー — 8ステップ

**ステップ1：プロジェクト憲章** (`/speckit-constitution`)
プロジェクトのガバナンスルールを確立します。プロジェクト開始時に一度実行します。それ以降のすべての仕様書が継承します。プロジェクトのプロジェクト憲章がすでに存在する場合はこのステップをスキップします。

**ステップ2：仕様書作成** (`/speckit-specify`)
仕様書を作成します。平易な言語で機能を説明すると、Spec Kitはユーザーストーリー、受け入れ基準、スコープ外項目、エッジケースを含む構造化された`spec.md`を生成します。

**ステップ3：明確化** (`/speckit-clarify`)
仕様書の曖昧さを特定して解消します。Spec Kitが不明確な領域について最大5つの目標が定まった質問をします。答えが`spec.md`にエンコードされます。

**ステップ4：チェックリスト** (`/speckit-checklist`)
仕様書に対する品質チェックリストを生成します。すべての必須仕様書セクションが完成しているか、受け入れ基準が測定可能か、エッジケースが対処されているかを確認します。

**ステップ5：計画** (`/speckit-plan`)
実装計画を生成します。Spec Kitが仕様書を読み、技術的アプローチ、ファイル構造、段階的な実装順序が含まれた`plan.md`を生み出します。

**ステップ6：タスク** (`/speckit-tasks`)
タスクリストを生成します。Spec Kitが計画を読み`tasks.md`を生み出します — 順序付きで独立して完了可能な実装ステップのチェックリスト。

**ステップ7：分析** (`/speckit-analyze`)
すべての成果物の一貫性をクロスチェックします。`spec.md`、`plan.md`、`tasks.md`が互いに一貫しているか、すべての受け入れ基準が最低1つのタスクでカバーされているかを確認します。

**ステップ8：実装** (`/speckit-implement`)
タスクリストを実行します。AIが順番に各タスクを実装し、成果物を仕様書と計画に対して確認します。完了したタスクは`tasks.md`で`[X]`とマークされます。

### 簡略化ワークフロー — 4ステップ

明確な要件、孤立したスコープ、シンプルな実装を持つ機能では、簡略化ワークフローが明確化・チェックリスト・分析をスキップします：

1. `/speckit-specify` — 仕様書を作成
2. `/speckit-plan` — 実装計画を作成
3. `/speckit-tasks` — タスクリストを生成
4. `/speckit-implement` — タスクを実行

**簡略化ワークフローを使うとき**：
- 機能が明確な要件でよく理解されている
- スコープが狭い（1つまたは2つのファイルへの影響）
- チームが実装前にアプローチをレビューする必要がない
- 機能がリスクが低い（何かが間違っても修正しやすい）

**完全ワークフローを使うとき**：
- 機能が複数のコンポーネントやファイルを触る
- 要件が部分的に不明確かステークホルダーの整合が必要
- 実装アプローチがコードが書かれる前にチームのレビューが必要
- 機能がリスクが高い（何かが間違うと修正が難しい）

## たとえ話

> 完全ワークフローは大陸横断ロードトリップを計画するようなものです。地図を確認し、目的地を確定し、曖昧な区間を把握し、給油ポイントを計画し、車に荷物を積み、ルートをレビューしてから運転します。簡略化ワークフローは2ブロック先のスーパーマーケットに行くようなものです。どこに行くかを知っていて、ルートが慣れ親しんでいて、間違った方向に行くコストが低いです。

どちらのアプローチが普遍的に正しいわけではありません。カギは計画の努力を旅の複雑さとリスクに合わせることです。

## 実際の活用事例

**適切なワークフローの選択**：

| 機能の特性 | 推奨ワークフロー |
|-----------|---------------|
| 明確な要件、狭いスコープ | 簡略化（4ステップ） |
| 不明確な要件、複数のステークホルダー | 完全（8ステップ） |
| 複数のファイル/コンポーネントへの影響 | 完全（8ステップ） |
| 単一ファイルの変更、孤立 | 簡略化（4ステップ） |
| リスクが高い（認証・決済・データ移行） | 完全（8ステップ） |
| リスクが低い（UIテキスト・スタイル更新） | 簡略化（4ステップ） |

**各ステップでレビューする**: ワークフローのすべてのステップで、次のコマンドを実行する前に成果物をレビューします。ワークフローの価値は各ステップで立ち止まって読む人間のレビューから来ます — 途中で止まらずにすべてのコマンドを順番に実行することではありません。

**いつ立ち止まって再明確化するか**: 後のステップで初期成果物が間違っていたことが判明した場合（例えば、計画が仕様書の曖昧さを明らかにする）、続行する前に初期成果物を修正します。ワークフローは反復可能に設計されています、一度限りのプロセスではありません。

## よくある誤解

1. **「完全ワークフローは常により良い。」** — 些細な変更に8ステップを適用すると不必要なオーバーヘッドが生まれます。機能の複雑さにワークフローを合わせます。
2. **「中間成果物をレビューせずにすべてのステップを実行できる。」** — 成果物をレビューせずにコマンドを実行することが、Spec Kitが悪い結果を生み出す最も一般的な理由です。レビューはオプションではありません — それが核心です。
3. **「ワークフローは実装で終わる。」** — 実装が終わりではありません。検証ステップ（`/speckit-verify-run`）が実装が仕様書と一致しているかを確認します。スキップすると、タスクを満たしているが意図を見逃しているコードをリリースする可能性があります。
