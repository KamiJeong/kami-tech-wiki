# Spec Kitコアコンセプト

> Spec Kitのワークフローは5つのコアコンセプトの上に構築されています。プロジェクト憲章（プロジェクトのルール）、仕様書（何を作るか）、計画（どうやって作るか）、タスク（作業単位）、品質ゲート（正しくできたか？）。

## このページの対象読者

このページは、Spec Kitワークフローの各ステップが何を生み出すか、なぜ各ステップが存在するかを理解したい開発者とチームリードを対象としています。ソフトウェア開発コンセプトへの基本的な理解があると役立ちます。

## なぜこれが重要か

Spec Kitのコアコンセプトを理解することで、機械的にではなく意図的にツールキットを使えるようになります。各成果物が何のためにあり、良いものがどのような見た目かを知ることが、安定した結果を出すSpec Kitワークフローと、単に同じ曖昧なプロンプトをより整形式にしたものを生み出すワークフローの違いを作ります。

## わかりやすい説明

Spec Kitのワークフローはパイプラインアーティファクトと呼ばれる一連のドキュメントを生み出します。各成果物は特定の質問に答えます：

- **プロジェクト憲章（Constitution）** → このプロジェクトは常にどんなルールに従うか？
- **仕様書（spec.md）** → この機能は何で、「完了」はどのような状態か？
- **計画（plan.md）** → この機能をどのように作るか？
- **タスク（tasks.md）** → 個別のステップは何か？
- **品質ゲート** → 実装は仕様書と一致しているか？

これらのドキュメントは順序通りに生成され、それぞれが前のものの上に構築されます。合わせると機能開発の完全な文書化された記録を形成します。

## 核心概念

### プロジェクト憲章（Constitution）

プロジェクト憲章（constitution）は、すべての機能が従うべきルールを定義するプロジェクト全体のガバナンスドキュメントです。技術的な選択、コーディング規約、品質基準、アーキテクチャパターン、レビュー要件。プロジェクトレベルで一度書かれ、それ以降のすべての仕様書に継承されます。

**なぜ重要か**: プロジェクト憲章なしでは、すべての新機能仕様書が同じ基本ルールを再び述べる必要があります。プロジェクト憲章があれば、それらのルールは自動的に継承されます。仕様書はこの機能にだけ特有のことに集中できます。

**例**: プロジェクト憲章にはこう書けます。「すべてのAPIはTypeScriptで実装しなければならない。すべてのデータベースクエリはパラメータ化された構文を使用しなければならない。すべての機能は実装を始める前にspec.mdが必要だ。」すべての機能仕様書はこれらのルールを繰り返すことなく継承します。

**生成コマンド**: `/speckit-constitution`

### 仕様書（spec.md）

仕様書は1つの機能を説明する構造化ドキュメントです。何か、誰のためか、「完了」はどのような状態か（受け入れ基準）、明示的にスコープ外のものは何か、どのエッジケースを処理すべきか。

**なぜ重要か**: 仕様書はそれ以降のすべてのステップへの基本的な入力です。明確で正確な仕様書は明確な計画と安定した実装を生み出します。曖昧な仕様書は曖昧な計画と修正が必要な実装を生み出します。

**例**: タスクフィルター機能の仕様書には次が含まれます。ユーザーストーリー（誰がフィルターを使いなぜか）、受け入れ基準（「アクティブ」でフィルタリングするとステータスが「アクティブ」のタスクのみ表示される）、スコープ外項目（ソートはこの機能のスコープに含まれない）、エッジケース（フィルター結果がない場合どうなるか）。

**生成コマンド**: `/speckit-specify`

### 計画（plan.md）

計画は仕様書から作られる技術実装ドキュメントです。技術スタック、ファイル構造、アーキテクチャアプローチ、段階的な実装順序を説明します。

**なぜ重要か**: 実装前に計画をレビューすることで、チームはコードが書かれた後では修正が高コストなアーキテクチャミスを、コードが書かれる前に発見できます。計画はまたAIを制約します。AIは自分のアーキテクチャの好みではなく計画から作業します。

**例**: タスクフィルター機能の計画にはこう書けます。フィルターは`filters.ts`ファイルに純粋関数として実装され、UIはfilter propがある既存の`TaskList`コンポーネントを使い、フィルター状態はstateライブラリなしに親コンポーネントで管理されます。

**生成コマンド**: `/speckit-plan`

### タスク（tasks.md）

タスクリストは計画から導き出された順序付きの独立して完了可能な実装ステップのチェックリストです。各タスクは検証できるほど小さく、完了がいつかわかるほど具体的で、仕様書の要件に追跡可能です。

**なぜ重要か**: レビューされた仕様書から導き出されたタスクリストはAIにスコープが定まってレビュー可能な作業単位を提供します。スコープクリープを防ぎ（AIが計画されたもの以上を実装すること）、進捗を可視化します（どのタスクが完了し、何が残っているか確認可能）。

**フィルター機能のタスク例**:
- `filterByStatus(tasks, status)`関数を含む`filters.ts`を作成
- `filter` propを受け取るよう`TaskList`コンポーネントを更新
- メインUIにフィルターボタン（全件・アクティブ・完了）を追加
- 空のデータセット・部分データセット・全件データセットで`filterByStatus`のテストを書く

**生成コマンド**: `/speckit-tasks`

### 品質ゲート（Quality Gate）

品質ゲートは実装が仕様書を満たしているかを確認する検証ステップです。完了したタスクを受け入れ基準に対してレビューし、満たされていない要件を特定します。

**なぜ重要か**: 品質ゲートなしでは、タスクを完了とマークして、そのタスクが満たすべきだった受け入れ基準がまだ失敗していることを見逃しがちです。品質ゲートは開発者の記憶に頼る代わりに検証を体系的にします。

**例**: 品質ゲートは確認します。「アクティブ」でフィルタリングするとステータスが「アクティブ」のタスクのみ表示されるか？ユーザーが移動して戻ってきたときフィルターがリセットされるか？spec.mdの各受け入れ基準が最低1つの実装された動作で満たされているか？

**実行コマンド**: `/speckit-verify-run`

## たとえ話

> Spec Kitで機能を作ることは絵を依頼することに似ています。プロジェクト憲章はギャラリーの常時ルールです（5フィート以下、アーカイブ素材を使用）。仕様書は依頼のブリーフィングです（特定の視点から夜明けの港の風景）。計画はアーティストのスケッチです（大まかな構図、カラーパレット、レイヤーの順序）。タスクは日々の作業指示です（キャンバスを準備し、空を塗り、水を追加する）。品質ゲートは絵が納品される前の依頼ブリーフィングに対する最終検査です。

各ステップは前のステップが提起した質問に答えます。各ステップは次が始まる前にレビューされます。

## 実際の活用事例

**プロジェクト憲章のセットアップ（プロジェクトごとに一度）**:
- `/speckit-constitution`を実行
- 文書化する：技術スタック、コーディング規約、必須の品質基準、レビュープロセス
- gitにコミット — それ以降のすべての仕様書が継承

**機能ごとのワークフロー**:
1. `/speckit-specify` — 仕様書を作成；続行前に注意深くレビュー
2. `/speckit-clarify` — 曖昧さを解消；仕様書を更新
3. `/speckit-plan` — 実装前に技術アプローチをレビュー
4. `/speckit-tasks` — タスクリストをレビュー；すべての受け入れ基準がカバーされているか確認
5. `/speckit-implement` — 実装を進める；質問が出たら答える
6. `/speckit-verify-run` — 実装が仕様書を満たしているか確認

## よくある誤解

1. **「プロジェクト憲章はREADMEだ。」** — READMEはプロジェクトを人間の読者に説明します。プロジェクト憲章はSpec Kitパイプラインが読んで施行するガバナンスドキュメントです。それ以降のすべての仕様書と実装を制約します。
2. **「仕様書が十分に詳細ならplan.mdはオプションだ。」** — 計画は仕様書を技術的アプローチに変換します。計画なしでは、AIはアプローチを自由に選択します — これはプロジェクトのアーキテクチャと衝突する可能性があります。計画レビューステップがアーキテクチャの整合が行われる場所です。
3. **「タスクはただのチェックリストだ。」** — タスクはAIの作業指示です。各タスクはAIが正確に何をすべきかを知り、開発者が正確に何を検証すべきかを知れるほど具体的でなければなりません。曖昧なタスク（「フィルタリングを実装する」）はタスクではありません — 残りのすべての思考をAIに委任することです。
