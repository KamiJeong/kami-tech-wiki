# ハンズオンチュートリアル：Spec Kitで最初の機能を作る

> 個人タスク管理アプリを練習プロジェクトとして使い、仕様書の作成から検証済みの実装まで完全なSpec Kitワークフローを実行します。

## このページの対象読者

このページは、Spec Kitの概要記事を読み終え、具体的な練習プロジェクトでワークフローを直接実行してみたい開発者を対象としています。Claude Codeが環境で実行中である必要があります。コードエディターとターミナルへの基本的な理解があることを前提としています。

## なぜこれが重要か

Spec Kitを一度実行してみること — 本物の仕様書を書き、本物の計画をレビューし、AIが本物の機能を実装するのを見る — は、読むだけでは得られない方法でコンセプトを具体的にします。このチュートリアルは安全な練習環境でその体験を提供するよう設計されています。

## わかりやすい説明

このチュートリアルはSpec Kitの簡略化された4ステップワークフロー（仕様書 → 計画 → タスク → 実装）を使って個人タスク管理アプリを作る過程をガイドします。練習プロジェクトには4つの機能が含まれます。タスクを追加、完了としてマーク、ステータスでフィルタリング、削除。

完了すると、1つの完全なSpec Kitワークフローサイクルを実行し、`spec.md`、`plan.md`、`tasks.md`の3つのパイプライン成果物を生み出したことになります。

## 核心概念

### 練習プロジェクトのスコープ

個人タスク管理アプリには：
- タイトル付きタスクの追加
- タスクを完了としてマーク
- タスクのフィルタリング（全件表示 / アクティブ表示 / 完了表示）
- タスクの削除

これが完全なスコープです。このチュートリアルではこの4つの機能以外はスコープ外です。

### 最新情報について

> **2026-05-19時点**: Spec Kitはこのリポジトリのクロードコードスキルとして実装されています。正確なコマンド呼び出し方法はClaude Codeの設定によって異なります。コマンドが示された通りに動作しない場合は、現在の呼び出し形式についてClaude Codeのドキュメントまたは`.claude/skills/`ディレクトリを確認してください。

## たとえ話

> このチュートリアルを実行することは、高速道路に行く前に駐車場で運転を学ぶようなものです。駐車場（このチュートリアル）はシンプルでリスクの低いシナリオを使います。高速道路（実際のプロジェクト機能）は同じスキルを使いますが、結果がより重要です。駐車場での練習が高速道路を安全にします。

## 実際の活用事例

### 事前準備

- 開発環境でClaude Codeが実行中
- Claude Codeでプロジェクトディレクトリが開かれている
- Spec Kitスキルが使用可能（Claude Codeで`/speckit-specify`が応答するか確認）

### ステップ1：仕様書を作成する

Claude Codeで実行します：

```
/speckit-specify
```

説明を入力するよう求められたら次を入力します：

```
個人タスク管理アプリを作って。ユーザーが次のことができるようにしたい：
- タイトル付きのタスクを追加
- タスクを完了としてマーク
- タスクのフィルタリング：全件・アクティブ（未完了）のみ・完了のみを表示
- タスクを削除

シングルユーザーアプリ。アカウントなし。セッションをまたいでタスクが保存される（localStorage）。
```

**成果物レビュー**: `spec.md`を開きます。次を確認します：
- 4つの機能すべてがユーザーストーリーとして現れているか
- 各ユーザーストーリーに測定可能な受け入れ基準があるか
- 「アカウントなし」と「localStorage」がスコープ制約として現れているか
- エッジケースが対処されているか（空のタイトル、空のリスト、結果なしのフィルター）

**よくある間違い**: 受け入れ基準を読まずに仕様書を承認すること。AIが測定可能でない基準（「タスクが追加される」）ではなく「タスクは一意のIDと入力したタイトルと共にリストに即座に表示される」のような基準を書いているかもしれません。続行前に修正します。

**期待される成果物**: 4つのユーザーストーリーと、それぞれに具体的な受け入れ基準を持つ`specs/task-tracker/spec.md`。

### ステップ2：実装計画を作成する

Claude Codeで実行します：

```
/speckit-plan
```

Spec Kitが`spec.md`を読んで`plan.md`を生成します。次をレビューします：
- 提案された技術スタックがプロジェクトのスタックと一致しているか？
- タスクのデータモデル（フィールド、型）が期待通りか？
- ファイル構造が合理的か？
- 同意できないアーキテクチャの選択があるか？

**何かが間違っている場合**: `spec.md`を修正するかSpec Kitに変更内容を伝えて`/speckit-plan`を再実行します。

**期待される成果物**: 技術スタックの決定、タスクデータモデル、提案されたファイル構造、段階的な実装アプローチを含む`specs/task-tracker/plan.md`。

### ステップ3：タスクリストを生成する

Claude Codeで実行します：

```
/speckit-tasks
```

Spec Kitが`plan.md`を読んで`tasks.md`を生成します。次をレビューします：
- タスクが論理的な順序か？（UIコンポーネントの前にデータモデル、など）
- `spec.md`のすべての受け入れ基準が最低1つのタスクに対応しているか？
- タスクが具体的か？（「`filterByStatus()`関数を作成」は具体的で、「フィルタリングを実装する」は具体的でない）

**期待される成果物**: 6〜10の具体的で順序付けられた実装タスクを持つ`specs/task-tracker/tasks.md`。

### ステップ4：実装する

Claude Codeで実行します：

```
/speckit-implement
```

Spec Kitがタスクリストを処理するのを見てください。各タスクは1つ以上のファイル変更に対応します。Spec Kitは各タスクが完了したとき`[X]`とマークします。

**実装中**: そばにいてください。仕様書の曖昧さに遭遇した場合、Spec Kitが質問します。仕様書をもとに答えます — 答えに仕様書の変更が必要なら`spec.md`で変更してから確認します。

**期待される成果物**: `tasks.md`のすべてのタスクが`[X]`とマークされ、指定されたすべてのファイルが作成または更新されている。

### ステップ5：実装を検証する

`spec.md`の各受け入れ基準をテストします：
- タイトル付きのタスクを追加できるか？
- 一意のIDと共にタスクが即座に表示されるか？
- タスクを完了としてマークできるか？
- フィルターは正しいサブセットのタスクを表示するか？
- タスクを削除できるか？

**機能が動かない場合**: `spec.md`の関連する受け入れ基準をより正確に更新し、タスクリスト全体ではなく影響を受けたタスクに対して`/speckit-implement`を再実行します。

**オプション**: すべての受け入れ基準の体系的な確認のために`/speckit-verify-run`を実行します。

### ステップ6：次のステップ

このチュートリアルを完了後：
- ステップ1と2の間に`/speckit-clarify`、`/speckit-checklist`、`/speckit-analyze`を追加した完全ワークフローを試す
- 既存のプロジェクトの実際の機能に同じワークフローを適用する
- Spec Kitを最大限に活用するためのアドバイスについて[ベストプラクティス](07-best-practices.md)記事を探る

## よくある誤解

1. **「Spec Kitが生成するすべてのファイルを理解する必要がある。」** — 実装ファイルではなく仕様書ドキュメント（`spec.md`、`plan.md`、`tasks.md`）に集中します。あなたの役割は生成されたすべてのコード行を読むことではなく、実装が受け入れ基準を満たしているかをレビューすることです。
2. **「Spec Kitがミスをしたらチュートリアルが壊れた。」** — AIのアウトプットは完璧ではありません。レビューして、仕様書を修正して、再実装します。これが意図されたワークフローで、失敗モードではありません。
3. **「簡略化ワークフローはシンプルな機能のためだ。」** — 簡略化ワークフローは明確で曖昧さのない要件を持つ機能のためです。実装の複雑さは要件の複雑さとは別です。明確な要件の複雑な機能は簡略化ワークフローを使えます。不明確な要件のシンプルな機能は完全ワークフローが必要かもしれません。
