# Kiro 훅 가이드

> Kiro 훅(hooks)은 파일 시스템 이벤트에 의해 트리거되는 자동화된 AI 작업으로, 무언가 변경될 때 관련 파일을 자동으로 업데이트해 수동 유지보수 작업을 줄여줍니다.

## 이 페이지의 대상 독자

이 페이지는 Kiro에서 자동화된 AI 워크플로를 설정하는 방법을 이해하고 싶은 개발자를 위한 글입니다. 파일 시스템과 "트리거" 개념(이메일 필터가 작동하는 방식과 유사)에 대한 기본적인 이해가 있으면 도움이 됩니다.

## 왜 이 내용이 중요한가

소프트웨어 프로젝트는 일관성 유지 문제를 안고 있습니다. 한 파일이 바뀌면 관련 파일도 바뀌어야 하는 경우가 많습니다. 개발자가 API 엔드포인트를 수정하면 테스트도 업데이트되어야 합니다. 새 컴포넌트를 추가하면 문서도 업데이트되어야 합니다. 데이터 모델을 변경하면 마이그레이션 스크립트도 업데이트되어야 합니다.

자동화 없이는 이 업데이트들이 개발자의 성실함에 의존합니다. 마감 압박 아래서의 성실함으로는 이것들이 자주 누락됩니다. 테스트가 테스트하는 코드와 동기화되지 않습니다. 문서가 더 이상 설명대로 작동하지 않는 기능을 설명합니다. 결과적으로 테스트와 문서에 대한 신뢰가 시간이 지남에 따라 무너지는 코드베이스가 됩니다.

Kiro 훅이 이 유지보수를 자동화합니다. 특정 패턴과 일치하는 파일이 변경될 때 어떤 AI 작업을 실행할지 정의함으로써, 팀은 수동 노력 없이 관련 파일들을 동기화 상태로 유지할 수 있습니다.

## 쉬운 설명

훅은 `.kiro/hooks/`에 저장된 설정 파일입니다. 다음을 정의합니다:
- **언제**: 어떤 파일 이벤트가 이 훅을 트리거하는지 (저장 시, 생성 시, 삭제 시, 파일 패턴 일치 시)
- **무엇**: 이벤트 발생 시 어떤 AI 작업을 실행할지

개발자가 훅의 패턴과 일치하는 파일을 저장하면, Kiro의 AI가 정의된 작업을 자동으로 실행합니다. 개발자는 적용되기 전에 AI의 결과물을 검토합니다 — 훅이 작업을 실행하지만 개발자가 여전히 결과를 승인합니다.

훅은 YAML 또는 유사한 설정 형식으로 정의되며 `.kiro/hooks/`에 저장됩니다. 스티어링 파일처럼 git에 커밋되어 팀 전체에 공유됩니다.

## 핵심 개념

### 파일 이벤트 트리거

훅은 파일 시스템 이벤트 — 특히 지정된 패턴과 일치하는 파일의 변경 — 에 의해 트리거됩니다. 일반적인 트리거 이벤트에는 저장 시(파일이 저장됨), 생성 시(새 파일이 생성됨), 삭제 시(파일이 삭제됨)가 포함됩니다.

**왜 중요한가**: 파일 이벤트를 기반으로 훅을 설정한다는 것은 실제 개발자 활동에 반응해 실행된다는 의미입니다. 관련 작업이 진행될 때 훅이 실행되며, 예약된 타이머나 수동 호출이 아닙니다.

**예시**: `api/**/*.ts`(api/ 디렉토리 내부의 모든 TypeScript 파일)를 트리거로 설정한 훅은 개발자가 API 파일에 변경을 저장할 때마다 실행됩니다.

### 범위가 정해진 AI 작업

각 훅은 구체적이고 범위가 정해진 AI 작업을 정의합니다 — 열린 형식의 지시가 아닙니다. 잘 작성된 훅 작업은 좁습니다: "발견된 문제를 수정해줘"가 아닌 "이 파일의 테스트를 업데이트해줘".

**왜 중요한가**: 범위가 정해진 작업은 예측 가능하고 검토 가능합니다. API 변경에 대응해 특정 테스트 파일 하나를 업데이트하는 훅이라면, 개발자는 정확히 무엇을 검토해야 할지 압니다. 열린 형식의 훅 작업은 평가하고 신뢰하기 더 어려운 예측 불가능한 결과물을 만들어냅니다.

**훅의 작업 예시**: "업데이트된 API 파일과 해당 테스트 파일을 읽으세요. 새로운 또는 변경된 API 메서드를 반영하도록 테스트 파일을 업데이트하세요. API 변경의 직접적인 영향을 받지 않는 테스트는 수정하지 마세요."

### 잘못 정의된 훅의 위험

너무 광범위하거나, 너무 느슨하게 트리거되거나, 많은 파일을 수정하는 훅은 혼란을 만들 수 있습니다. 모든 파일 저장 시 실행되어 수십 개의 파일에 접근하는 훅은 시끄럽고 검토하기 어렵습니다.

**왜 중요한가**: 훅의 목표는 유지보수 부담을 줄이는 것이지, 새로운 검토 부담을 만드는 것이 아닙니다. 잘못 설계된 훅은 대체하려 했던 수동 업데이트보다 검토하기 더 어려운 AI 변경사항을 만들어낼 수 있습니다.

**위험 예시**: 모든 파일 저장 시 트리거되어 모든 문서를 동시에 업데이트하려는 훅은 지속적으로 실행되어 검토하기 어려운 대규모 diff를 만들어냅니다.

## 비유와 예시

> 건물의 냉난방 시스템이 작동하는 방식을 생각해보세요. 온도 센서가 이벤트(방이 너무 차가워짐)를 감지하고 자동화된 응답(히터가 켜짐)을 트리거합니다. 응답은 범위가 정해지고 구체적입니다 — 히터이지, 조명이나 잠금장치가 아닙니다.

Kiro 훅도 같은 방식으로 작동합니다. 파일 이벤트(API 파일이 저장됨)가 특정 AI 작업(해당 테스트 업데이트)을 트리거합니다. 응답은 범위가 정해져 있습니다. 개발자는 결과가 적용되기 전에 검토합니다.

## 실제 활용 사례

**테스트 동기화 훅 설정**:
1. `.kiro/hooks/sync-tests.yml`(또는 Kiro 버전에 맞는 형식)을 생성합니다.
2. 트리거 설정: `src/**/*.ts` 패턴과 일치하는 파일.
3. 작업 설정: "소스 파일이 변경되면 `tests/`의 해당 테스트 파일을 검토하고 변경된 함수나 타입을 참조하는 테스트를 업데이트하세요."
4. 작은 변경을 만들고 AI가 제안하는 테스트 업데이트를 검토해 훅을 테스트합니다.

**문서 업데이트 훅 설정**:
1. `.kiro/hooks/update-docs.yml`을 생성합니다.
2. 트리거 설정: `api/**/*.ts` 패턴과 일치하는 파일.
3. 작업 설정: "API 파일이 변경되면 `docs/api/`의 해당 항목을 검토하고 변경된 엔드포인트, 매개변수, 반환값을 반영하도록 업데이트하세요."

**훅 결과물 검토**:
훅이 실행되면 Kiro는 적용하기 전에 개발자 검토를 위해 제안된 변경사항을 제시합니다. 훅 결과물을 다른 AI 생성 코드와 같이 다루세요: 읽고, 확인하고, 승인하거나 거부하세요.

## 흔한 오해

1. **"훅이 검토 없이 코드를 자동으로 실행한다."** — 훅은 AI 작업을 트리거하지, 자동으로 코드를 커밋하지 않습니다. 개발자는 여전히 AI의 제안된 변경사항을 검토하고 적용 전에 승인합니다.
2. **"훅이 테스트를 대체한다."** — 훅은 테스트를 최신 상태로 유지하는 데 도움을 줍니다. 테스트를 작성하거나, 올바른 동작이 무엇인지 정의하거나, 테스트 커버리지를 검토해야 하는 필요성을 대체하지는 않습니다.
3. **"훅이 많을수록 자동화가 많다."** — 너무 많은 훅은 노이즈를 만듭니다. 하나 또는 두 개의 높은 가치 훅(테스트 동기화, 문서 동기화)으로 시작하고, 더 추가하기 전에 유지보수 부담을 줄이는지 아니면 늘리는지 평가하세요.

## 심화 내용

> **심화 독자를 위한 내용** — 이 섹션을 읽지 않아도 Kiro 훅을 이해하는 데 지장이 없습니다.

Kiro 훅은 개념적으로 git 훅(pre-commit, post-commit)과 유사하지만, 셸 스크립트가 아닌 AI 작업을 트리거합니다. git 훅과 달리, Kiro 훅은 git 작업이 아닌 파일 저장 이벤트에서 실행됩니다. 이는 커밋 시점이 아닌 개발 도중에 실행될 수 있다는 것을 의미합니다 — 팀이 커밋이 스테이지되기 전에도 파일 동기화를 위한 AI 지원을 받을 수 있게 해줍니다. "테스트 업데이트를 잊은" 커밋의 수를 줄이는 효과가 있습니다.
