# Kiro 명세서(스펙) 워크플로

> Kiro 명세서(스펙)는 세 가지 구조화된 문서 — 요구사항, 설계, 작업 목록 — 로 구성되며, AI가 기능 아이디어에서 검토 가능한 구현까지 나아갈 수 있도록 이끕니다.

## 이 페이지의 대상 독자

이 페이지는 Kiro 명세서의 전체 생명주기를 이해하고 싶은 개발자와 프로덕트 매니저를 위한 글입니다 — 초기 설명 작성부터 완성된 구현 검토까지. 소프트웨어 개발 개념에 대한 기본적인 이해가 있으면 좋지만 필수는 아닙니다.

## 왜 이 내용이 중요한가

명세서는 Kiro 워크플로의 핵심 산출물입니다. Kiro가 하는 모든 것이 명세서로부터 출발합니다. 명세서가 어떻게 작동하는지 — 어떤 문서를 담는지, 서로 어떻게 연결되는지, 어떻게 업데이트하는지 — 를 이해하는 것이 Kiro를 효과적으로 활용하기 위한 토대입니다.

명세서 생명주기를 이해하지 못하면 팀들은 Kiro를 단순한 코드 생성기로 취급하게 되고, 핵심 가치를 놓치게 됩니다. 핵심 가치는 AI의 결과물을 팀의 실제 의도와 일치시키는 구조화된 프로세스에 있습니다.

## 쉬운 설명

Kiro 명세서는 만들고 싶은 기능에 대한 간단한 설명으로 시작합니다. 그 설명으로부터 Kiro의 AI가 세 가지 구조화된 문서를 생성합니다. 그 문서들을 검토하고 수정합니다. 그리고 Kiro가 그 문서들을 바탕으로 기능을 구현합니다.

세 가지 문서는 다음과 같습니다:

- **`requirements.md`** — 사용자가 할 수 있어야 하는 것 (사용자 스토리와 인수 기준)
- **`design.md`** — 기능이 어떻게 만들어질 것인지 (기술 아키텍처, 컴포넌트, 데이터 구조)
- **`tasks.md`** — 설계를 구현하는 순서가 있는 단계 목록

이 문서들은 프로젝트의 `.kiro/specs/[기능명]/` 아래에 저장됩니다. 코드와 함께 버전 관리되므로 명세서 이력과 코드 이력이 동기화됩니다.

## 핵심 개념

### requirements.md (요구사항 문서)

요구사항 문서는 사용자 스토리를 담습니다 — 사용자가 무엇을 하고 싶고 왜 하고 싶은지를 짧게 기술한 문장들. 각 사용자 스토리에는 인수 기준이 있습니다. 요구사항이 충족되었는지 판단하는 측정 가능한 조건들입니다.

**왜 중요한가**: 평문(사용자 스토리)으로 작성된 요구사항은 비기술 이해관계자들이 코드가 작성되기 전에 무엇이 만들어지는지 검토하고 승인할 수 있게 합니다. 인수 기준은 AI에게 구현해야 할 구체적인 목표를 제공합니다.

**예시**:
```
## 사용자 스토리 1: 할 일 추가
사용자로서, 할 일을 추적하기 위해 제목이 있는 작업을 추가할 수 있기를 원합니다.
인수 기준:
- 추가 즉시 작업 목록에 작업이 나타납니다
- 제목이 없는 작업은 제출할 수 없습니다
- 작업에는 자동으로 고유 ID가 부여됩니다
```

### design.md (설계 문서)

설계 문서는 요구사항을 기술 계획으로 변환합니다: 어떤 컴포넌트를 만들지, 어떤 데이터 구조를 사용할지, 어떻게 상호작용할지. 코드가 작성되기 전에 아키텍처 결정이 문서화되는 곳입니다.

**왜 중요한가**: 구현 전에 설계를 작성하면 코드가 작성된 후보다 훨씬 저렴한 시점에 — 변경이 아직 쉬울 때 — 충돌과 불명확한 요구사항이 드러납니다.

**예시**: 할 일 관리 기능의 설계 문서에는 작업이 로컬 JSON 파일에 저장된다, UI는 목록 컴포넌트를 렌더링한다, 각 작업 객체에는 `id`, `title`, `status`, `created_at` 필드가 있다고 명시될 수 있습니다.

### tasks.md (작업 목록 문서)

작업 목록 문서는 설계에서 도출된 순서가 있는 체크리스트입니다. 각 작업은 독립적으로 완료 가능하고, 검증할 수 있을 만큼 구체적이며, 한 세션에서 구현할 수 있을 만큼 작습니다.

**왜 중요한가**: 명세서에서 도출된 작업 목록은 AI에게 명확하고 범위가 정해진 작업 단위를 제공합니다. 각 작업은 정의된 범위를 가지며 설계 문서에 대해 검증될 수 있습니다. AI가 합의된 계획 밖으로 벗어나는 것을 방지합니다.

**예시**:
```
- [ ] id, title, status, created_at 필드를 가진 Task 데이터 모델 생성
- [ ] 제목을 받아 Task를 반환하는 addTask() 함수 구현
- [ ] 모든 작업을 표시하는 renderTaskList() 컴포넌트 구현
- [ ] 사용자 스토리 1에 대한 인수 테스트 작성
```

## 비유와 예시

> 집을 짓는 상황을 생각해보세요. 요구사항 문서는 집주인이 원하는 방과 기능 목록입니다 ("침실 3개, 홈 오피스, 차고"). 설계 문서는 건축가의 도면입니다 — 실제 레이아웃, 치수, 명세. 작업 목록 문서는 시공업자의 작업 지시서입니다 — "1일차에 기초 타설, 2-3일차에 골조 세우기, 4-5일차에 지붕 설치."

각 문서는 서로 다른 질문에 답합니다: 무엇을 원하는가, 어떻게 만들 것인가, 누가 언제 무엇을 하는가. 공사를 시작하기 전에 세 가지 모두가 필요합니다.

## 실제 활용 사례

**단계별 워크플로**:

1. **기능 설명하기**: 만들고 싶은 것을 Kiro에게 평문으로 설명합니다. "작업을 추가하고, 완료하고, 필터링하고, 삭제할 수 있는 개인 할 일 관리 앱을 만들고 싶습니다."

2. **requirements.md 검토하기**: Kiro가 사용자 스토리와 인수 기준을 생성합니다. 주의 깊게 읽으세요. 당신의 의도와 일치하나요? 인수 기준이 측정 가능한가요? 원하는 것을 정확히 표현할 때까지 문서를 수정하세요.

3. **design.md 검토하기**: Kiro가 기술 설계를 생성합니다. 기술적 배경이 있다면 동의하지 않는 아키텍처 선택이 있는지 확인하세요. 비기술적이라면 설계가 요구사항을 충족하는지 확인하세요.

4. **tasks.md 검토하기**: Kiro가 구현 작업 목록을 생성합니다. 작업들이 논리적인 순서로 되어 있는지, 모든 요구사항을 다루는지 확인하세요.

5. **구현하기**: Kiro에게 작업들을 구현하라고 지시합니다. Kiro는 작업 목록을 순서대로 실행하며 각 작업을 요구사항과 설계 문서에 대해 확인합니다.

6. **검토하고 반복하기**: requirements.md의 인수 기준에 대해 구현을 검토합니다. 무언가 잘못됐다면 먼저 명세서를 업데이트하고, 그 다음 Kiro에게 영향받은 작업들을 재구현하도록 요청합니다.

## 흔한 오해

1. **"명세서 문서가 코드 주석과 문서를 대체한다."** — 명세서 문서는 구현 전의 의도를 설명합니다. 코드 주석은 구현이 어떻게 작동하는지를 설명합니다. 둘 다 서로 다른 목적을 가지며 모두 필요합니다.
2. **"명세서를 한번 작성하면 바꾸면 안 된다."** — 명세서는 살아있는 문서입니다. 요구사항이 바뀌거나 구현 과정에서 설계의 결함이 드러나면, 먼저 명세서를 업데이트하고 구현이 따라옵니다. 절대 바뀌지 않는 명세서는 현실과 동떨어진 명세서입니다.
3. **"Kiro가 자동으로 명세서를 작성해준다 — 나는 그냥 승인만 하면 된다."** — Kiro는 당신의 설명을 바탕으로 명세서 초안을 작성합니다. 검토하고 수정하는 것은 당신의 책임입니다. 구현 품질은 전적으로 승인하는 명세서의 품질에 달려 있습니다.

## 심화 내용

> **심화 독자를 위한 내용** — 이 섹션을 읽지 않아도 Kiro 명세서가 어떻게 작동하는지 이해하는 데 지장이 없습니다.

세 문서 명세서 형식(요구사항, 설계, 작업)은 요구사항 명세(무엇)를 설계 명세(어떻게)와 작업 분류 구조(순서)로 분리하는 폭포수 방식에서 영향받은 시스템 엔지니어링 접근법을 반영합니다. Kiro는 명세서를 동결된 승인 산출물이 아닌 살아있는 문서로 취급함으로써 반복적 개발에 맞게 이를 조정했습니다. 고전적 폭포수 방식과의 핵심 차이: Kiro에서 명세서 업데이트는 예상된 것이고 계획된 것입니다. 사전 프로세스의 실패가 아닙니다.
