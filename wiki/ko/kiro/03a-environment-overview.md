# Kiro 핵심 개념: 환경

> Kiro의 환경 개념 — 에이전틱 챗(Agentic Chat), MCP 서버, IDE vs. CLI — 은 AI와 어떻게 상호작용하고 어떤 외부 도구에 접근할 수 있는지를 결정합니다.

## 이 페이지의 대상 독자

이 페이지는 프로덕트 매니저, Kiro를 처음 접하는 개발자, 그리고 Kiro가 외부와 어떻게 연결되는지 이해하고 싶은 분들을 위한 글입니다. 이 페이지를 읽기 전에 [Kiro 핵심 개념: 워크플로](03-core-concepts.md)에서 명세서(스펙), 스티어링, 훅을 먼저 이해하시기 바랍니다.

## 왜 이 내용이 중요한가

Kiro의 워크플로 개념(명세서, 스티어링, 훅)이 무엇을 하는지 아는 것은 전체 그림의 절반입니다. 나머지 절반은 실제로 그 개념들과 어떻게 상호작용하는지를 이해하는 것입니다 — 에이전틱 챗을 통해, MCP로 연결된 도구를 통해, 그리고 IDE 또는 CLI 중 어느 것을 통해서. 이 선택들은 Kiro가 무엇을 할 수 있는지, 어떤 외부 데이터에 접근할 수 있는지, 그리고 팀의 기존 개발 환경에 어떻게 맞는지에 영향을 줍니다.

## 쉬운 설명

Kiro는 지시를 내리는 두 가지 방법(에이전틱 챗과 CLI)과 외부 정보 소스에 연결하는 한 가지 메커니즘(MCP)을 제공합니다. 이 세 가지 개념을 이해하면 Kiro가 무엇을 볼 수 있는지, 무엇을 할 수 있는지, 어떻게 이끌어가는지 알 수 있습니다.

**에이전틱 챗(Agentic Chat)** 은 IDE 내부의 대화형 인터페이스입니다. 요청을 입력하면, Kiro는 프로젝트 파일을 읽고 맥락을 파악한 뒤 여러 단계의 작업을 취합니다 — 단순히 코드를 제안하는 것이 아니라, 실제로 변경을 가하고 검증 단계를 수행하며 결과를 보고합니다.

**MCP(Model Context Protocol, 모델 컨텍스트 프로토콜)** 은 Kiro가 외부 도구와 데이터 소스 — 데이터베이스, API, 문서 사이트, 프로젝트 관리 시스템 — 에 연결하기 위해 사용하는 표준입니다. MCP는 개방형 표준이므로, MCP 호환 도구라면 무엇이든 Kiro에 연결할 수 있습니다.

**IDE vs. CLI** 는 인터페이스 선택입니다. IDE는 파일 탐색, 코드 하이라이팅, 통합 채팅 패널이 있는 시각적 환경을 제공합니다. CLI는 터미널 기반 접근을 제공하며, 자동화와 서버 환경에 유용합니다.

## 핵심 개념

### 에이전틱 챗 (Agentic Chat)

에이전틱 챗은 Kiro IDE 내부의 대화형 AI 인터페이스입니다. 텍스트만 출력하는 일반 채팅 인터페이스와 달리, 에이전틱 챗은 실제로 행동할 수 있습니다. 파일을 읽고 수정하고, 명령을 실행하고, MCP로 연결된 도구에 접근하고, 여러 단계의 작업을 수행합니다.

**왜 중요한가**: '에이전틱(agentic)'이라는 단어가 핵심 차이점입니다. 에이전트는 무언가를 합니다. 일반 AI 채팅 인터페이스는 질문에 답하고 텍스트를 제안합니다. 에이전틱 챗은 프로젝트를 읽고, 맥락을 파악하고, 행동합니다 — 채팅 창의 제안이 아닌 코드베이스의 실제 변경을 만들어냅니다.

**예시**: 에이전틱 챗을 열고 "결제 API에 에러 처리를 추가해줘"라고 입력합니다. Kiro는 결제 API 파일을 읽고, 에러가 처리되지 않는 곳을 찾아내고, 에러 처리 코드를 작성하고, 변경 내용을 보고합니다 — 모두 한 번의 대화에서.

### MCP (Model Context Protocol, 모델 컨텍스트 프로토콜)

MCP는 AI 모델이 외부 도구와 데이터 소스에 연결하는 방법을 정의하는 개방형 표준입니다. Kiro는 MCP를 지원하므로, 데이터베이스를 쿼리하거나, API를 호출하거나, 문서 시스템을 읽거나, 프로젝트 관리 도구와 상호작용하도록 설정할 수 있습니다.

**왜 중요한가**: MCP 없이는 Kiro가 프로젝트 내부 파일만 볼 수 있습니다. MCP가 있으면 Kiro는 외부 맥락을 볼 수 있습니다 — 데이터베이스의 현재 상태, 라이브 API의 응답 형식, 서드파티 서비스의 문서. 외부 시스템에 의존하는 프로젝트에서 Kiro의 결과물이 더 정확해집니다.

**예시**: PostgreSQL 데이터베이스에 MCP로 연결된 Kiro 설정은 현재 데이터베이스 스키마를 읽고, 실제 컬럼 이름과 타입에 맞는 코드를 생성합니다 — 개발자가 스키마를 직접 붙여넣을 필요가 없습니다.

### IDE vs. CLI

Kiro는 두 가지 인터페이스를 제공합니다: IDE(Visual Studio Code 기반)와 CLI(터미널에서 실행). 둘 다 동일한 AI 모델을 사용하고, 동일한 명세서(스펙), 스티어링, 훅 파일을 읽습니다.

**왜 중요한가**: 어떤 인터페이스가 맞는지는 맥락에 달려 있습니다. 기능을 대화식으로 개발하는 개발자는 보통 시각적 환경 때문에 IDE를 사용합니다. 워크플로를 자동화하거나, CI/CD 파이프라인에서 Kiro를 실행하거나, 서버 환경에서 작업하는 팀은 CLI를 사용합니다.

**IDE 예시**: 개발자가 Kiro IDE를 열고, 프로젝트 파일을 탐색하고, 명세서 문서를 검토하고, 에이전틱 챗으로 작업을 구현합니다.

**CLI 예시**: CI/CD 파이프라인이 자동화된 빌드 프로세스의 일부로 `kiro implement --spec feature-login`을 실행하며, 인간의 개입 없이 진행됩니다.

## 비유와 예시

> Kiro를 사무실에서 일하는 유능한 어시스턴트라고 생각해보세요. 에이전틱 챗은 그 어시스턴트와 대화하는 방법입니다 — 필요한 것을 설명하면 조언이 아닌 행동으로 응답합니다. MCP는 어시스턴트의 출입 카드입니다 — 어떤 방, 데이터베이스, 시스템에 들어갈 수 있는지를 결정합니다. IDE vs. CLI 구분은 어시스턴트가 모든 장비를 갖춘 자신의 책상(IDE)에서 일하는지, 아니면 원격지에서 전화로(CLI) 일하는지의 차이입니다.

어시스턴트를 최대한 활용하려면 세 가지 모두 중요합니다.

## 실제 활용 사례

**시나리오 1 — 대화식 개발**: 개발자가 IDE의 에이전틱 챗을 사용해 새 기능을 단계별로 구현하며, 파일 탐색기에서 변경 사항을 실시간으로 확인합니다.

**시나리오 2 — 외부 데이터 연동**: 팀이 MCP를 통해 회사 내부 API 문서를 Kiro에 연결합니다. 해당 API를 호출하는 코드를 작성할 때, Kiro가 실제 문서를 참조해 정확한 요청/응답 처리를 생성합니다.

**시나리오 3 — 자동화된 파이프라인**: 팀이 CI/CD 파이프라인에서 Kiro CLI를 실행합니다. 명세서가 승인되고 병합되면 자동화된 작업이 `kiro implement`를 실행해 초기 구현을 생성하고, 개발자가 병합 전에 검토합니다.

## 흔한 오해

1. **"에이전틱 챗은 그냥 IDE 안에 있는 ChatGPT다."** — 에이전틱 챗은 프로젝트 파일 시스템에 직접 접근하고, 파일을 수정하며, 명령을 실행할 수 있습니다. 개발 환경과 맥락에 통합된 것이지, 범용 채팅 인터페이스가 아닙니다.
2. **"MCP를 사용하려면 특정 외부 서비스가 필요하다."** — MCP는 개방형 표준입니다. MCP 프로토콜을 구현한 서비스라면 무엇이든 Kiro에 연결할 수 있습니다. 생태계가 발전하면서 사용 가능한 통합의 수도 늘어납니다.
3. **"CLI는 IDE의 기능이 제한된 버전이다."** — CLI는 IDE와 동일한 AI 모델을 사용하고 동일한 명세서와 스티어링 파일을 읽습니다. 다른 인터페이스일 뿐, 축소된 버전이 아닙니다.

## 심화 내용

> **심화 독자를 위한 내용** — 이 섹션을 읽지 않아도 Kiro의 환경 개념을 이해하는 데 지장이 없습니다.

MCP(Model Context Protocol, 모델 컨텍스트 프로토콜)는 원래 AI 모델이 추론 시 외부 맥락을 받는 방식을 표준화하기 위해 개발되었습니다. MCP를 채택함으로써 Kiro는 일반적인 도구(데이터베이스, 이슈 트래커, 문서 플랫폼)를 위한 사전 구축된 커넥터의 성장하는 생태계를 활용합니다. 커스텀 MCP 커넥터를 구축하는 개발자들은 프로그래밍 가능한 인터페이스를 노출하는 모든 시스템으로 Kiro의 접근 범위를 확장할 수 있습니다.
