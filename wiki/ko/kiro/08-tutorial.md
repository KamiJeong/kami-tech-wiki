# 핸즈온 튜토리얼: Kiro로 첫 기능 만들기

> 개인 할 일 관리 앱을 연습 프로젝트로 삼아 명세서 작성부터 구현 검토까지 Kiro 워크플로 전체를 직접 실행해봅니다.

## 이 페이지의 대상 독자

이 페이지는 Kiro 개요 아티클들을 읽고 구체적인 연습 프로젝트로 워크플로를 직접 실행해보고 싶은 개발자를 위한 글입니다. 코드 에디터 사용에 대한 기본적인 이해가 있다고 가정하지만, 깊은 기술 지식은 필요하지 않습니다.

## 왜 이 내용이 중요한가

Kiro 워크플로에 대해 읽는 것은 유용합니다. 실제로 한 번 실행해보는 것 — 진짜 명세서를 작성하고, 진짜 설계를 검토하고, AI가 진짜 기능을 구현하는 것을 지켜보는 것 — 은 개념을 구체적이고 기억에 남게 만듭니다. 이 튜토리얼은 그 경험을 안전한 연습 환경에서 제공하도록 설계되었습니다. 한 시간 안에 완료할 수 있을 만큼 작지만, Kiro의 핵심 기능 모두를 실습할 수 있을 만큼 복잡한 프로젝트를 사용합니다.

## 쉬운 설명

이 튜토리얼은 Kiro의 명세서 기반 워크플로를 사용해 개인 할 일 관리 앱을 만드는 과정을 안내합니다. 연습 프로젝트는 네 가지 기능을 포함합니다: 할 일 추가, 완료 표시, 상태로 필터링, 삭제. 이 네 가지 기능은 한눈에 이해할 만큼 단순하지만 요구사항 작성, 설계 검토, 작업 실행을 모두 실습할 수 있을 만큼 복잡합니다.

명세서를 작성하고, 검토하고, Kiro가 구현하는 것을 지켜보고, 결과를 검토합니다. 완료하면 처음부터 끝까지 하나의 Kiro 워크플로 사이클을 완성한 것입니다.

## 핵심 개념

### 연습 프로젝트 범위

개인 할 일 관리 앱에는 다음이 포함됩니다:
- 제목이 있는 할 일 추가
- 할 일을 완료로 표시
- 할 일 필터링 (전체 보기 / 활성 보기 / 완료 보기)
- 할 일 삭제

이것이 전체 범위입니다. 이 튜토리얼에서는 이 네 가지 기능 이외의 것은 범위에 포함되지 않습니다.

**이 범위를 선택한 이유**: 네 가지 기능은 명세서의 모든 부분(여러 사용자 스토리, 여러 설계 컴포넌트, 여러 작업)을 실습할 수 있을 만큼 충분한 복잡성을 제공하면서도, 이해하기 위한 전문 지식이 필요하지 않습니다.

### 최신 정보 안내

> **2026-05-19 기준**: Kiro는 이 튜토리얼이 작성될 당시 활발히 개발 중이었습니다. 특정 UI 요소, 명령 이름, 파일 형식이 변경되었을 수 있습니다. 불일치가 있으면 현재 문서를 위해 [kiro.dev](https://kiro.dev)를 참조하세요.

## 비유와 예시

> Kiro로 만드는 것은 소규모 집 리모델링을 일반 시공업자와 함께 하는 것과 같습니다. 원하는 것을 설명하면(요구사항), 시공업자가 계획을 그립니다(설계). 작업 순서에 합의하고(작업 목록), 시공업자가 실행합니다. 당신의 역할은 각 단계에서 검토하고 "네, 맞아요" 또는 "잠깐, 수정이 필요해요"라고 말하는 것입니다.

이 튜토리얼은 이 시공업자와의 첫 번째 협업입니다. 완료하고 나면 협업의 리듬을 알게 됩니다.

## 실제 활용 사례

### 사전 준비

시작하기 전에:
- Kiro 설치됨 ([kiro.dev](https://kiro.dev)에서 다운로드)
- Kiro IDE에서 프로젝트 디렉토리 열림
- 명세서가 무엇인지 기본 이해 (먼저 [Kiro 명세서 워크플로](04-specs-workflow.md)를 읽으세요)

### 1단계: 기능 설명 작성

Kiro의 에이전틱 챗을 엽니다. 다음을 입력합니다 (또는 같은 내용의 자신만의 표현):

```
개인 할 일 관리 앱을 만들고 싶습니다. 사용자가 다음을 할 수 있어야 합니다:
- 제목이 있는 할 일 추가
- 할 일을 완료로 표시
- 할 일 필터링: 전체, 활성(미완료)만, 완료만 보기
- 할 일 삭제

단일 사용자 애플리케이션. 계정 없음, 클라우드 저장소 없음.
```

**예상 결과**: Kiro가 설명을 확인하고 명세서 생성을 제안합니다.

**흔한 실수**: 범위에 없는 요구사항(사용자 계정, 공유, 클라우드 동기화)을 추가하는 것. 이 튜토리얼에서는 범위를 최소로 유지하세요.

### 2단계: requirements.md 검토

Kiro가 사용자 스토리와 인수 기준이 있는 `requirements.md`를 생성합니다. 파일을 열고 읽어보세요.

네 가지 기능 각각에 다음이 있는지 확인합니다:
- 사용자 스토리 ("사용자로서, 나는 ... 하기를 원합니다, 그래서...")
- 하나 이상의 측정 가능한 인수 기준

**검토 예시**: "할 일 추가"의 인수 기준이 "할 일이 추가됩니다"라고만 되어 있다면, Kiro에게 더 구체적으로 해달라고 요청하세요: "추가 즉시 목록에 할 일이 나타나고, 고유 ID가 부여되며, 제목이 빈 할 일은 제출할 수 없습니다."

**예상 결과**: 네 가지 사용자 스토리와 각각에 대한 구체적이고 측정 가능한 인수 기준이 있는 requirements.md.

### 3단계: design.md 검토

Kiro가 할 일 관리 앱이 어떻게 만들어질지 설명하는 `design.md`를 생성합니다. 기술적이지 않더라도 다음을 확인하세요:
- 할 일을 위한 데이터 모델이 포함되어 있는가? (할 일에 어떤 필드가 있는가?)
- 할 일 목록을 표시하는 UI 컴포넌트가 설명되어 있는가?
- 필터링이 어떻게 작동할지 설명되어 있는가?

**예상 결과**: 할 일 데이터 모델, UI 컴포넌트, 필터링 로직을 설명하는 design.md.

**흔한 실수**: 읽지 않고 설계를 승인하는 것. 설계는 아키텍처 선택이 이루어지는 곳입니다. 지금 동의하지 않는 선택이 있다면 말하세요 — 나중에 바꾸는 것이 더 비쌉니다.

### 4단계: tasks.md 검토

Kiro가 구현 단계의 순서가 있는 체크리스트인 `tasks.md`를 생성합니다. 다음을 검토합니다:
- 작업들이 논리적인 순서로 되어 있는가? (예를 들어, UI 컴포넌트보다 데이터 모델이 먼저)
- requirements.md의 모든 인수 기준이 최소 하나의 작업에 대응하는가?
- 작업들이 충분히 구체적인가? ("Task 데이터 모델 생성"은 구체적이고, "백엔드 구현"은 구체적이지 않음)

**예상 결과**: 6-12개의 구체적이고 순서가 있는 구현 작업이 있는 tasks.md.

### 5단계: 구현

Kiro에게 구현을 시작하라고 지시합니다:

```
명세서가 좋아 보입니다. tasks.md의 모든 작업을 구현해주세요.
```

Kiro가 작업 목록을 처리하는 것을 지켜봅니다. 각 작업은 특정 파일 변경 또는 추가에 대응해야 합니다.

**예상 결과**: 네 가지 기능 모두에 대한 구현된 코드. Kiro는 각 작업이 완료될 때 보고하고 완료할 수 없는 작업을 표시합니다.

**흔한 실수**: 구현 중에 자리를 비우는 것. Kiro가 명세서에서 다루지 않은 모호한 부분을 만나면 질문할 수 있습니다. 명세서를 기반으로 답변하세요 — 답변에 명세서 변경이 필요하다면 먼저 `requirements.md`를 수정하고 확인하세요.

### 6단계: 구현 검토

구현 후, requirements.md의 인수 기준에 대해 결과를 테스트합니다:

- 제목이 있는 할 일을 추가할 수 있나요?
- 추가 즉시 목록에 할 일이 나타나나요?
- 할 일을 완료로 표시할 수 있나요?
- 필터가 올바른 할 일을 보여주나요?
- 할 일을 삭제할 수 있나요?

**기능이 명세서대로 작동하지 않는 경우**: requirements.md의 관련 섹션을 수정해 의도한 동작을 명확히 하고, Kiro에게 해당 특정 작업을 재구현하도록 요청하세요.

**예상 결과**: requirements.md에 설명된 대로 작동하는 네 가지 기능 모두.

### 7단계: 다음 단계

이 튜토리얼을 완료한 후, 이제 다음을 할 준비가 되었습니다:
- 이 빌드 과정에서 설정한 규칙들을 담은 [스티어링 파일](05-steering.md)을 프로젝트에 추가
- 미래 변경에 따라 테스트를 동기화하는 [훅](06-hooks.md) 설정
- 기존 프로젝트의 실제 기능에 같은 워크플로 적용

## 흔한 오해

1. **"이 튜토리얼을 완료하려면 생성된 코드를 모두 이해해야 한다."** — 구현 코드가 아닌 명세서 문서를 이해해야 합니다. 모든 코드 줄을 이해하는지가 아니라, 코드가 requirements.md의 인수 기준을 충족하는지 검토하세요.
2. **"Kiro가 실수를 하면 뭔가 잘못된 것이다."** — AI 결과물이 처음부터 완벽한 경우는 드뭅니다. 검토하고, 명세서를 수정하고, 재구현하세요. 이것이 정상적인 워크플로이지 실패가 아닙니다.
3. **"모든 단계를 한 세션에 완료해야 한다."** — 명세서 문서들은 git에 커밋됩니다. 어느 단계에서든 잠시 멈추고 나중에 계속할 수 있습니다. 명세서가 세션 간에 의도를 보존합니다.
