# Kiro 핵심 개념: 워크플로

> Kiro의 워크플로는 세 가지 기둥 위에 세워져 있습니다 — 명세서(스펙), 스티어링(steering), 훅(hooks). 각각은 AI 에이전트에게 서로 다른 종류의 구조화된 가이드라인을 제공합니다.

## 이 페이지의 대상 독자

이 페이지는 Kiro의 핵심 워크플로 기능들이 어떻게 맞물리는지 이해하고 싶은 분들을 위한 글입니다. 사전 기술 지식 없이도 세 가지 개념과 각각의 역할을 설명합니다. 아직 읽지 않으셨다면 먼저 [Kiro란 무엇인가?](01-what-is-kiro.md)를 읽어보세요.

## 왜 이 내용이 중요한가

AI 코딩 도구는 주어진 맥락을 바탕으로 작동합니다. 맥락이 짧고 비공식적인 프롬프트라면 AI는 많은 것을 가정합니다. 맥락이 구조화되어 있다면 — 명세서, 지속적인 가이드라인 세트, 자동화된 트리거 — AI의 결과물은 더 예측 가능하고 일관성 있게 됩니다.

Kiro의 세 가지 워크플로 개념은 그 구조화된 맥락이 만들어지고 유지되는 메커니즘입니다. 이것들을 이해하면 Kiro가 비공식 AI 프롬프트보다 더 일관된 결과를 내는 이유, 그리고 Kiro를 잘 활용하기 위해 무엇이 필요한지 알 수 있습니다.

## 쉬운 설명

Kiro의 워크플로는 AI 에이전트에게 세 가지 유형의 입력을 제공하는 방식으로 구성됩니다.

1. **명세서(스펙, Specs)** — 이 특정 기능에서 무엇을 만들 것인지 (단기적, 기능 특화)
2. **스티어링(Steering)** — 이 프로젝트의 모든 것을 어떻게 만들 것인지 (지속적, 프로젝트 전반)
3. **훅(Hooks)** — 무언가 변경될 때 자동으로 무엇을 할 것인지 (이벤트 기반, 트리거식)

이 세 가지 입력이 함께 작동합니다. 명세서는 목표를 정의합니다. 스티어링은 규칙을 정의합니다. 훅은 프로젝트가 발전하면서 일관성을 자동으로 유지합니다.

## 핵심 개념

### 명세서 (스펙, Specs)

명세서(스펙)는 Kiro가 기능 설명으로부터 생성하는 세 가지 구조화된 문서의 모음입니다: `requirements.md`, `design.md`, `tasks.md`. 이 문서들은 함께 다음 질문에 답합니다. 무엇을 만들 것인가, 어떻게 만들 것인가, 어떤 단계로 진행할 것인가?

**왜 중요한가**: 명세서 없이는 AI가 채팅에 입력한 내용만으로 작업합니다. 명세서가 있으면 AI는 코드가 작성되기 전에 검토된 구조화된 문서를 바탕으로 작업합니다. 명세서는 당신의 의도와 AI 결과물 사이의 계약이 됩니다.

**예시**: 애플리케이션에 "비밀번호 찾기" 기능을 추가하고 싶습니다. "비밀번호 찾기 플로우 추가"라고 입력하는 대신, Kiro에게 명세서를 생성해달라고 합니다. Kiro는 requirements.md(사용자가 할 수 있어야 하는 것), design.md(어떤 컴포넌트를 만들 것인지), tasks.md(순서가 있는 구현 단계 목록)를 생성합니다. 명세서를 검토하고 승인하면 Kiro가 구현합니다.

### 스티어링 (Steering)

스티어링 파일은 프로젝트의 `.kiro/steering/` 폴더에 저장된 마크다운 문서입니다. AI 에이전트를 위한 지속적인 가이드라인을 담습니다. 코딩 규칙, 명명 규칙, 사용할 라이브러리, 따라야 할 아키텍처 패턴 등이 그 예입니다.

**왜 중요한가**: 스티어링 파일은 일관성 문제를 해결합니다. Kiro가 실행될 때마다 스티어링 파일을 읽습니다. 즉, 어떤 팀원이 트리거했든, 어떤 세션에서 실행됐든, 모든 AI 작업이 동일한 프로젝트 전반의 규칙을 따릅니다.

**예시**: `conventions.md`라는 스티어링 파일에 이렇게 적을 수 있습니다. "모든 API 응답은 `status`, `data`, `error` 필드를 가진 JSON 봉투를 사용해야 합니다. 모든 변수 이름은 camelCase를 사용합니다. 에러 메시지는 사람이 읽을 수 있어야 합니다." Kiro가 생성하는 모든 API 엔드포인트는 자동으로 이 규칙을 따릅니다.

### 훅 (Hooks)

훅은 파일 시스템 이벤트에 의해 트리거되는 자동화된 AI 작업입니다. `.kiro/hooks/`에 정의되며, 특정 패턴과 일치하는 파일이 변경될 때 어떤 AI 작업을 실행할지 지정합니다.

**왜 중요한가**: 프로젝트가 커지면서 관련 파일들을 동기화 상태로 유지하는 것이 유지보수 부담이 됩니다. 훅을 통해 그 유지보수를 자동화할 수 있습니다. API가 바뀌면 Kiro가 관련 테스트를 자동으로 업데이트할 수 있습니다. 새 컴포넌트가 추가되면 Kiro가 자동으로 문서를 생성할 수 있습니다.

**예시**: "api/*.ts가 변경되면 tests/api/의 해당 테스트를 업데이트"라고 정의된 훅은 개발자가 API 파일에 변경을 저장할 때마다 자동으로 실행됩니다. AI는 변경 사항을 검토하고 테스트 파일을 그에 맞게 업데이트합니다.

## 비유와 예시

> 프랜차이즈 레스토랑을 운영한다고 생각해보세요. 명세서(스펙)는 오늘 저녁 특선 메뉴의 레시피입니다 — 이 요리를 위해 작성되었고, 주방이 요리를 시작하기 전에 검토됩니다. 스티어링은 프랜차이즈 운영 매뉴얼입니다 — 모든 매장의 모든 요리에 적용되는 규칙입니다. 훅은 주방 타이머입니다 — 무언가 주의가 필요할 때 자동으로 알려줍니다.

세 가지가 함께 작동합니다. 레시피는 오늘 요리사에게 무엇을 만들지 알려줍니다. 운영 매뉴얼은 다른 모든 요리와 일관성 있게 만들어지도록 보장합니다. 타이머는 요리사가 다른 곳에 집중하는 동안 아무것도 타지 않도록 합니다.

## 실제 활용 사례

**초기 설정 시나리오**: Kiro로 새 프로젝트를 시작하는 팀이 기술 선택과 코딩 규칙을 담은 스티어링 파일을 먼저 작성합니다. 기능 작업이 시작되기 전에 AI는 프로젝트 전체에 걸쳐 따를 명확한 규칙을 가지게 됩니다.

**기능 개발 시나리오**: 새 로그인 기능을 위해 개발자가 Kiro에게 명세서를 생성해달라고 합니다. 명세서를 검토하고 승인한 뒤 구현에 사용합니다. AI는 명세서(무엇을 만들 것인가)와 스티어링 파일(어떻게 만들 것인가)을 모두 읽습니다.

**유지보수 시나리오**: API 파일이 변경될 때마다 API 문서를 업데이트하는 훅이 설정되어 있습니다. 6개월 후 개발자가 API 엔드포인트를 수정하면, 별도의 수동 작업 없이도 문서가 자동으로 업데이트됩니다.

## 흔한 오해

1. **"명세서는 그냥 문서다."** — 명세서는 능동적으로 작동하는 문서입니다. Kiro는 명세서를 읽고 그에 따라 실행합니다. AI 기반 구현의 주요 입력입니다 — 만들어진 것의 기록이 아니라 만들 것의 청사진입니다.
2. **"스티어링은 README와 같은 것이다."** — README는 프로젝트를 사람 독자에게 설명합니다. 스티어링 파일은 AI 에이전트를 위한 지침입니다. 자동화된 동작을 이끌기 위해 작성되었지, 프로젝트의 역사나 설치 단계를 설명하기 위한 것이 아닙니다.
3. **"훅이 코드를 자동으로 실행한다."** — 훅은 AI 작업을 트리거하지, 임의의 코드를 실행하지 않습니다. 파일 이벤트가 발생하면 AI 에이전트가 특정하고 범위가 정해진 작업을 수행합니다. 개발자는 결과가 적용되기 전에 AI의 결과물을 검토합니다.

## 심화 내용

> **심화 독자를 위한 내용** — 이 섹션을 읽지 않아도 Kiro의 핵심 워크플로 개념을 이해하는 데 지장이 없습니다.

명세서(스펙), 스티어링, 훅은 세 가지 고전적인 소프트웨어 엔지니어링 개념에 대응합니다: 명세(무엇), 규약(어떻게), 자동화(언제). Kiro의 기여는 이 세 가지를 사람이 읽는 문서가 아닌 AI 에이전트를 위한 기계 판독 가능한 입력으로 만든 것입니다. `.kiro/` 디렉토리 구조는 이 세 가지 입력을 명시적이고 버전 관리 가능한 형태로 만들어, AI의 동작이 프로젝트의 git 이력을 통해 추적되고 감사될 수 있습니다.
