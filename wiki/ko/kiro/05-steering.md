# Kiro 스티어링 가이드

> 스티어링(steering) 파일은 `.kiro/steering/`에 저장된 마크다운 문서로, AI 에이전트에게 모든 세션과 모든 팀원에게 적용되는 지속적이고 프로젝트 전반의 규칙을 제공합니다.

## 이 페이지의 대상 독자

이 페이지는 전체 프로젝트에 적용되는 일관되고 신뢰할 수 있는 가이드라인을 Kiro에 제공하는 방법을 이해하고 싶은 개발자와 팀 리더를 위한 글입니다. 코딩 규칙이라는 개념에 대한 기본적인 이해가 있으면 도움이 됩니다.

## 왜 이 내용이 중요한가

AI 코딩 도구에서 가장 큰 불만 중 하나는 일관성 없음입니다. AI가 월요일에는 한 스타일로 코드를 생성하고, 수요일에는 다른 스타일로 생성합니다. 개발자 한 명의 AI 생성 코드는 camelCase를 사용하고, 다른 개발자의 코드는 snake_case를 사용합니다. AI가 한 파일에서는 HTTP 요청에 한 라이브러리를 사용하고 다른 파일에서는 다른 라이브러리를 사용합니다.

스티어링 파일이 이 문제를 해결합니다. 팀의 규칙, 아키텍처 원칙, 프로젝트별 제약사항이 AI에 대한 지속적인 입력이 되는 메커니즘입니다 — 자동으로, 항상, 모든 팀원에게.

스티어링 파일에 시간을 투자한 팀은 훨씬 더 일관된 AI 결과물을 보고합니다. 스티어링 파일을 건너뛰는 팀은 AI 불일관성을 수정하는 데 더 많은 시간을 씁니다.

## 쉬운 설명

스티어링 파일은 프로젝트의 `.kiro/steering/`에 저장된 마크다운 파일입니다. AI를 위한 상시 지침처럼 작성합니다: "이 명명 규칙을 항상 사용하라", "이 라이브러리는 절대 사용하지 마라", "API를 만들 때는 항상 이 응답 구조를 따르라".

Kiro는 모든 AI 세션 시작 시 스티어링 파일을 읽습니다. 코드를 생성하거나, 질문에 답하거나, 작업을 실행할 때마다 스티어링 파일의 규칙을 적용합니다. 스티어링 파일은 프로젝트의 git 저장소에 커밋되기 때문에 모든 팀원이 공유하고 코드와 함께 버전 관리됩니다.

스티어링 파일을 여러 개 가질 수 있습니다. 일반적인 방식은 관심사별로 구성하는 것입니다: 하나는 코딩 규칙, 하나는 아키텍처 규칙, 하나는 보안 요구사항.

## 핵심 개념

### 스티어링 파일에 들어가는 내용

스티어링 파일은 다음과 같은 규칙에 가장 유용합니다:
- 전체 프로젝트에 적용되는 규칙 (특정 기능에만 해당하는 것이 아닌)
- 모든 명세서나 프롬프트마다 반복하기 번거로운 규칙
- 모든 팀원이 일관되게 따라야 하는 규칙

**왜 중요한가**: 규칙이 특정 기능에만 적용된다면 그 기능의 명세서에 넣어야 합니다. 어디에나 적용되는 규칙이라면 스티어링 파일에 넣어야 합니다.

**스티어링 파일 예시 내용**:
```markdown
# 프로젝트 규칙

## 명명
- 변수 이름: camelCase (JavaScript/TypeScript)
- 파일 이름: kebab-case
- 데이터베이스 컬럼: snake_case

## API 응답
모든 API 응답은 다음 봉투 형식을 사용해야 합니다:
{
  "status": "success" | "error",
  "data": <payload>,
  "error": null | { "code": string, "message": string }
}

## 의존성
- HTTP 요청: `fetch` API 사용 — axios 추가 금지
- 날짜 처리: date-fns 사용 — moment.js 사용 금지
```

### Kiro가 스티어링 파일을 읽는 방법

Kiro는 모든 세션 시작 시 `.kiro/steering/`의 모든 파일을 읽습니다. 규칙들은 AI 모델의 지속적인 맥락으로 적용됩니다 — 해당 세션 동안 AI가 만드는 모든 코드 생성, 검토, 제안에 영향을 줍니다.

**왜 중요한가**: 한 번 입력하고 잊어버리는 프롬프트와 달리, 스티어링 파일은 항상 존재합니다. 반복할 필요가 없습니다. AI는 자신이 하는 모든 것에 대한 배경 맥락으로 스티어링 파일을 가지고 있습니다.

**예시**: API 응답 봉투 규칙을 설정한 기능과 완전히 다른 기능을 작업하는 개발자도, 스티어링 파일이 항상 읽히기 때문에 AI는 여전히 그 규칙을 따릅니다.

## 비유와 예시

> 프랜차이즈 레스토랑 체인을 생각해보세요. 모든 매장이 같은 운영 매뉴얼을 따릅니다: 각 요리를 어떻게 준비하는지, 어떤 재료를 사용하는지, 어떻게 플레이팅하는지. 매뉴얼은 각 교대 근무마다 다시 작성되지 않습니다 — 모두에게, 항상 적용되는 상시 지침 세트입니다.

Kiro의 스티어링 파일은 프로젝트의 운영 매뉴얼입니다. 모든 AI 작업이 매뉴얼을 따릅니다. 새 팀원, 새 기능, 새 세션 — 매뉴얼이 모두에게 적용됩니다.

## 실제 활용 사례

**새 프로젝트 스티어링 설정**:

1. 프로젝트 루트에 `.kiro/steering/`을 생성합니다.
2. `conventions.md` 생성 — 명명 규칙, 코드 스타일 선호도, 라이브러리 선택을 문서화합니다.
3. `architecture.md` 생성 — 프로젝트 구조를 문서화합니다: 무엇이 어디에 가는지, 계층이 어떻게 통신하는지, 어떤 패턴을 따르는지.
4. `security.md` 생성 (해당 시) — 보안 요구사항 문서화: 하드코딩된 자격증명 금지, 필수 입력 유효성 검사, 인증 방식.
5. 모든 스티어링 파일을 git에 커밋합니다.

**스티어링 파일 유지보수**:
- 팀이 새로운 아키텍처 결정을 내릴 때 스티어링 파일을 업데이트하세요.
- 스티어링 파일을 코드와 같은 수준으로 관리하세요: 풀 리퀘스트에서 변경 사항을 검토하세요.
- AI가 규칙을 어기는 결과물을 만들어낼 때, 해당 규칙이 스티어링 파일에 문서화되어 있는지 확인하세요. 없다면 추가하세요.

**스티어링 파일 커버리지 확인**:
AI가 같은 규칙을 반복적으로 어긴다면, 그 규칙이 스티어링 파일에 있어야 할 가능성이 높습니다. AI 오류를 스티어링 문서의 빈틈에 대한 신호로 활용하세요.

## 흔한 오해

1. **"스티어링 파일은 README와 같다."** — README는 프로젝트를 사람 독자에게 설명합니다. 스티어링 파일은 AI 에이전트에 지침을 줍니다. 특정 AI 동작을 유도하기 위해 작성된 것으로, 신규 팀원의 첫날을 위해 읽히는 것이 아닙니다 (물론 그 목적으로도 활용될 수는 있지만).
2. **"스티어링 내용이 많을수록 좋다."** — 지나치게 길거나 모순된 스티어링 파일은 AI 성능을 저하시킵니다. 각 규칙을 명확하고 모호하지 않게 유지하세요. 두 규칙이 충돌한다면 문서에서 그 충돌을 해결하세요.
3. **"스티어링 파일이 기능별 명세서를 대체한다."** — 스티어링 파일과 명세서는 서로 다른 목적을 가집니다. 스티어링은 항상 적용되는 것을 다룹니다. 명세서는 특정 기능에만 적용되는 것을 다룹니다. 둘 다 필요합니다.

## 심화 내용

> **심화 독자를 위한 내용** — 이 섹션을 읽지 않아도 Kiro 스티어링을 이해하는 데 지장이 없습니다.

스티어링 파일은 시스템 프롬프트 주입의 한 형태로 기능합니다 — 사용자 입력이 처리되기 전에 모든 세션 시작 시 AI 모델의 컨텍스트에 삽입됩니다. 이는 기본 동작을 설정하는 데 효과적이지만, 매우 긴 스티어링 파일은 프로젝트별 내용에 사용될 수 있는 컨텍스트 토큰을 소비한다는 의미이기도 합니다. 스티어링 파일에서 명확성과 간결성을 목표로 하세요: 구체적이고 모호하지 않은 규칙이 긴 설명적 산문보다 더 효과적입니다.
