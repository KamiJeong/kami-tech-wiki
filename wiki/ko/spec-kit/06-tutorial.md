# 핸즈온 튜토리얼: Spec Kit으로 첫 기능 만들기

> 개인 할 일 관리 앱을 연습 프로젝트로 삼아 명세서 작성부터 검증된 구현까지 완전한 Spec Kit 워크플로를 실행해봅니다.

## 이 페이지의 대상 독자

이 페이지는 Spec Kit 개요 아티클들을 읽고 구체적인 연습 프로젝트로 워크플로를 직접 실행해보고 싶은 개발자를 위한 글입니다. Claude Code가 환경에서 실행 중이어야 합니다. 코드 에디터와 터미널에 대한 기본적인 이해가 있다고 가정합니다.

## 왜 이 내용이 중요한가

Spec Kit을 한 번 실행해보는 것 — 진짜 명세서를 작성하고, 진짜 계획을 검토하고, AI가 진짜 기능을 구현하는 것을 지켜보는 것 — 은 읽기만으로는 얻을 수 없는 방식으로 개념을 구체적으로 만들어줍니다. 이 튜토리얼은 안전한 연습 환경에서 그 경험을 제공하도록 설계되었습니다.

## 쉬운 설명

이 튜토리얼은 Spec Kit의 간소화된 네 단계 워크플로(명세서 → 계획 → 작업 → 구현)를 사용해 개인 할 일 관리 앱을 만드는 과정을 안내합니다. 연습 프로젝트는 네 가지 기능을 포함합니다: 할 일 추가, 완료로 표시, 상태로 필터링, 삭제.

완료하면 하나의 완전한 Spec Kit 워크플로 사이클을 실행하고 `spec.md`, `plan.md`, `tasks.md` 세 가지 파이프라인 산출물을 만들어낸 것입니다.

## 핵심 개념

### 연습 프로젝트 범위

개인 할 일 관리 앱에는 다음이 포함됩니다:
- 제목이 있는 할 일 추가
- 할 일을 완료로 표시
- 할 일 필터링 (전체 보기 / 활성 보기 / 완료 보기)
- 할 일 삭제

이것이 전체 범위입니다. 이 튜토리얼에서는 이 네 가지 기능 이외의 것은 범위에 포함되지 않습니다.

### 최신 정보 안내

> **2026-05-19 기준**: Spec Kit은 이 저장소의 Claude Code 스킬로 구현됩니다. 정확한 명령어 호출 방법은 Claude Code 설정에 따라 달라집니다. 명령어가 보여준 대로 작동하지 않는다면, 현재 호출 형식을 위해 Claude Code 문서 또는 `.claude/skills/` 디렉토리를 확인하세요.

## 비유와 예시

> 이 튜토리얼을 실행하는 것은 고속도로에 가기 전에 주차장에서 운전을 배우는 것과 같습니다. 주차장(이 튜토리얼)은 단순하고 위험이 낮은 시나리오를 사용합니다. 고속도로(실제 프로젝트 기능)는 같은 기술을 사용하지만 결과가 더 중요합니다. 주차장 연습이 고속도로를 안전하게 만드는 것입니다.

## 실제 활용 사례

### 사전 준비

- 개발 환경에서 Claude Code 실행 중
- Claude Code에서 프로젝트 디렉토리 열림
- Spec Kit 스킬 사용 가능 (Claude Code에서 `/speckit-specify`가 응답하는지 확인)

### 1단계: 명세서 작성

Claude Code에서 실행합니다:

```
/speckit-specify
```

설명을 입력하라는 메시지가 나타나면 다음을 입력합니다:

```
개인 할 일 관리 앱을 만들어줘. 사용자가 다음을 할 수 있어야 해:
- 제목이 있는 할 일 추가
- 할 일을 완료로 표시
- 할 일 필터링: 전체, 활성(미완료)만, 완료만 보기
- 할 일 삭제

단일 사용자 앱. 계정 없음. 세션 간에 할 일이 유지됨 (localStorage).
```

**결과물 검토**: `spec.md`를 엽니다. 다음을 확인하세요:
- 네 가지 기능이 모두 사용자 스토리로 나타나는가
- 각 사용자 스토리에 측정 가능한 인수 기준이 있는가
- "계정 없음"과 "localStorage"가 범위 제약으로 나타나는가
- 엣지 케이스가 다루어졌는가 (빈 제목, 빈 목록, 결과 없는 필터)

**흔한 실수**: 인수 기준을 읽지 않고 명세서를 승인하는 것. AI가 측정 가능하지 않은 기준 ("할 일이 추가됨")이 아닌 "할 일이 즉시 고유 ID와 입력한 제목과 함께 목록에 나타남" 같은 기준을 작성할 수 있습니다. 계속하기 전에 수정하세요.

**예상 결과**: 네 가지 사용자 스토리와 각각에 대한 구체적인 인수 기준이 있는 `specs/task-tracker/spec.md`.

### 2단계: 구현 계획 작성

Claude Code에서 실행합니다:

```
/speckit-plan
```

Spec Kit이 `spec.md`를 읽고 `plan.md`를 생성합니다. 다음을 검토하세요:
- 제안된 기술 스택이 프로젝트의 스택과 일치하는가?
- 할 일의 데이터 모델(필드, 타입)이 기대에 부합하는가?
- 파일 구조가 합리적인가?
- 동의하지 않는 아키텍처 선택이 있는가?

**무언가 잘못된 경우**: `spec.md`를 수정하거나 Spec Kit에 변경할 내용을 알려주고 `/speckit-plan`을 다시 실행합니다.

**예상 결과**: 기술 스택 결정, 할 일 데이터 모델, 제안된 파일 구조, 단계적 구현 접근법이 있는 `specs/task-tracker/plan.md`.

### 3단계: 작업 목록 생성

Claude Code에서 실행합니다:

```
/speckit-tasks
```

Spec Kit이 `plan.md`를 읽고 `tasks.md`를 생성합니다. 다음을 검토하세요:
- 작업들이 논리적인 순서인가? (데이터 모델 먼저, 그 다음 UI 컴포넌트, 그 다음 통합)
- `spec.md`의 모든 인수 기준이 최소 하나의 작업에 대응하는가?
- 작업들이 구체적인가? ("`filterByStatus()` 함수 생성"은 구체적이고, "필터링 구현"은 구체적이지 않음)

**예상 결과**: 6-10개의 구체적이고 순서가 있는 구현 작업이 있는 `specs/task-tracker/tasks.md`.

### 4단계: 구현

Claude Code에서 실행합니다:

```
/speckit-implement
```

Spec Kit이 작업 목록을 처리하는 것을 지켜봅니다. 각 작업은 하나 이상의 파일 변경에 대응합니다. Spec Kit은 각 작업이 완료될 때 `[X]`로 표시합니다.

**구현 중**: 자리를 지키세요. 명세서의 모호성을 만나면 Spec Kit이 물어볼 것입니다. 명세서를 기반으로 답변하세요 — 답변에 명세서 변경이 필요하다면 `spec.md`에서 변경을 하고 확인하세요.

**예상 결과**: `tasks.md`의 모든 작업이 `[X]`로 표시되고, 지정된 모든 파일이 생성되거나 업데이트됨.

### 5단계: 구현 검증

`spec.md`의 각 인수 기준을 테스트합니다:
- 제목이 있는 할 일을 추가할 수 있나요?
- 고유 ID와 함께 즉시 할 일이 나타나나요?
- 할 일을 완료로 표시할 수 있나요?
- 필터가 올바른 부분 집합의 할 일을 보여주나요?
- 할 일을 삭제할 수 있나요?

**기능이 작동하지 않는 경우**: `spec.md`의 관련 인수 기준을 더 정확하게 업데이트하고, 전체 작업 목록이 아닌 영향받은 작업에 대해 `/speckit-implement`를 다시 실행합니다.

**선택사항**: 모든 인수 기준에 대한 체계적인 확인을 위해 `/speckit-verify-run`을 실행합니다.

### 6단계: 다음 단계

이 튜토리얼을 완료한 후:
- 전체 워크플로 시도 (1단계와 2단계 사이에 `/speckit-clarify`, `/speckit-checklist`, `/speckit-analyze` 추가)
- 기존 프로젝트의 실제 기능에 같은 워크플로 적용
- Spec Kit을 최대한 활용하기 위한 조언을 위해 [모범 사례](07-best-practices.md) 아티클 탐색

## 흔한 오해

1. **"Spec Kit이 생성하는 모든 파일을 이해해야 한다."** — 구현 파일이 아닌 명세서 문서(`spec.md`, `plan.md`, `tasks.md`)에 집중하세요. 당신의 역할은 모든 생성된 코드 줄을 읽는 것이 아니라 구현이 인수 기준을 충족하는지 검토하는 것입니다.
2. **"Spec Kit이 실수를 하면 튜토리얼이 망가진 것이다."** — AI 결과물은 완벽하지 않습니다. 검토하고, 명세서를 수정하고, 재구현하세요. 이것이 의도된 워크플로이지, 실패 모드가 아닙니다.
3. **"간소화된 워크플로는 단순한 기능을 위한 것이다."** — 간소화된 워크플로는 명확하고 모호하지 않은 요구사항을 가진 기능을 위한 것입니다. 구현의 복잡성은 요구사항의 복잡성과 별개입니다. 명확한 요구사항의 복잡한 기능은 간소화된 워크플로를 사용할 수 있습니다. 불명확한 요구사항의 단순한 기능은 전체 워크플로가 필요할 수 있습니다.
