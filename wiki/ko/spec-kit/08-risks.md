# Spec Kit 위험과 한계

> Spec Kit은 구조화된 AI 보조 개발을 위한 강력한 도구지만, 제품 기획을 대체하지 않고, 구현 품질을 보장하지 않으며, 소규모 또는 탐색적 작업에는 오히려 부담이 됩니다.

## 이 페이지의 대상 독자

이 페이지는 Spec Kit이 하지 못하는 것이 무엇인지, 그리고 가치를 더하기보다 오히려 마찰을 만들어내는 상황이 어디인지 솔직하게 알고 싶은 개발자와 팀 리더를 위한 글입니다. Spec Kit 워크플로에 대한 이해가 있다고 가정합니다.

## 왜 이 내용이 중요한가

모든 도구에는 가치를 더하는 범위와 그에 상응하는 혜택 없이 오버헤드만 추가하는 범위가 있습니다. 어울리지 않는 상황에까지 Spec Kit을 적용하면 좌절감을 주고 시간을 낭비합니다. 한계를 이해하면 도움이 되는 곳에는 활용하고, 도움이 되지 않는 곳에서는 피할 수 있게 됩니다.

## 쉬운 설명

Spec Kit은 명세서를 작성할 수 있을 만큼 요구사항이 충분히 명확한 기능의 AI 보조 구현을 구조화하는 데 적합합니다. 명확한 요구사항을 만들어내는 사고를 대신하거나, AI 결과물의 품질을 보장하거나, 소규모 또는 일회용 작업을 더 빠르게 만드는 데는 적합하지 않습니다.

아래의 위험과 한계는 Spec Kit을 피해야 할 이유가 아닙니다. Spec Kit이 평소보다 가치를 덜 제공하거나, Spec Kit을 잘못 적용했을 때 상황을 나아지게 하기보다 나빠지게 만들 수 있는 조건들입니다.

## 핵심 개념

### Spec Kit은 제품 기획을 대체하지 않는다

Spec Kit은 개발 프로세스를 구조화합니다. 무엇을 만들지는 결정해주지 않습니다. 무엇을 원하는지 모르는 팀은 혼란을 반영하는 명세서를 만들어내고 — 그것을 정확하게 구현하게 됩니다.

**왜 중요한가**: Spec Kit 재작업의 가장 흔한 원인은 충분한 제품 사고 없이 승인된 명세서입니다. AI는 명세서에 명시된 것을 정확히 구현합니다. 명세서가 틀리면 구현도 틀립니다.

**예시**: 팀이 어떤 이벤트가 알림을 트리거해야 하는지, 알림 대상이 누구인지, 푸시/이메일/인앱 중 어떤 형태의 알림이어야 하는지 합의하기 전에 알림 기능 명세서를 작성합니다. 결과 명세서는 내부적으로는 일관되지만 제품에는 맞지 않습니다. Spec Kit은 이를 충실히 구현해 올바르지만 쓸모없는 기능을 만들어냅니다.

**완화 방법**: `/speckit-specify`를 실행하기 전에 제품 관련 질문을 해결하세요. Spec Kit은 개발 도구이지, 제품 설계 도구가 아닙니다.

### 약한 명세서는 약한 구현을 만든다

구현의 품질은 명세서의 품질에 의해 제한됩니다. Spec Kit은 명세서 품질을 증폭합니다 — 좋은 명세서는 좋은 구현을 만들고, 모호한 명세서는 모호한 구현을 만듭니다.

**왜 중요한가**: 개발자들은 때때로 Spec Kit이 자신이 모호하게 남겨둔 세부사항을 "알아서 해결할" 것이라고 믿습니다. 그렇지 않습니다. Spec Kit은 모호한 부분을 포함해 명세서에 명시된 것을 구현합니다.

**예시**: "필터가 올바르게 작동함"이라는 인수 기준은 모든 의미 있는 세부사항을 AI의 재량에 맡깁니다. AI는 어떤 정의에서는 "작동하는" 필터를 만들어냅니다. 원하는 방식으로 작동하는지는 그 정의가 의도와 얼마나 일치하느냐에 달려 있습니다.

**완화 방법**: 계획 전에 `/speckit-clarify`를 실행해 모호성을 드러내세요. 명세서를 승인하기 전에 모든 인수 기준의 측정 가능성을 검토하세요.

### 소규모 실험에는 프로세스 오버헤드가 생긴다

전체 Spec Kit 파이프라인(여덟 단계, 세 개의 검토 산출물)은 코드 작성 전에 상당한 시간 투자를 요구합니다. 소규모 실험, 일회용 프로토타입, 또는 수명이 하루인 기능에서는 이 오버헤드가 제공하는 가치를 초과할 수 있습니다.

**왜 중요한가**: 두 줄짜리 변경에 전체 파이프라인을 적용하는 것은 낭비입니다. 이 프로세스는 패치가 아닌 기능을 위해 설계되었습니다.

**예시**: 개발자가 두 가지 UI 접근법을 커밋하기 전에 어느 것이 더 나은지 시험해보고 싶습니다. 각 접근법에 전체 Spec Kit 파이프라인을 실행하는 것은 과도합니다. 탐색 단계에서는 직접 프롬프팅 — 또는 수동 코딩 — 이 더 적합합니다.

**완화 방법**: 워크플로를 기능에 맞추세요. 위험이 낮은 기능에는 간소화된 워크플로(네 단계)를 사용하고, 일회용 실험에는 직접 프롬프팅을 고려하세요.

### AI 에이전트의 가변성

명확한 명세서와 계획이 있더라도 AI 결과물은 결정론적이지 않습니다. 동일한 명세서라도 다른 세션, 다른 모델, 다른 Claude Code 버전에서 약간씩 다른 구현을 만들어낼 수 있습니다.

**왜 중요한가**: Spec Kit은 AI 가변성을 줄이지만 없애지는 못합니다. 팀은 동일한 명세서로 다른 실행에서 동일한 결과물을 기대해서는 안 됩니다.

**예시**: 동일한 `tasks.md`에서 `/speckit-implement`를 두 번 실행하면 모든 인수 기준을 충족하지만 다른 변수명, 다른 함수 시그니처, 다른 오류 처리 접근법을 사용하는 코드가 만들어질 수 있습니다.

**완화 방법**: 예상되는 정확한 결과물이 아닌 인수 기준에 대해 구현 결과물을 검토하세요. 명세서는 목표를 정의합니다. 목표에 이르는 정확한 경로는 다를 수 있습니다.

### 산출물 정합성 유지에는 규율이 필요하다

Spec Kit은 세 가지 파이프라인 산출물(`spec.md`, `plan.md`, `tasks.md`)을 생성하며 이것들은 생성 당시의 기능을 설명합니다. 구현이 진행되고 요구사항이 진화하면서 이 산출물들은 실제 코드와 동기화가 맞지 않게 될 수 있습니다.

**왜 중요한가**: 명세서 드리프트(spec drift) — 명세서가 실제로 구현된 방식과 다르게 기능을 설명하는 상태 — 는 Spec Kit이 방지하도록 설계된 문서-코드 동기화 문제의 특정 형태입니다. 팀이 요구사항 변경 시 명세서 업데이트에 대한 규율을 유지하지 않으면 산출물이 유용하기보다 오해를 불러일으킵니다.

**완화 방법**: 명세서 업데이트를 개발 프로세스의 일부로 취급하세요. 요구사항이 변경되면 먼저 `spec.md`를 업데이트하고, 필요하면 계획과 작업을 업데이트한 후 재구현하세요.

## 비유와 예시

> GPS 내비게이션은 가진 정보를 바탕으로 최적의 경로를 알려줍니다. 지도 데이터가 낡았다면 자신 있게 틀린 길을 안내합니다. 안내를 무시하면 도움을 줄 수 없습니다. 존재하지 않는 주소로 내비게이션을 요청하면 정상적으로 실패합니다.

Spec Kit은 GPS입니다. 명세서는 지도 데이터입니다. 제품 기획은 어느 목적지를 입력할지 아는 것입니다. 워크플로를 따르는 규율은 안내에 따르는 것입니다. 시스템이 작동하려면 네 가지 모두가 필요합니다.

## 실제 활용 사례

**Spec Kit이 적합한 신호**:
- 사용자 스토리와 인수 기준을 작성할 수 있을 만큼 요구사항이 충분히 명확한 기능
- 코드 작성 전에 여러 사람이 만들 것에 합의해야 하는 경우
- 몇 주 이상 유지될 기능
- 구현 후 수정이 비쌀 경우

**다른 접근법이 더 나은 신호**:
- 결정된 솔루션을 구현하는 것이 아닌 어떤 솔루션을 사용할지 탐색 중인 경우
- 변경이 너무 작아 명세서가 필요하지 않은 경우 (한 시간 이하의 작업)
- 요구사항이 너무 불명확해 명세서 작성이 추측이 되는 경우
- 기능이 일회용인 경우 (프로토타입, 스파이크, 실험)

## 흔한 오해

1. **"Spec Kit이 나쁜 결과를 만들면 도구가 고장난 것이다."** — 나쁜 결과는 대개 나쁜 명세서, 낡은 프로젝트 규약, 또는 건너뛴 검토 단계로 추적됩니다. 도구는 설계대로 작동하고 있습니다. 개선이 필요한 것은 입력입니다.
2. **"Spec Kit은 올바른 구현을 보장한다."** — Spec Kit은 구조를 제공하고 모호성을 줄입니다. 정확성을 보장하지는 않습니다. 인수 기준에 대한 구현의 인간 검토는 항상 필요합니다.
3. **"Spec Kit 명령어를 더 많이 실행할수록 결과물이 더 좋아진다."** — 더 많은 명령어를 실행한다고 품질이 낮은 입력을 보완하지 못합니다. 간소화된 네 단계 워크플로로 작성된 좋은 명세서가 전체 여덟 단계 워크플로의 모호한 명세서보다 낫습니다.
