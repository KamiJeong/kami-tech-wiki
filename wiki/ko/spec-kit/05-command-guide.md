# Spec Kit 명령어 가이드

> 모든 Spec Kit 명령어의 기능, 사용 시기, 만들어내는 결과물에 대한 쉬운 설명.

## 이 페이지의 대상 독자

이 페이지는 Spec Kit 명령어에 대한 실용적인 참조를 원하는 개발자를 위한 글입니다. 일반적으로 사용되는 순서로 각 명령어를 다루며, 쉬운 설명과 구체적인 예시를 제공합니다.

## 왜 이 내용이 중요한가

명령어가 존재한다는 것을 아는 것은 필요한 지식의 절반에 불과합니다. 명령어가 무엇을 하는지, 어떤 입력이 필요한지, 무엇을 만들어내는지 이해하는 것이 효과적으로 사용할 수 있는지를 결정합니다. 이 가이드가 그 격차를 해소합니다.

## 쉬운 설명

Spec Kit 명령어는 Claude Code 스킬로 구현됩니다. Claude Code에서는 `/speckit-[이름]` 명령어로 나타납니다. 스킬 모드(Claude Code 스킬이 직접 호출되는 경우)에서는 슬래시 접두사 없이 참조될 수 있습니다. 이 가이드에서는 전체 명령어 형식(`/speckit-[이름]`)을 사용합니다.

## 핵심 개념

### /speckit-constitution

**무엇을 하는가**: 프로젝트 규약(프로젝트 전반의 거버넌스 문서로, 모든 기능이 따라야 하는 규칙을 정의함)을 생성하거나 업데이트합니다.

**언제 사용하는가**: 새 프로젝트 시작 시 한 번. 프로젝트의 기본 규칙이 크게 변경될 경우 다시 실행합니다 (새 기술 스택, 새 품질 표준).

**입력**: 프로젝트의 맥락, 기술 선택, 품질 표준, 아키텍처 원칙을 설명합니다.

**결과물**: `.specify/memory/constitution.md` — 이후의 모든 명령어가 읽는 프로젝트 규약.

**예시**: "Node.js REST API 프로젝트의 프로젝트 규약을 만들어줘. TypeScript, PostgreSQL, 테스팅에 Jest를 사용해. 모든 API 엔드포인트는 구현 전에 명세서가 있어야 해. 보안: 어디에도 자격증명을 하드코딩하지 않아."

---

### /speckit-specify

**무엇을 하는가**: 평문 설명에서 구조화된 기능 명세서(`spec.md`)를 생성합니다.

**언제 사용하는가**: 모든 기능의 시작 시, 계획이나 구현 전에.

**입력**: 기능에 대한 평문 설명 — 무엇을 하는지, 누가 사용하는지, 성공이 어떤 모습인지.

**결과물**: `specs/[기능명]/spec.md` — 사용자 스토리, 인수 기준, 범위 외 항목, 엣지 케이스, 의존성 포함.

**예시**: "사용자가 제목이 있는 할 일을 추가하고, 완료로 표시하고, 상태(전체/활성/완료)로 필터링하고, 삭제할 수 있는 개인 할 일 관리 앱을 만들어줘. 단일 사용자, 계정 없음, 로컬 스토리지만."

---

### /speckit-clarify

**무엇을 하는가**: 현재 `spec.md`에서 모호하거나 불충분하게 명시된 영역을 분석하고 최대 다섯 개의 목표가 정해진 명확화 질문을 합니다. 답변이 명세서에 인코딩됩니다.

**언제 사용하는가**: `/speckit-specify` 후, `/speckit-plan` 전. 명세서가 이미 완전하고 모호하지 않은 기능에서만 건너뜁니다.

**입력**: 자동으로 `spec.md`를 읽습니다. 생성하는 질문들에 답합니다.

**결과물**: 명확화 내용이 반영된 업데이트된 `spec.md`.

**예시**: Spec Kit이 물을 수 있습니다: "로컬 스토리지만'이라고 지정했습니다 — 브라우저 세션 간에 할 일이 유지되어야 하나요, 아니면 세션 전용인가요?" 답변이 계획 시작 전에 명세서를 업데이트합니다.

---

### /speckit-checklist

**무엇을 하는가**: 현재 명세서의 품질 체크리스트를 생성하고 선택적으로 완성도를 검증합니다.

**언제 사용하는가**: `/speckit-clarify` 후, 계획 전에 최종 확인으로. 명세서가 복잡하거나 여러 사람이 검토해야 할 때 가장 유용합니다.

**입력**: 자동으로 `spec.md`를 읽습니다.

**결과물**: `specs/[기능명]/checklists/requirements.md` — 모든 필수 명세서 섹션이 완성되었는지, 기준이 측정 가능한지, 엣지 케이스가 다루어졌는지 확인하는 체크리스트.

**결과물 예시**: "[ ] 상태별 필터링에 대한 인수 기준이 측정 가능합니다 ('활성'이 무엇을 의미하는지 명시됨). [X] 범위 외 항목이 식별됨. [ ] 빈 할 일 목록에 대한 엣지 케이스가 다루어짐."

---

### /speckit-plan

**무엇을 하는가**: 명세서에서 구현 계획을 생성합니다. 기술 접근법, 아키텍처, 파일 구조, 단계적 구현 순서를 설명합니다.

**언제 사용하는가**: 명세서가 완성되고 검토된 후 (명확화, 체크리스트 통과).

**입력**: 자동으로 `spec.md`와 `constitution.md`를 읽습니다.

**결과물**: `specs/[기능명]/plan.md` (구현 접근법), `research.md` (기술 결정 및 고려된 대안), `data-model.md` (해당 시 엔티티 관계).

**예시**: 할 일 관리 앱 계획은 다음을 명시할 수 있습니다: 할 일은 localStorage에 JSON으로 저장, CRUD 작업을 위한 `TaskStore` 클래스, 목록을 위한 함수형 React 컴포넌트, 별도 `utils/filters.ts` 파일의 필터 함수.

---

### /speckit-tasks

**무엇을 하는가**: 구현 계획에서 의존성 순서의 작업 목록을 생성합니다.

**언제 사용하는가**: 계획이 검토되고 승인된 후.

**입력**: 자동으로 `plan.md`와 `spec.md`를 읽습니다.

**결과물**: `specs/[기능명]/tasks.md` — 순서가 있고 독립적으로 완료 가능한 구현 단계의 체크리스트.

**예시 작업**:
```
- [ ] T001 add(), complete(), delete(), getAll() 메서드가 있는 TaskStore 클래스 생성
- [ ] T002 [P] utils/filters.ts에 filterByStatus() 유틸리티 함수 생성
- [ ] T003 filter prop을 지원하는 TaskList React 컴포넌트 생성
- [ ] T004 메인 뷰에 필터 UI (전체/활성/완료 버튼) 추가
```

---

### /speckit-analyze

**무엇을 하는가**: `spec.md`, `plan.md`, `tasks.md`를 일관성에 대해 교차 확인합니다. 불일치를 식별합니다: 해당 작업이 없는 인수 기준, 요구사항으로 추적 불가능한 작업, 작업에 의해 다루어지지 않는 계획 요소.

**언제 사용하는가**: 작업 생성 후, 구현 전. 전체 워크플로의 일부; 간소화된 워크플로에서는 선택사항.

**입력**: 자동으로 `spec.md`, `plan.md`, `tasks.md`를 읽습니다.

**결과물**: 일관성 확인 결과와 식별된 격차를 문서화하는 분석 보고서.

**예시 발견**: "인수 기준 SC-003 (빈 결과 상태를 우아하게 처리하는 필터)이 tasks.md의 어떤 작업에도 다루어지지 않았습니다. 빈 상태 UI 컴포넌트에 대한 작업 추가를 제안합니다."

---

### /speckit-implement

**무엇을 하는가**: 작업 목록을 실행하고, 각 작업을 순서대로 구현합니다. 완료된 작업을 `tasks.md`에서 `[X]`로 표시합니다.

**언제 사용하는가**: 작업 목록이 완성되고, 검토되고, 명세서와 일관성이 있을 때.

**입력**: 자동으로 `tasks.md`, `plan.md`, `spec.md`, `constitution.md`를 읽습니다.

**결과물**: `tasks.md`에 명시된 모든 파일과 코드, 완료 시 `[X]`로 표시된 작업들.

**참고**: tasks.md에서 `[P]`로 표시된 병렬 작업은 작업 설계가 허용하는 경우 동시에 실행될 수 있습니다.

---

## 비유와 예시

> Spec Kit 명령어는 레시피의 단계들과 같습니다. `/speckit-constitution`은 한 시즌 요리를 시작하기 전에 기본 식재료를 준비하는 것입니다. `/speckit-specify`는 요리를 선택하는 것입니다. `/speckit-clarify`는 레시피를 주의 깊게 읽고 쇼핑 전에 궁금한 점을 적어두는 것입니다. `/speckit-plan`은 쇼핑 목록과 준비 순서를 작성하는 것입니다. `/speckit-tasks`는 단계별 요리 지시입니다. `/speckit-analyze`는 시작하기 전에 모든 재료가 있는지 확인하는 것입니다. `/speckit-implement`는 요리입니다.

레시피나 재료 없이 요리를 시작하는 것처럼 — 계획 전에 구현하거나, 명확화 전에 계획하는 것은 순서가 맞지 않습니다.

## 실제 활용 사례

**빠른 참조**: 확신이 없을 때, 이 순서대로 명령어를 실행하세요:
1. `/speckit-constitution` (프로젝트당 한 번)
2. `/speckit-specify`
3. `/speckit-clarify`
4. `/speckit-checklist`
5. `/speckit-plan`
6. `/speckit-tasks`
7. `/speckit-analyze`
8. `/speckit-implement`
9. `/speckit-verify-run` (구현 후)

**스킬 모드 참고**: 일부 설정에서 Spec Kit 명령어는 슬래시 명령어가 아닌 Claude Code 스킬로 실행됩니다. 명령어 이름은 동일하게 유지됩니다. 호출 방법은 프로젝트에서 Claude Code가 설정된 방식에 따라 달라질 수 있습니다.

## 흔한 오해

1. **"/speckit-implement만 필요하다."** — 구현은 마지막 단계이지, 유일한 단계가 아닙니다. 구현 품질은 그에 앞서는 명세서, 계획, 작업 목록의 품질에 달려 있습니다.
2. **"명확화는 선택사항이다."** — 명확화는 단순하고 명확한 기능에서는 선택사항입니다. 모호성이 있는 기능에서 명확화를 건너뛰면 AI가 모호성을 어떻게 해소할지 선택해야 한다는 것을 의미합니다 — 그리고 그 선택이 당신의 의도와 맞지 않을 수 있습니다.
3. **"각 단계를 수동으로 검토하면 분석은 중복이다."** — 수동 검토는 명확한 격차를 잡아냅니다. `/speckit-analyze`는 세 산출물을 체계적으로 일관성에 대해 교차 확인합니다. 둘 다 가치 있고, 어느 것도 다른 것을 대체하지 않습니다.
