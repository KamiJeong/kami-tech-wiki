# Spec Kit 핵심 개념

> Spec Kit의 워크플로는 다섯 가지 핵심 개념 위에 세워져 있습니다: 프로젝트 규약(constitution, 프로젝트 규칙), 명세서(무엇을 만들 것인가), 계획(어떻게 만들 것인가), 작업(작업 단위), 품질 검증(제대로 됐는가?).

## 이 페이지의 대상 독자

이 페이지는 Spec Kit 워크플로의 각 단계가 무엇을 만들어내는지, 왜 각 단계가 존재하는지 이해하고 싶은 개발자와 팀 리더를 위한 글입니다. 소프트웨어 개발 개념에 대한 기본적인 이해가 있으면 도움이 됩니다.

## 왜 이 내용이 중요한가

Spec Kit의 핵심 개념을 이해하면 기계적이 아닌 의도적으로 툴킷을 사용할 수 있습니다. 각 산출물이 무엇을 위한 것이고 좋은 것이 어떤 모습인지 아는 것이 안정적인 결과를 내는 Spec Kit 워크플로와 같은 모호한 프롬프트의 더 잘 형식화된 버전을 만들어내는 워크플로 사이의 차이를 만듭니다.

## 쉬운 설명

Spec Kit의 워크플로는 파이프라인 산출물이라는 일련의 문서를 만들어냅니다. 각 산출물은 특정 질문에 답합니다:

- **프로젝트 규약(Constitution)** → 이 프로젝트는 항상 어떤 규칙을 따르는가?
- **명세서(spec.md)** → 이 기능은 무엇이고 '완료'는 어떤 모습인가?
- **계획(plan.md)** → 이 기능을 어떻게 만들 것인가?
- **작업(tasks.md)** → 개별 단계들은 무엇인가?
- **품질 검증** → 구현이 명세서와 일치하는가?

이 문서들은 순서대로 생성되며, 각각이 이전 것을 기반으로 합니다. 함께 기능 개발의 완전하고 문서화된 기록을 형성합니다.

## 핵심 개념

### 프로젝트 규약 (Constitution)

프로젝트 규약(constitution)은 모든 기능이 따라야 하는 규칙을 정의하는 프로젝트 전반의 거버넌스 문서입니다: 기술 선택, 코딩 규칙, 품질 표준, 아키텍처 패턴, 검토 요구사항. 프로젝트 수준에서 한 번 작성되며 이후의 모든 명세서에 상속됩니다.

**왜 중요한가**: 프로젝트 규약 없이는 모든 새 기능 명세서가 동일한 기본 규칙을 다시 명시해야 합니다. 프로젝트 규약이 있으면 그 규칙들이 자동으로 상속됩니다. 명세서는 이 기능에만 특화된 것에만 집중할 수 있습니다.

**예시**: 프로젝트 규약에 이렇게 명시될 수 있습니다: "모든 API는 TypeScript로 구현되어야 합니다. 모든 데이터베이스 쿼리는 파라미터화된 구문을 사용해야 합니다. 모든 기능은 구현 시작 전에 spec.md가 있어야 합니다." 모든 기능 명세서는 이 규칙들을 반복하지 않고 상속합니다.

**생성 명령**: `/speckit-constitution`

### 명세서 (spec.md)

명세서는 단일 기능을 설명하는 구조화된 문서입니다: 무엇인지, 누구를 위한 것인지, '완료'는 어떤 모습인지(인수 기준), 명시적으로 범위에 없는 것은 무엇인지, 어떤 엣지 케이스를 처리해야 하는지.

**왜 중요한가**: 명세서는 모든 이후 단계의 기본 입력입니다. 명확하고 정확한 명세서는 명확한 계획과 안정적인 구현을 만들어냅니다. 막연한 명세서는 막연한 계획과 수정이 필요한 구현을 만들어냅니다.

**예시**: 할 일 필터 기능에 대한 명세서에는 다음이 포함됩니다: 사용자 스토리(누가 필터를 사용하고 왜), 인수 기준("활성"으로 필터링하면 상태가 "활성"인 작업만 표시됨), 범위 외 항목(정렬은 이 기능의 범위에 포함되지 않음), 엣지 케이스(필터 결과가 없을 때 어떻게 됨).

**생성 명령**: `/speckit-specify`

### 계획 (plan.md)

계획은 명세서에서 만들어진 기술 구현 문서입니다. 기술 스택, 파일 구조, 아키텍처 접근법, 단계적 구현 순서를 설명합니다.

**왜 중요한가**: 구현 전에 계획을 검토하면 팀이 코드가 작성될 때 수정하기 비싼 아키텍처 실수를 코드가 작성되기 전에 잡을 수 있습니다. 계획은 또한 AI를 제약합니다: AI는 자신의 아키텍처 선호도가 아닌 계획에서 작업합니다.

**예시**: 할 일 필터 기능에 대한 계획에는 다음이 명시될 수 있습니다: 필터는 `filters.ts` 파일에 순수 함수로 구현되고, UI는 filter prop이 있는 기존 `TaskList` 컴포넌트를 사용하며, 필터 상태는 상태 관리 라이브러리 없이 부모 컴포넌트에서 관리됩니다.

**생성 명령**: `/speckit-plan`

### 작업 (tasks.md)

작업 목록은 계획에서 도출된 순서가 있는 독립적으로 완료 가능한 구현 단계의 체크리스트입니다. 각 작업은 검증할 수 있을 만큼 작고, 완료가 언제인지 알 수 있을 만큼 구체적이며, 명세서의 요구사항으로 추적 가능합니다.

**왜 중요한가**: 검토된 명세서에서 도출된 작업 목록은 AI에게 범위가 정해지고 검토 가능한 작업 단위를 제공합니다. 범위 확장을 방지하고(AI가 계획된 것 이상을 구현하는 것) 진행 상황을 가시적으로 만듭니다(어떤 작업이 완료되었고 어떤 것이 남았는지 확인 가능).

**필터 기능 예시 작업**:
- `filterByStatus(tasks, status)` 함수를 포함한 `filters.ts` 생성
- `filter` prop을 받도록 `TaskList` 컴포넌트 업데이트
- 메인 UI에 필터 버튼 (전체, 활성, 완료) 추가
- 빈 데이터셋, 부분 데이터셋, 전체 데이터셋으로 `filterByStatus` 테스트 작성

**생성 명령**: `/speckit-tasks`

### 품질 검증 (Quality Gate)

품질 검증은 구현이 명세서를 충족하는지 확인하는 검증 단계입니다. 완료된 작업을 인수 기준에 대해 검토하고 충족되지 않은 요구사항을 식별합니다.

**왜 중요한가**: 품질 검증 없이는 작업을 완료로 표시하고 해당 작업이 충족해야 했던 인수 기준이 여전히 실패하고 있다는 것을 놓치기 쉽습니다. 품질 검증은 개발자의 기억에 의존하는 대신 검증을 체계적으로 만듭니다.

**예시**: 품질 검증은 확인합니다: "활성"으로 필터링하면 상태가 "활성"인 작업만 표시되는가? 사용자가 이동하고 돌아올 때 필터가 리셋되는가? spec.md의 인수 기준 각각이 최소 하나의 구현된 동작으로 충족되는가?

**실행 명령**: `/speckit-verify-run`

## 비유와 예시

> Spec Kit으로 기능을 만드는 것은 그림을 의뢰하는 것과 같습니다. 프로젝트 규약은 갤러리의 상시 규칙입니다 (5피트 이하, 아카이브 자재 사용). 명세서는 의뢰 브리핑입니다 (일출 시 항구 풍경, 특정 시점에서). 계획은 예술가의 스케치입니다 (대략적인 구도, 색상 팔레트, 레이어 순서). 작업은 일일 작업 지시입니다 (캔버스 준비, 하늘 칠하기, 물 추가). 품질 검증은 그림이 납품되기 전 의뢰 브리핑에 대한 최종 검수입니다.

각 단계는 이전 단계가 제기한 질문에 답합니다. 각 단계는 다음이 시작되기 전에 검토됩니다.

## 실제 활용 사례

**프로젝트 규약 설정 (프로젝트당 한 번)**:
- `/speckit-constitution` 실행
- 문서화: 기술 스택, 코딩 규칙, 필수 품질 표준, 검토 프로세스
- git에 커밋 — 이후의 모든 명세서가 상속

**기능별 워크플로**:
1. `/speckit-specify` — 명세서 작성; 계속하기 전에 주의 깊게 검토
2. `/speckit-clarify` — 모호성 해소; 명세서 업데이트
3. `/speckit-plan` — 구현 전에 기술 접근법 검토
4. `/speckit-tasks` — 작업 목록 검토; 모든 인수 기준이 포함되었는지 확인
5. `/speckit-implement` — 구현 진행; 질문이 생기면 답변
6. `/speckit-verify-run` — 구현이 명세서를 충족하는지 확인

## 흔한 오해

1. **"프로젝트 규약은 README다."** — README는 프로젝트를 사람 독자에게 설명합니다. 프로젝트 규약은 Spec Kit 파이프라인이 읽고 시행하는 거버넌스 문서입니다. 뒤따르는 모든 명세서와 구현을 제약합니다.
2. **"명세서가 충분히 상세하면 plan.md는 선택사항이다."** — 계획은 명세서를 기술 접근법으로 변환합니다. 계획 없이는 AI가 자유롭게 접근법을 선택합니다 — 이는 프로젝트의 아키텍처와 충돌할 수 있습니다. 계획 검토 단계가 아키텍처 정렬이 이루어지는 곳입니다.
3. **"작업은 그냥 체크리스트다."** — 작업은 AI의 작업 지시입니다. 각 작업은 AI가 정확히 무엇을 해야 하는지 알고 개발자가 정확히 무엇을 검증해야 하는지 알 수 있을 만큼 구체적이어야 합니다. 막연한 작업("필터 구현")은 작업이 아닙니다 — 남은 모든 생각을 AI에게 위임하는 것입니다.
