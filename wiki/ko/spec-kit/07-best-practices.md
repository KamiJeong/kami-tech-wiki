# Spec Kit 모범 사례

> 팀이 Spec Kit에서 신뢰할 수 있고 일관된 결과를 얻는 데 도움이 되는 다섯 가지 검증된 방법 — 피해야 할 흔한 함정과 함께.

## 이 페이지의 대상 독자

이 페이지는 Spec Kit을 최소 한 번 사용해본 경험이 있고 "작동한다"에서 "안정적으로 작동한다"로 나아가고 싶은 개발자를 위한 글입니다. Spec Kit 워크플로에 대한 이해가 있다고 가정합니다.

## 왜 이 내용이 중요한가

Spec Kit의 결과물 품질은 받는 입력의 품질에 정비례합니다. 좋은 인수 기준, 현재 상태의 프로젝트 규약, 각 단계에서의 사려 깊은 검토는 신뢰할 수 있는 결과를 만들어냅니다. 막연한 기준, 낡은 프로젝트 규약, 건너뛴 검토는 Spec Kit이 방지하도록 설계된 것과 같은 즉흥적인 결과를 만들어냅니다.

아래의 방법들은 가장 흔한 Spec Kit 불만의 원인을 해결하고, 습관으로 삼을 때 가장 일관되게 영향을 미치는 것들입니다.

## 쉬운 설명

Spec Kit에서 신뢰할 수 있는 결과를 얻는 것은 설정이 아닌 습관입니다. 아래의 다섯 가지 방법은 처음에는 의도적인 노력이 필요합니다. 수십 번의 기능을 거친 후에는 자동이 됩니다 — 그리고 그 보상은 수정이 적게 필요하고 실제로 만들어진 것을 반영하는 문서를 만들어내는 AI 구현입니다.

## 핵심 개념

### 방법 1: 명세서를 승인하기 전에 인수 기준 작성

인수 기준은 명세서의 가장 중요한 필드입니다. AI가 구현하는 기준이고 품질 검증이 확인하는 것입니다. 막연한 기준 ("사용자가 할 일을 필터링할 수 있다")은 막연한 구현을 만들어냅니다. 구체적인 기준 ("'활성' 선택 시 상태가 '미완료'인 할 일만 표시됨; 개수가 즉시 업데이트됨")은 구체적인 구현을 만들어냅니다.

**왜 중요한가**: 인수 기준의 모든 모호성은 당신의 입력 없이 AI가 내리는 결정입니다. 그 결정이 마음에 들지 않을 수 있습니다.

**피해야 할 함정**: 모든 인수 기준이 측정 가능하고 모호하지 않은지 확인하기 전에 `/speckit-plan`을 실행하는 것.

### 방법 2: 기본 결정이 변경될 때 프로젝트 규약 업데이트

프로젝트 규약은 모든 기능 명세서를 관리합니다. 기술 스택이 변경되거나, 새로운 보안 요구사항이 채택되거나, 아키텍처 패턴이 더 이상 사용되지 않는 경우, 프로젝트 규약을 새 기능이 명세서로 작성되기 전에 현재 상태를 반영하도록 업데이트해야 합니다.

**왜 중요한가**: 낡은 프로젝트 규약은 오래된 기술 스택을 권장하고 더 이상 사용되지 않는 패턴을 구현하는 계획을 만들어냅니다. 명세서는 맞고, 구현은 잘못된 규칙을 따릅니다.

**피해야 할 함정**: 프로젝트 규약을 일회성 문서로 취급하는 것. 프로젝트의 기본 규칙이 변경될 때 업데이트되는 살아있는 거버넌스 문서입니다.

### 방법 3: 다음 명령어를 실행하기 전에 각 산출물 검토

각 명령어의 결과물이 다음의 입력입니다. 중간 산출물을 검토하지 않고 전체 명령어 순서를 실행하는 것이 나쁜 구현으로 가는 가장 빠른 길입니다. `/speckit-plan` 전에 `spec.md`를 검토하세요. `/speckit-tasks` 전에 `plan.md`를 검토하세요. `/speckit-implement` 전에 `tasks.md`를 검토하세요.

**왜 중요한가**: 명세서에서 잡힌 오류는 수정에 몇 분이 걸립니다. 구현 후에 잡힌 오류는 몇 시간이 걸릴 수 있습니다. 각 단계의 검토 단계가 그 레버리지가 적용되는 곳입니다.

**피해야 할 함정**: Spec Kit을 원클릭 솔루션으로 취급하는 것. 명령어들이 프로세스를 구조화하지만, 인간 검토가 프로세스를 신뢰할 수 있게 만드는 것입니다.

### 방법 4: 기능 중간에 요구사항이 변경될 때 명세서를 최신 상태로 유지

요구사항은 변합니다. 변할 때, 명령어를 다시 실행하거나 AI에게 변경을 요청하기 전에 `spec.md`를 업데이트하세요. 이는 명세서를 진실의 원천으로 유지하고 변경이 코드화만 되지 않고 문서화되도록 보장합니다.

**왜 중요한가**: 해당 명세서 변경 없이 코드가 변경되면 명세서가 구현과 동기화되지 않습니다. 명세서를 기반으로 한 미래 AI 작업은 실제 코드와 충돌하는 결과물을 만들어냅니다.

**피해야 할 함정**: 명세서를 업데이트하지 않고 AI에게 "이것만 바꿔줘"라고 요청하는 것. 작은 문서화되지 않은 변경들이 명세서와 코드 사이의 큰 격차로 쌓입니다.

### 방법 5: 패치가 아닌 기능에 Spec Kit 활용

Spec Kit은 기능을 위해 설계되었습니다: 요구사항, 인수 기준, 의미 있는 구현 계획을 가진 추가 또는 변경. 사소한 패치(오타 수정, 상수 업데이트, 변수 이름 변경)에서는 전체 Spec Kit 파이프라인이 상응하는 혜택 없이 오버헤드를 추가합니다.

**왜 중요한가**: 한 줄짜리 변경에 전체 워크플로를 적용하면 가치 없는 오버헤드가 생깁니다. 의미 있는 기능에 전체 워크플로를 적용하면 가치가 생깁니다. 판단력을 사용하세요.

**피해야 할 함정**: Spec Kit을 모든 것에 적용하거나, 아무것에도 적용하지 않는 것. 올바른 범위는 의미 있는 복잡성의 기능입니다.

## 비유와 예시

> 익숙한 비행기와 익숙한 노선에서도 매번 비행 전 체크리스트를 따르는 조종사는 서두를 때 건너뛰는 조종사보다 사고가 적습니다. 체크리스트는 긴 비행에서 조종사를 느리게 하지 않습니다. 훨씬 더 많이 느리게 할 사고를 예방합니다.

Spec Kit의 검토 단계는 비행 전 체크리스트입니다. 서두를 때 건너뛰는 것이 정확히 가장 필요한 때입니다.

## 실제 활용 사례

**1인 개발자 습관**:
- 다음 명령어를 실행하기 전에 모든 생성된 산출물을 검토하세요 — 추가 5분이 걸리더라도
- 중요한 아키텍처 결정 후 프로젝트 규약을 업데이트하세요
- 명세서를 승인하기 전에 첫 번째 인수 기준을 작성하세요 — "측정 가능"이 기준입니다

**팀 습관**:
- 풀 리퀘스트에서 명세서 검토 — 코드만이 아닌 `spec.md`와 `plan.md` 검토
- 프로젝트 규약을 유지 관리할 사람을 지정하세요 — 모두가 아니고, 아무도 아닌
- 명세서 드리프트(코드와 동기화되지 않은 명세서)를 허용 가능한 기술 부채가 아닌 버그로 취급하세요

**Spec Kit이 지속적으로 잘못된 결과를 만들 때**:
1. `spec.md`의 인수 기준을 읽으세요 — 충분히 구체적인가?
2. 프로젝트 규약을 읽으세요 — 현재 프로젝트 규칙을 반영하는가?
3. 계획을 검토하세요 — 의도하지 않은 접근법을 승인했는가?
4. 재구현하기 전에 상위 문서를 수정하세요.

## 흔한 오해

1. **"모범 사례는 팀을 위한 것이지 1인 개발자를 위한 것이 아니다."** — 1인 개발자는 명확한 명세서와 현재 프로젝트 규약으로부터 팀만큼 혜택을 받습니다. 규율은 결과물을 읽는 다른 사람에 관계없이 생각의 명확성을 강제합니다.
2. **"AI가 내 막연한 인수 기준을 잡아줄 것이다."** — AI는 작성된 기준을 기반으로 구현합니다. 측정 가능한 기준을 식별하고 그렇지 않은 것을 거부하지 않습니다. 그것이 인간 검토자의 역할입니다.
3. **"모든 명령어를 사용하면 Spec Kit을 올바르게 사용하는 것이다."** — 명령어를 실행하는 것은 필요하지만 충분하지 않습니다. Spec Kit을 효과적으로 만드는 것은 각 단계에서의 검토 품질이지, 실행된 명령어의 수가 아닙니다.
