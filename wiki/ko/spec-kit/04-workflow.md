# Spec Kit 워크플로

> Spec Kit의 전체 워크플로는 여덟 개의 명령어를 순서대로 실행합니다 — 프로젝트 규칙 수립(프로젝트 규약)부터 구현 검증까지. 위험이 낮은 기능에는 네 단계의 간소화된 버전을 사용할 수 있습니다.

## 이 페이지의 대상 독자

이 페이지는 완전한 Spec Kit 워크플로, 전체 파이프라인과 간소화된 버전을 언제 사용해야 하는지, 각 단계가 무엇을 만들어내는지 이해하고 싶은 개발자와 팀 리더를 위한 글입니다. 이 페이지를 읽기 전에 [Spec Kit 핵심 개념](03-core-concepts.md)에 익숙해지는 것을 권장합니다.

## 왜 이 내용이 중요한가

워크플로 순서를 아는 것이 중요한 이유는 각 단계가 이전 단계를 기반으로 하기 때문입니다. 검토되지 않은 계획에 대해 구현하는 것은 검토되지 않은 설계에 대해 구현하는 것을 의미합니다. 명확화되지 않은 명세서에 대해 계획하는 것은 모호한 명세서에 대해 계획하는 것을 의미합니다. 순서는 임의적이지 않습니다 — 각 단계는 나중에 수정하기 비쌀 때가 아닌 싸울 때 특정 유형의 불확실성을 해소하기 위해 존재합니다.

전체 파이프라인과 간소화된 버전을 언제 사용해야 하는지 아는 것도 중요합니다. 세 줄짜리 변경에 여덟 단계를 적용하는 것은 낭비입니다. 여섯 개의 엣지 케이스가 있는 기능에 네 단계를 적용하는 것은 위험입니다. 어떤 것을 선택할지는 각 단계가 무엇을 하는지 이해하는 것이 필요합니다.

## 쉬운 설명

Spec Kit에는 두 가지 워크플로 옵션이 있습니다:

**전체 워크플로** (8단계): 의미 있는 복잡성의 기능, 교차 관심사가 있는 기능, 또는 구현 전에 팀 검토가 필요한 기능.

```
프로젝트 규약 → 명세서 → 명확화 → 체크리스트 → 계획 → 작업 → 분석 → 구현
```

**간소화된 워크플로** (4단계): 명확한 요구사항과 간단한 구현을 가진 낮은 위험 기능.

```
명세서 → 계획 → 작업 → 구현
```

구현 후, 검증 단계(`/speckit-verify-run`)가 구현이 명세서와 일치하는지 확인합니다. 이 단계는 두 워크플로 모두에 적용됩니다.

## 핵심 개념

### 전체 워크플로 — 여덟 단계

**1단계: 프로젝트 규약** (`/speckit-constitution`)
프로젝트의 관리 규칙을 수립합니다. 프로젝트 시작 시 한 번 수행합니다. 이후의 모든 명세서가 상속합니다. 프로젝트의 프로젝트 규약이 이미 존재한다면 이 단계를 건너뜁니다.

**2단계: 명세서 작성** (`/speckit-specify`)
명세서를 작성합니다. 평문으로 기능을 설명하면 Spec Kit이 사용자 스토리, 인수 기준, 범위 외 항목, 엣지 케이스가 있는 구조화된 `spec.md`를 생성합니다.

**3단계: 명확화** (`/speckit-clarify`)
명세서의 모호성을 식별하고 해소합니다. Spec Kit이 불명확한 영역에 대해 최대 다섯 개의 목표가 정해진 질문을 합니다. 답변이 `spec.md`에 인코딩됩니다.

**4단계: 체크리스트** (`/speckit-checklist`)
명세서에 대한 품질 체크리스트를 생성합니다. 모든 필수 명세서 섹션이 완성되었는지, 인수 기준이 측정 가능한지, 엣지 케이스가 다루어졌는지 확인합니다.

**5단계: 계획** (`/speckit-plan`)
구현 계획을 생성합니다. Spec Kit이 명세서를 읽고 기술 접근법, 파일 구조, 단계적 구현 순서가 담긴 `plan.md`를 만들어냅니다.

**6단계: 작업** (`/speckit-tasks`)
작업 목록을 생성합니다. Spec Kit이 계획을 읽고 `tasks.md`를 만들어냅니다 — 순서가 있고 독립적으로 완료 가능한 구현 단계의 체크리스트.

**7단계: 분석** (`/speckit-analyze`)
모든 산출물의 일관성을 교차 확인합니다. `spec.md`, `plan.md`, `tasks.md`가 서로 일관성 있는지, 모든 인수 기준이 최소 하나의 작업에 의해 다루어졌는지 확인합니다.

**8단계: 구현** (`/speckit-implement`)
작업 목록을 실행합니다. AI가 순서대로 각 작업을 구현하고, 결과물을 명세서와 계획에 대해 확인합니다. 완료된 작업은 `tasks.md`에서 `[X]`로 표시됩니다.

### 간소화된 워크플로 — 네 단계

명확한 요구사항, 고립된 범위, 간단한 구현을 가진 기능에서는 간소화된 워크플로가 명확화, 체크리스트, 분석을 건너뜁니다:

1. `/speckit-specify` — 명세서 작성
2. `/speckit-plan` — 구현 계획 작성
3. `/speckit-tasks` — 작업 목록 생성
4. `/speckit-implement` — 작업 실행

**간소화된 워크플로를 사용할 때**:
- 기능이 명확한 요구사항으로 잘 이해됨
- 범위가 좁음 (하나 또는 두 개의 파일 영향)
- 팀이 구현 전에 접근 방식을 검토할 필요 없음
- 기능이 위험이 낮음 (무언가 잘못되면 수정하기 쉬움)

**전체 워크플로를 사용할 때**:
- 기능이 여러 컴포넌트나 파일을 건드림
- 요구사항이 부분적으로 불명확하거나 이해관계자 정렬이 필요함
- 구현 접근법이 코드가 작성되기 전에 팀 검토가 필요함
- 기능이 높은 위험을 가짐 (무언가 잘못되면 수정하기 어려움)

## 비유와 예시

> 전체 워크플로는 대륙 횡단 로드 트립을 계획하는 것과 같습니다: 지도를 확인하고, 목적지를 확정하고, 모호한 구간을 파악하고, 주유 지점을 계획하고, 차에 짐을 싣고, 경로를 검토한 후 운전합니다. 간소화된 워크플로는 두 블록 떨어진 슈퍼마켓에 가는 것과 같습니다: 어디 가는지 알고, 경로가 익숙하며, 잘못된 방향으로 가는 비용이 낮습니다.

어느 접근법이 보편적으로 옳은 것이 아닙니다. 핵심은 계획 노력을 여정의 복잡성과 위험에 맞추는 것입니다.

## 실제 활용 사례

**올바른 워크플로 선택**:

| 기능 특성 | 권장 워크플로 |
|-----------|--------------|
| 명확한 요구사항, 좁은 범위 | 간소화 (4단계) |
| 불명확한 요구사항, 여러 이해관계자 | 전체 (8단계) |
| 여러 파일/컴포넌트 영향 | 전체 (8단계) |
| 단일 파일 변경, 고립됨 | 간소화 (4단계) |
| 높은 위험 (인증, 결제, 데이터 마이그레이션) | 전체 (8단계) |
| 낮은 위험 (UI 텍스트, 스타일 업데이트) | 간소화 (4단계) |

**각 단계에서 검토하기**: 워크플로의 모든 단계에서 다음 명령어를 실행하기 전에 산출물을 검토하세요. 워크플로의 가치는 각 단계에서 멈추고 읽는 인간 검토에서 나옵니다 — 중간에 멈추지 않고 모든 명령어를 순서대로 실행하는 것이 아닙니다.

**언제 멈추고 재명확화할지**: 나중 단계에서 초기 산출물이 잘못되었다는 것이 밝혀지면(예를 들어 계획이 명세서의 모호성을 드러냄), 계속하기 전에 초기 산출물을 수정하세요. 워크플로는 반복 가능하도록 설계되었습니다, 한 번 통과 프로세스가 아닙니다.

## 흔한 오해

1. **"전체 워크플로가 항상 더 낫다."** — 사소한 변경에 여덟 단계를 적용하면 불필요한 오버헤드가 생깁니다. 워크플로를 기능의 복잡성에 맞추세요.
2. **"중간 산출물을 검토하지 않고 모든 단계를 실행할 수 있다."** — 결과물을 검토하지 않고 명령어를 실행하는 것이 Spec Kit이 나쁜 결과를 만들어내는 가장 흔한 이유입니다. 검토는 선택사항이 아닙니다 — 그것이 핵심입니다.
3. **"워크플로는 구현으로 끝난다."** — 구현이 끝이 아닙니다. 검증 단계(`/speckit-verify-run`)가 구현이 명세서와 일치하는지 확인합니다. 건너뛰면 작업을 충족하지만 의도를 놓치는 코드를 배포할 수 있습니다.
