# GitHub Spec Kit이란 무엇인가?

> GitHub Spec Kit은 SDD 툴킷입니다 — Claude Code를 구조화된 명세서 우선 워크플로로 이끄는 일련의 명령어로, 일관성 있고 검토 가능한 AI 구현을 만들어냅니다.

## 이 페이지의 대상 독자

이 페이지는 Spec Kit이 무엇인지, 무엇을 하는지, 그리고 Claude에게 직접 프롬프트를 보내는 것과 어떻게 다른지 이해하고 싶은 Claude Code 사용 개발자를 위한 글입니다. Claude Code에 대한 기본적인 이해 외에 특별한 기술 배경 지식은 필요하지 않습니다.

## 왜 이 내용이 중요한가

Claude Code는 강력한 AI 코딩 어시스턴트이지만, 구조 없이는 결과물의 품질이 프롬프트의 품질에 크게 좌우됩니다. 명확하고 상세한 프롬프트는 좋은 결과를 냅니다. 모호한 프롬프트는 모호한 결과를 냅니다. 복잡한 기능에서는 프롬프트와 결과물 사이의 수정 사이클이 부담스러울 수 있습니다.

Spec Kit은 구조를 도입합니다. 하나의 프롬프트가 하나의 결과물을 만드는 대신, Spec Kit은 정의된 워크플로를 통해 안내합니다: 명세서 작성, 모호성 해소, 구현 계획 작성, 작업 목록 생성, 그리고 구현. 각 단계는 다음의 입력이 되는 문서를 만들어냅니다. AI는 단 하나의 대화형 프롬프트가 아닌 구조화된 산출물로부터 작업합니다.

결과는 더 예측 가능한 결과물, 의도와 구현 사이의 더 명확한 추적성, 그리고 무엇이 만들어졌고 왜 만들어졌는지에 대한 문서화된 기록입니다.

## 쉬운 설명

Spec Kit은 Claude Code 스킬 세트로 구현됩니다 — Claude Code에서 워크플로의 각 단계를 실행하기 위해 호출하는 특수 명령어들. 별도의 애플리케이션이나 GitHub이 판매하는 제품이 아닙니다. 특정 저장소에 내장된 툴킷으로, Claude Code와 함께 사용하도록 설계되었습니다.

툴킷의 명령어는 특정 순서를 따릅니다: 프로젝트 규약(constitution, 프로젝트의 관리 규칙)으로 시작해, 기능에 대한 명세서를 작성하고, 모호성을 해소하고, 구현 계획을 작성하고, 계획을 작업으로 나누고, 마지막으로 작업을 구현합니다.

Spec Kit은 각 단계에서 마크다운 파일을 생성합니다 — `spec.md`, `plan.md`, `tasks.md` — 이 파일들은 코드와 함께 저장소에 존재합니다. 이 파일들은 기능 개발 과정에서 내려진 모든 결정의 문서화된 기록으로 작용합니다.

중요한 구분: Spec Kit은 공식 GitHub 제품이 아닙니다. 이름은 GitHub 호스팅 저장소에서 명세서 기반 워크플로로 사용하도록 설계된 툴킷임을 반영합니다. 상업적으로 배포되는 도구가 아닌 내부 툴킷입니다.

## 핵심 개념

### SDD 툴킷

Spec Kit은 명세서 중심 개발(SDD, Spec-Driven Development)의 구현입니다 — 코드 작성 전에 명세서를 작성하는 방법론. 각 단계를 안내하는 명령어 순서를 제공해 SDD를 실천 가능하게 만듭니다.

**왜 중요한가**: 원칙으로서의 SDD는 명확하지만 실천하는 데는 규율이 필요합니다. Spec Kit은 각 단계를 구조화하는 구체적인 명령어를 제공함으로써 SDD를 따르기 쉽게 만듭니다. 좋은 명세서를 어떻게 작성하는지 기억할 필요가 없습니다 — 툴킷이 안내해줍니다.

**예시**: 빈 문서를 열고 처음부터 명세서를 작성하는 대신, `/speckit-specify`를 실행하고 평문으로 기능을 설명합니다. Spec Kit이 구조화된 명세서 문서를 생성합니다. 검토하고 수정합니다.

### Claude Code 스킬

Spec Kit 명령어는 Claude Code 스킬로 구현됩니다 — Claude Code에 새로운 명령어를 추가하는 확장 유형. Spec Kit 명령어를 실행하면 Claude Code가 스킬의 로직을 실행하고, 이는 AI에게 특정 구조화된 작업을 수행하도록 지시합니다.

**왜 중요한가**: Spec Kit이 Claude Code 스킬로 구현되기 때문에, 이미 사용하고 있는 Claude Code 환경에서 실행됩니다. 설치하거나 배울 별도의 도구가 없습니다. 명령어들이 다른 Claude Code 명령어들과 함께 나타납니다.

**예시**: Claude Code에서 `/speckit-plan`을 실행하면 계획 생성 스킬이 트리거됩니다. 이 스킬은 기존 `spec.md`를 읽고 구현 접근법, 기술 스택 결정, 파일 구조가 담긴 `plan.md`를 생성합니다.

### 파이프라인 산출물

Spec Kit 워크플로의 각 단계는 파이프라인 산출물이라는 마크다운 파일을 만들어냅니다. 이 파일들(`spec.md`, `plan.md`, `tasks.md`)은 저장소에 존재하며 기능 개발의 문서화된 기록으로 작용합니다.

**왜 중요한가**: 파이프라인 산출물은 개발 프로세스를 감사 가능하게 만듭니다. 팀원(또는 AI)은 명세서를 읽고 무엇이 만들어졌고 왜 만들어졌는지 이해할 수 있습니다 — 코드를 추적하거나 원 작성자에게 물어볼 필요 없이.

**예시**: 기능이 만들어진 지 6개월 후, 새 개발자가 `spec.md`를 읽어 기능이 무엇을 해야 하는지 이해하고, `tasks.md`를 읽어 어떻게 구현되었는지 이해합니다. 문서가 저장소에 있지, 누군가의 기억에 있는 것이 아닙니다.

## 비유와 예시

> Spec Kit은 AI 개발을 위한 프로젝트 관리 템플릿과 같습니다. 매번 프로세스를 처음부터 생각해내는 대신, 정의된 단계들을 따릅니다: 요구사항 수집, 계획 작성, 작업으로 나누기, 실행. 템플릿이 일을 해주는 것이 아니라 일이 되는 방식을 구조화합니다.

AI가 일하는 사람입니다. Spec Kit은 일이 일관되고 문서화된 프로세스를 따르도록 보장하는 템플릿입니다.

## 실제 활용 사례

**시나리오 1 — 명확한 요구사항의 새 기능**: 개발자가 검색 기능을 추가하고 싶습니다. Claude에 직접 프롬프트를 보내는 대신, `/speckit-specify`로 구조화된 명세서를 작성하고, `/speckit-plan`으로 구현 계획을 세우고, `/speckit-tasks`로 작업 목록을 생성하고, `/speckit-implement`로 작업을 실행합니다. 각 단계에서 검토 가능한 문서를 만들어냅니다.

**시나리오 2 — 불명확한 요구사항의 기능**: 개발자가 막연한 기능 아이디어를 가지고 있습니다. `/speckit-specify`로 초기 명세서를 작성하고, `/speckit-clarify`로 계획을 시작하기 전에 모호성을 식별하고 해소합니다. 명확화 단계가 초기에 오해를 잡아냅니다.

**시나리오 3 — 팀 검토 워크플로**: 팀이 구현 시작 전에 Spec Kit이 생성한 `spec.md`와 `plan.md`를 검토합니다. 비기술적 팀원은 명세서를 읽고, 기술적 팀원은 계획을 검토합니다. AI가 코드를 작성하기 전에 두 그룹 모두 무엇이 만들어지는지 합의합니다.

## 흔한 오해

1. **"Spec Kit은 GitHub이 만든 것이다."** — Spec Kit은 공식 GitHub 제품이 아닙니다. 특정 저장소에 구축된 툴킷으로 GitHub 기반 워크플로와 함께 작동하도록 설계되었습니다. 이름은 그 맥락을 반영하지, 출처를 반영하지 않습니다.
2. **"Spec Kit이 Claude Code를 대체한다."** — Spec Kit은 Claude Code 내부에서 실행됩니다. Claude Code에 구조화된 워크플로 명령어를 추가하지, 대체하거나 경쟁하지 않습니다.
3. **"모든 기능에 Spec Kit 명령어 전부를 사용해야 한다."** — 전체 파이프라인(프로젝트 규약 → 명세서 → 명확화 → 체크리스트 → 계획 → 작업 → 분석 → 구현)은 전체 프로세스가 필요한 기능을 위한 것입니다. 작고 위험이 낮은 변경에는 간소화된 플로우(명세서 → 계획 → 작업 → 구현)가 적합합니다.

## 심화 내용

> **심화 독자를 위한 내용** — 이 섹션을 읽지 않아도 Spec Kit이 무엇인지 이해하는 데 지장이 없습니다.

Spec Kit의 명령어는 저장소의 `.claude/skills/` 디렉토리에 구현됩니다. 각 명령어는 Claude Code의 AI에게 특정 구조화된 작업을 수행하도록 지시하는 마크다운 프롬프트 파일입니다. 스킬 기반 구현은 마크다운 파일을 편집해 툴킷을 업데이트할 수 있다는 것을 의미합니다 — 컴파일이나 배포가 필요 없습니다. 이는 Spec Kit을 동작을 커스터마이즈하거나 확장하고 싶은 팀에게 유독 접근하기 쉬운 툴킷으로 만듭니다.
