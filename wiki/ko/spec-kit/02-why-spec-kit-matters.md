# 왜 Spec Kit이 중요한가?

> Spec Kit이 중요한 이유는 즉흥적인 AI 프롬프트가 모호한 구현을 만들어내기 때문입니다. 명세서 우선 방식의 Spec Kit 파이프라인은 그에 대한 실질적인 해결책입니다.

## 이 페이지의 대상 독자

이 페이지는 직접 프롬프팅 대신 Spec Kit을 도입해야 할 실무적 이유를 이해하고 싶은 Claude Code 사용 개발자를 위한 글입니다. Claude Code와 AI 보조 코딩에 대한 기본 이해가 있다고 가정합니다.

## 왜 이 내용이 중요한가

Claude Code는 능력 있는 도구입니다. 좋은 프롬프트를 주면 유용한 코드를 만들어냅니다. 그런데 "좋은 프롬프트를 주면"이라는 조건은 들리는 것보다 훨씬 높은 기준입니다. 기능을 완전하고, 정확하고, 모호함 없이 단 하나의 프롬프트로 설명하는 것은 상당한 노력이 필요합니다 — 그리고 대부분의 개발자는 그 노력을 들이지 않습니다. 짧은 설명을 입력하고, 원하던 것과 비슷한 무언가를 받고, 수정하고, 다시 프롬프트하고, 다시 수정합니다.

이 수정 사이클이 문제입니다. 각 수정 회차는 시간을 소비합니다. 수정 내용은 거의 문서화되지 않습니다. 최종 구현은 맞을 수 있지만, 그 뒤에 있는 이유 — 왜 이 설계인지, 왜 이 접근 방식인지 — 는 대화 이력 속에 사라집니다.

Spec Kit은 명세서 작성 단계를 명시적이고 구조화된 방식으로 만들어 이 사이클을 끊습니다. 모든 것을 전달하려는 하나의 프롬프트 대신, Spec Kit은 각각 특정 질문에 집중하는 일련의 단계를 통해 안내합니다. 결과는 AI가 구현하는 문서화된 명세서입니다 — 그리고 팀이 요구사항이 변경될 때 읽고, 검토하고, 업데이트할 수 있는 문서.

## 쉬운 설명

Spec Kit이 왜 중요한지 이해하는 가장 간단한 방법은 같은 작업에 대한 두 가지 접근 방식을 비교하는 것입니다.

**Spec Kit 없이**: 애플리케이션에 할 일 추적 기능을 추가하고 싶습니다. Claude Code를 열고 입력합니다: "사용자가 할 일을 추가, 완료, 삭제할 수 있는 할 일 추적 기능을 추가해줘." Claude가 구현을 생성합니다. 계획하지 않은 데이터베이스 방식을 사용하고, 기존 규칙과 다르게 함수 이름을 지었으며, 언급하는 것을 잊은 필터 기능이 없습니다. 수정합니다. 프롬프트, 수정, 프롬프트, 수정.

**Spec Kit으로**: `/speckit-specify`를 실행하고 같은 평문으로 기능을 설명합니다. Spec Kit이 사용자 스토리와 인수 기준이 있는 구조화된 명세서를 생성합니다. `/speckit-clarify`를 실행하면 Spec Kit이 필터링이 어떻게 작동해야 하는지 지정하지 않았다고 지적합니다. 명확화 질문에 답합니다. `/speckit-plan`을 실행하고 코드가 작성되기 전에 제안된 구현 접근법을 검토합니다. `/speckit-implement`를 실행하면 AI가 검토된 명세서에 따라 구현합니다. 구현이 처음에 맞거나, 거의 맞습니다.

차이는 AI의 능력이 아닙니다. 차이는 AI가 작업하는 입력의 품질입니다.

## 핵심 개념

### AI 프롬프팅의 모호함 문제

프롬프트에서의 모든 지름길은 AI가 가정으로 채우는 모호함입니다. 단순하고 고립된 작업에서는 그 가정들이 보통 수용 가능합니다. 코드베이스의 여러 부분을 건드리고, 명확하지 않은 엣지 케이스가 있거나, 기존 규칙을 따라야 하는 기능에서는 가정들이 쌓여 상당한 수정으로 이어집니다.

**왜 중요한가**: 수정 사이클은 두 가지 면에서 비쌉니다: 시간과 문서화. 시간은 명확합니다. 문서화 비용은 덜 명확합니다: Claude Code에서의 모든 비공식 수정 대화는 기록되지 않습니다. 6개월 후, 왜 구현이 그런 모습인지 아무도 모릅니다.

**예시**: 토큰 형식, 세션 길이, 로그아웃 동작, 오류 메시지를 지정하지 않고 "인증 추가"라고 프롬프팅하면 Claude가 선택해야 합니다. 그 선택이 합리적일 수 있지만 당신의 필요에 맞지 않을 수 있습니다. Spec Kit의 명확화 단계가 코드가 작성되기 전에 이 선택들을 표면에 드러냅니다.

### 문서화 격차

AI 보조 개발은 문서화된 근거 없이 코드를 생성하는 경향이 있습니다. 구현으로 이어진 대화는 결국 지워질 채팅 창에 존재합니다. Spec Kit이 생성하는 명세서 문서, 계획, 작업 목록은 저장소에 영구적으로 존재합니다.

**왜 중요한가**: 문서화된 의도 없이 만들어진 소프트웨어는 유지보수가 어렵고, 새 개발자를 온보딩하기 어려우며, 요구사항이 변경될 때 수정하기 어렵습니다. Spec Kit이 생성하는 명세서 문서는 가볍지만 지속 가능한 설계 문서의 한 형태입니다.

**예시**: Spec Kit으로 만들어진 기능 6개월 후에 새 개발자가 팀에 합류했습니다. `spec.md`를 읽어 기능이 무엇을 해야 하는지, `plan.md`를 읽어 아키텍처 결정을 이해합니다. 이 온보딩은 회의 대신 몇 분이면 됩니다.

## 비유와 예시

> Spec Kit은 법률 사무소가 재판 전에 고객에게서 받는 브리핑 문서와 같습니다. 브리핑이 법적 주장을 작성하는 것이 아닙니다 — 변호사가 합니다. 하지만 브리핑은 다음을 확립합니다: 상황은 이렇고, 사실은 이렇고, 원하는 결과는 이렇고, 제약은 이렇다. 변호사는 브리핑을 바탕으로 일합니다. 없이는 고객의 목표를 추측해야 하고 수정에 상당한 시간을 쓸 것입니다.

Spec Kit은 AI 개발을 위한 브리핑 프로세스입니다. 개발자가 브리핑(명세서)을 작성합니다. AI(Claude)는 브리핑을 바탕으로 일하는 변호사입니다.

## 실제 활용 사례

**Spec Kit이 가장 효과를 발휘하는 경우**:
- 여러 파일이나 컴포넌트를 건드리는 기능
- 명확하지 않은 엣지 케이스가 있는 기능 (인증, 결제, 데이터 유효성 검사)
- 구현 전에 팀이 검토해야 하는 기능
- 나중에 유지보수하거나 확장해야 할 기능

**더 간단한 접근법이 적합한 경우**:
- 고립된 낮은 위험 변경 (변수 이름 변경, 텍스트 문자열 업데이트)
- 버릴 탐색적 프로토타입
- 하나의 명확한 구현이 있는 작업

## 흔한 오해

1. **"Spec Kit이 개발을 느리게 만든다."** — Spec Kit은 시간을 개발 끝(수정 사이클)에서 처음(명세서 작성)으로 이동시킵니다. 의미 있는 복잡성의 기능에서는 전체 시간이 일반적으로 더 짧습니다.
2. **"명세서 문서는 오버헤드다."** — 명세서 문서는 프로세스의 작업 산출물이지 오버헤드가 아닙니다. AI의 결과물을 검토 가능하고, 수정 가능하고, 문서화되게 만드는 것입니다.
3. **"좋은 프롬프트를 작성하면 직접 프롬프팅으로도 충분하다."** — 복잡한 기능에 대한 완전하고 정확하며 모호함 없는 프롬프트를 작성하는 것은 명세서를 작성하는 것만큼 많은 생각이 필요합니다. Spec Kit은 그 생각을 구조화하고 지속 가능한 문서를 부산물로 만들어냅니다.

## 심화 내용

> **심화 독자를 위한 내용** — 이 섹션을 읽지 않아도 Spec Kit이 왜 중요한지 이해하는 데 지장이 없습니다.

Spec Kit의 가치는 프로젝트 복잡성에 따라 커집니다. 10줄 변경에서는 전체 파이프라인의 오버헤드가 예방하는 수정 사이클 비용을 초과합니다. 교차 관심사가 있는 200개 작업 기능에서는 파이프라인이 방지하는 첫 번째 수정 사이클에서 비용을 회수합니다. 기능별 수정 사이클을 추적하는 팀들은 일관되게 전체 명세서로 만들어진 기능이 직접 프롬프트로 만들어진 기능보다 구현 후 수정이 더 적게 필요하다는 것을 발견합니다.
