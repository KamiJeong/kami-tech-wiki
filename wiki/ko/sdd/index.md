# Spec-Driven Development (SDD)

> 코드를 작성하기 전에 명세를 먼저 작성하고, 그 명세를 기준으로 구현·검증·반복하는 소프트웨어 개발 방법론.

---

## 개요

Spec-Driven Development (SDD)는 소프트웨어 개발에서 명세(Specification)를 최우선 산출물로 다루는 방법론입니다. SDD에서는 기능 구현에 앞서 "무엇을 만들 것인가"를 먼저 명확하게 정의합니다. 이 명세는 개발이 진행되는 동안 살아있는 문서(Living Document)로서 지속적으로 갱신됩니다.

전통적인 개발 방식에서는 개발자가 구두 요구사항이나 이슈 티켓을 참고하여 코드를 작성하고, 완성 후 문서를 보완하는 경우가 많습니다. SDD는 이 순서를 뒤집어, 명세 → 구현 → 검증 → 반복의 사이클을 명확히 정의합니다.

SDD는 특히 AI 에이전트가 코드를 생성하는 환경에서 큰 강점을 발휘합니다. AI가 모호한 구두 지시 대신 구조화된 명세를 입력으로 받을 때, 결과물의 정확도와 일관성이 크게 향상됩니다. Kiro와 Spec-Kit는 각각 이 방법론을 실무에서 구현하는 대표적인 도구입니다.

---

## 핵심 원리

### 1. 명세 우선 (Spec-First)

모든 기능은 구현 전에 명세로 정의됩니다. 명세에는 기능 요구사항(FR), 성공 기준(SC), 수락 시나리오(Acceptance Scenario)가 포함됩니다. 개발자나 AI 에이전트는 이 명세를 기준으로 구현 작업을 시작합니다.

명세 우선 원칙은 "무엇을 만들어야 하는가"와 "어떻게 만들 것인가"를 명확히 분리합니다. 이로써 팀원, 스테이크홀더, AI 모두가 동일한 목표를 공유할 수 있습니다.

### 2. 살아있는 명세 (Living Specification)

SDD의 명세는 초기 작성 후 방치되지 않습니다. 구현 과정에서 새로운 제약이나 요구사항이 발견될 때마다 명세가 먼저 업데이트되고, 구현이 그 뒤를 따릅니다. 이 원칙은 문서와 코드 간의 괴리(Documentation Drift)를 방지합니다.

### 3. 반복적 정제 (Iterative Refinement)

SDD는 단번에 완벽한 명세를 작성하려 하지 않습니다. 명세 → 구현 → 검증의 사이클을 반복하며 점진적으로 정확도를 높입니다. 각 반복 주기에서 명세의 모호한 부분이 명확해지고, 구현이 검증을 통해 확인됩니다.

### 4. 검증 가능성 (Verifiability)

SDD에서 모든 요구사항은 검증 가능한 형태로 작성됩니다. "사용자가 만족한다"가 아니라 "응답 시간이 200ms 이내여야 한다"처럼 측정 가능한 기준을 사용합니다. 이는 자동화된 검증과 AI 에이전트의 자체 검증을 가능하게 합니다.

## 실제 적용 예시

다음은 SDD 방식으로 새 기능을 개발하는 일반적인 시나리오입니다.

```
시나리오: 로그인 기능 개발

1. [명세 작성]
   - FR-001: 사용자는 이메일과 비밀번호로 로그인할 수 있어야 한다.
   - FR-002: 로그인 실패 시 사용자에게 명확한 오류 메시지를 표시해야 한다.
   - SC-001: 올바른 자격증명으로 로그인 시 2초 이내에 대시보드로 이동한다.
   - SC-002: 잘못된 자격증명 입력 시 "이메일 또는 비밀번호가 올바르지 않습니다" 메시지를 표시한다.

2. [구현]
   - 명세를 기반으로 로그인 API, UI 컴포넌트, 오류 처리 로직 구현

3. [검증]
   - SC-001, SC-002를 기준으로 수동 및 자동 테스트 실행
   - 검증 실패 항목은 명세와 구현을 모두 재검토

4. [반복]
   - 검증에서 발견된 모호한 요구사항을 명세에 반영
   - 다음 사이클에서 재구현 및 재검증
```

## 관련 개념 비교

SDD는 다른 소프트웨어 개발 방법론과 다음과 같이 구분됩니다.

| 구분 | SDD | BDD (Behavior-Driven Development) | TDD (Test-Driven Development) | Ad-hoc 개발 |
|------|-----|----------------------------------|-------------------------------|-------------|
| 시작점 | 기능 명세 (spec.md) | 사용자 행동 시나리오 (Gherkin) | 실패하는 단위 테스트 | 구두 요구사항 또는 이슈 |
| 주요 산출물 | 명세 문서, 구현 코드, 검증 보고서 | Feature 파일, 스텝 정의, 구현 코드 | 테스트 코드, 구현 코드 | 구현 코드 (문서는 선택) |
| AI 에이전트 친화성 | 매우 높음 — 구조화된 명세가 입력 | 중간 — Gherkin 파싱 필요 | 낮음 — 테스트 코드가 주요 맥락 | 낮음 — 모호한 지시 |
| 문서-코드 동기화 | 명세가 항상 최신 상태 유지 | Feature 파일이 문서 역할 | 테스트가 유일한 문서 | 문서와 코드가 쉽게 괴리 |
