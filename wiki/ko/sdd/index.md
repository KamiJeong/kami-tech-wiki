# Spec-Driven Development (SDD)

> **Spec-Driven Development (SDD)**란, 코드를 작성하기 전에 명세서(specification)를 먼저 작성하고, 그 명세서를 기준으로 구현·검증·반복하는 소프트웨어 개발 방법론입니다.

---

## 개요

많은 팀이 소프트웨어를 만드는 방식은 이렇습니다. 누군가 "이런 기능이 필요해요"라고 말하면, 개발자는 바로 코드를 작성하기 시작합니다. 문서는 나중에 작성하거나 아예 건너뜁니다. 이 방식은 소규모 프로젝트에서는 빠르게 느껴지지만, 팀이 커지고 AI 코딩 도구를 사용하기 시작하면 큰 비용으로 돌아옵니다.

Spec-Driven Development(이하 SDD)는 이 순서를 바꿉니다. 먼저 **무엇을 만들 것인가**를 명세서로 정의하고, 그 명세서를 기준으로 구현하고, 구현 결과가 명세서와 일치하는지 검증합니다. AI 코딩 에이전트(AI coding agent — 사람의 지시를 받아 코드를 자동으로 작성하는 AI 소프트웨어 도구, 예: Claude Code, GitHub Copilot)가 코드를 생성하는 환경에서는 이 접근 방식이 특히 중요합니다. AI는 모호한 지시보다 구조화된 명세서를 받을 때 훨씬 정확하고 일관된 결과를 냅니다.

이 문서는 SDD가 무엇인지, 왜 중요한지, 어떻게 동작하는지, 그리고 다른 방법론과 어떻게 다른지를 개발자가 아닌 독자도 이해할 수 있게 설명합니다.

---

## 비개발자를 위한 쉬운 비유

SDD를 이해하는 가장 쉬운 방법은 **건축 설계도**로 생각하는 것입니다.

건물을 짓기 전에 건축가는 설계도를 먼저 그립니다. 설계도에는 방의 크기, 문과 창문의 위치, 전기 배선 계획 등이 상세하게 담겨 있습니다. 시공팀은 이 설계도를 기준으로 일합니다. 현장에서 문제가 생기면 설계도를 먼저 수정하고, 시공이 그 뒤를 따릅니다.

설계도 없이 건물을 짓는 팀을 상상해 보세요. 각자 머릿속의 계획대로 일하다 보면, 배관이 전선을 막고, 방의 크기가 맞지 않으며, 결국 다 짓고 나서 뜯어고쳐야 하는 상황이 벌어집니다.

소프트웨어 개발도 마찬가지입니다. 명세서는 건축 설계도입니다. SDD는 "설계도부터 그리고 짓는" 방식입니다.

---

## AI 코딩 에이전트란?

SDD를 이야기할 때 자주 등장하는 **AI 코딩 에이전트(AI coding agent)**를 먼저 이해하면 SDD의 필요성이 더 명확해집니다.

AI 코딩 에이전트는 개발자(또는 기획자)의 지시를 받아 코드를 자동으로 작성·수정·실행하는 AI 소프트웨어 도구입니다. 예를 들어 "사용자 로그인 기능을 만들어 줘"라고 입력하면, AI가 실제 코드를 작성해 줍니다. 대표적인 도구로는 Claude Code, GitHub Copilot, Kiro 등이 있습니다.

AI 코딩 에이전트의 능력은 2024년 이후 급격히 향상되었습니다(Anthropic Claude Code 문서 기준, 2025년 8월). 하지만 이 도구들에는 공통적인 약점이 있습니다. **지시가 모호하면 결과도 모호합니다.** "좋은 로그인 기능"이 어떤 것인지 AI가 스스로 판단할 수 없습니다. 구체적이고 구조화된 기준이 있어야 일관된 결과를 냅니다. SDD의 명세서가 바로 그 기준입니다.

---

## SDD 워크플로우 — 6단계

SDD는 다음 6단계 순서로 진행됩니다. 각 단계는 다음 단계의 입력이 됩니다.

1. **원칙 문서 (Constitution)**: 프로젝트 전체에 적용되는 규칙과 가이드라인을 문서화합니다. "이 프로젝트에서는 어떤 원칙으로 개발하는가?"를 정의하는 단계입니다. 원칙 문서(constitution)는 이후 모든 명세서의 기준이 됩니다.

2. **명세 작성 (Specify)**: 개별 기능에 대한 명세서(specification)를 작성합니다. "무엇을 만들 것인가, 성공은 어떻게 측정하는가, 어디까지가 범위인가"를 답합니다.

3. **계획 수립 (Plan)**: 명세서를 바탕으로 기술적 구현 계획을 세웁니다. "어떻게 만들 것인가"를 결정하는 단계입니다.

4. **작업 목록 생성 (Tasks)**: 계획을 실행 가능한 단위 작업으로 분해합니다. 각 작업은 독립적으로 완료 여부를 확인할 수 있어야 합니다.

5. **구현 (Implement)**: 작업 목록에 따라 코드를 작성합니다. AI 코딩 에이전트가 이 단계를 자동화할 수 있습니다.

6. **검토 (Review)**: 구현 결과가 명세서의 성공 기준을 충족하는지 검증합니다. 문제가 발견되면 명세서부터 수정하고 다음 사이클을 시작합니다.

이 6단계는 한 번으로 끝나지 않습니다. 검토 결과를 명세 작성에 반영하며 반복합니다.

## 명세서(명세서)란?

**명세서(specification)**는 "무엇을 만들 것인가"를 구체적으로 기술한 문서입니다. 명세서는 다음 질문에 답합니다.

- 이 기능은 어떤 사용자를 위한 것인가?
- 이 기능이 제대로 동작한다는 것을 어떻게 알 수 있는가? (성공 기준)
- 어떤 상황이 이 기능의 범위 밖인가?
- 예상되는 예외 상황은 무엇인가?

SDD에서 명세서는 한 번 쓰고 잊는 문서가 아닙니다. 개발이 진행되면서 새로운 정보가 생기면 명세서를 먼저 업데이트하고 구현이 따라옵니다. 이런 방식을 **살아있는 명세서(Living Specification)**라고 합니다.

---

## 원칙 문서(원칙 문서)란?

SDD에서 **원칙 문서(constitution)**는 프로젝트 전체에 걸쳐 적용되는 개발 원칙과 제약 조건을 담은 문서입니다. 건물로 따지면 건축 법규나 설계 기준에 해당합니다.

원칙 문서에는 예를 들어 다음과 같은 내용이 담깁니다.

- 이 프로젝트에서 사용하는 기술 스택
- 보안 원칙 (예: 개인정보를 코드에 하드코딩하지 않는다)
- 문서화 기준 (예: 모든 API는 명세서가 있어야 한다)
- 리뷰 프로세스

원칙 문서가 있으면 개별 기능의 명세서를 작성할 때마다 같은 기본 원칙을 반복할 필요가 없습니다. 모든 명세서는 원칙 문서를 상속합니다.

## 방법론 비교: SDD vs 전통적 개발 vs TDD vs BDD vs 바이브 코딩

아래 표는 주요 소프트웨어 개발 방법론을 비교합니다.

| 비교 항목 | 전통적 개발 | TDD (Test-Driven Development, 테스트 주도 개발) | BDD (Behavior-Driven Development, 행동 주도 개발) | SDD | 바이브 코딩 |
|-----------|------------|-----------------------------------------------|--------------------------------------------------|-----|-----------|
| **시작점** | 구두 요구사항 또는 티켓 | 실패하는 단위 테스트 | 사용자 행동 시나리오 (Gherkin 언어) | 구조화된 명세서 | AI에게 자연어로 요청 |
| **주요 산출물** | 구현 코드 (문서는 나중에) | 테스트 코드 + 구현 코드 | Feature 파일 + 구현 코드 | 명세서 + 구현 코드 + 검증 보고서 | 구현 코드 (명세서 없음) |
| **AI 에이전트 적합성** | 낮음 — 모호한 지시 | 낮음 — 테스트 코드가 주요 맥락 | 중간 — Gherkin 파싱 필요 | 매우 높음 — 구조화된 입력 | 중간 — 즉각적이나 일관성 낮음 |
| **문서-코드 동기화** | 문서가 코드와 쉽게 어긋남 | 테스트가 유일한 문서 역할 | Feature 파일이 문서 역할 | 명세서가 항상 최신 상태 유지 | 문서 없음 |
| **적합한 팀 규모** | 소규모 ~ 대규모 | 소규모 ~ 중규모 | 중규모 ~ 대규모 | 소규모 ~ 대규모 | 개인 ~ 소규모 |
| **학습 비용** | 낮음 | 중간 (테스트 작성 역량 필요) | 높음 (Gherkin 언어 학습 필요) | 중간 (명세서 작성 훈련 필요) | 매우 낮음 |

**바이브 코딩(vibe coding)**이란, AI 코딩 도구에게 "느낌대로(vibe)" 자연어로 지시하며 코드를 만들어 나가는 방식을 말합니다. 명세서 없이 대화하듯 AI에게 요청합니다. 빠르고 직관적이지만, 결과물의 일관성과 유지보수성이 떨어질 수 있습니다. SDD는 바이브 코딩을 완전히 대체하는 것이 아니라, 바이브 코딩의 속도를 유지하면서 구조를 더하는 방법입니다.

TDD(테스트 주도 개발, Test-Driven Development)와 BDD(행동 주도 개발, Behavior-Driven Development)도 "코드보다 먼저 무언가를 작성"하는 방법론입니다. TDD는 테스트를, BDD는 사용자 행동 시나리오를 먼저 작성합니다. SDD는 이보다 더 상위 수준에서 "기능 전체의 명세"를 먼저 작성하며, TDD나 BDD와 함께 사용할 수 있습니다.

## SDD를 사용해야 할 때

다음 상황에서 SDD는 특히 효과적입니다.

- **AI 코딩 에이전트를 사용하는 팀**: AI에게 명확한 기준을 제공하면 반복 수정 작업이 줄어듭니다.
- **여러 사람이 협업하는 프로젝트**: 명세서가 팀원 간의 공통 언어가 됩니다.
- **요구사항이 자주 바뀌는 환경**: 명세서를 먼저 업데이트하면 변경의 영향 범위를 파악하기 쉽습니다.
- **장기 유지보수가 필요한 소프트웨어**: 살아있는 명세서가 시간이 지나도 코드의 의도를 설명합니다.
- **비기술 이해관계자와 함께 일하는 팀**: 명세서는 코드를 모르는 사람도 읽고 피드백할 수 있는 문서입니다.

---

## SDD를 사용하지 않아도 될 때

모든 상황에 SDD가 맞는 것은 아닙니다. 다음 경우에는 더 가벼운 접근 방식이 나을 수 있습니다.

- **하루 이틀 만에 버릴 프로토타입**: 명세서 작성 시간이 구현 시간보다 길어질 수 있습니다.
- **완전히 혼자 하는 소규모 실험**: 머릿속의 계획이 충분한 경우.
- **요구사항 자체가 아직 불명확한 초기 탐색 단계**: 이 단계에서는 바이브 코딩으로 빠르게 탐색한 뒤, 방향이 잡히면 SDD로 전환하는 것이 효과적입니다.

## 흔한 오해

**오해 1: "SDD는 문서 작업을 너무 많이 요구한다"**

SDD의 명세서는 긴 보고서가 아닙니다. 핵심 질문에 답하는 구조화된 짧은 문서입니다. 처음에는 낯설지만, 익숙해지면 오히려 불필요한 재작업이 줄어 전체 개발 시간이 단축됩니다.

**오해 2: "SDD는 TDD나 BDD를 대체한다"**

SDD는 TDD나 BDD와 경쟁하지 않습니다. SDD는 더 상위 수준의 프레임워크입니다. TDD의 테스트 작성이나 BDD의 시나리오 작성을 SDD 워크플로우 안에서 함께 사용할 수 있습니다.

**오해 3: "AI가 있으면 명세서가 필요 없다"**

AI 코딩 에이전트는 지시에 따라 코드를 만들지만, 무엇을 만들어야 하는지를 판단하지 않습니다. 명세서 없이 AI에게 작업을 맡기면 AI는 자신의 추측대로 구현합니다. 명세서가 없을수록 AI의 결과물을 수정하는 시간이 늘어납니다.

**오해 4: "SDD는 애자일 방법론과 충돌한다"**

SDD는 애자일(Agile) 방법론과 잘 어울립니다. 스프린트마다 명세서를 작성하고, 구현하고, 검토하는 과정이 애자일의 반복 주기와 자연스럽게 맞습니다. SDD는 속도를 늦추는 것이 아니라, AI 시대의 애자일을 더 효과적으로 만드는 방법입니다.

## 실제 예시 — 비개발자도 이해할 수 있는 시나리오

**상황**: 온라인 쇼핑몰의 기획자 김민지 씨가 "주문 상태 알림 기능"을 추가하고 싶습니다.

**바이브 코딩 방식**: 김민지 씨가 AI에게 "주문이 배송되면 고객한테 알림 보내줘"라고 요청합니다. AI는 이메일 알림 코드를 만들었지만, 김민지 씨는 카카오톡 알림을 원했고, 알림 시점도 배송 출발이 아니라 배송 완료 시점을 원했습니다. 다시 요청하고, 다시 수정하는 과정이 반복됩니다.

**SDD 방식**: 김민지 씨가 먼저 명세서를 작성합니다.
- **기능**: 주문 상태 변경 시 고객 알림
- **알림 채널**: 카카오톡 (이메일은 범위 외)
- **알림 시점**: 배송 완료 시 (배송 출발 시는 범위 외)
- **성공 기준**: 배송 완료 후 5분 이내에 고객의 카카오톡으로 알림이 전송된다.

이 명세서를 받은 AI 코딩 에이전트는 처음부터 올바른 방향으로 구현합니다. 수정 횟수가 크게 줄어들고, 결과물이 처음 의도와 일치합니다.

## 바이브 코딩(바이브 코딩)이란?

**바이브 코딩(vibe coding)**은 AI 코딩 도구에게 자연어로 자유롭게 지시하며 코드를 만들어 나가는 방식을 말합니다. 형식적인 명세서나 계획 없이 즉흥적으로(vibe) AI와 대화하며 개발합니다.

바이브 코딩은 아이디어를 빠르게 프로토타입으로 만들거나, 개인 프로젝트를 탐색할 때 매우 효과적입니다. 하지만 팀 규모가 커지거나, 장기 유지보수가 필요하거나, AI에게 복잡한 작업을 맡길수록 일관성 문제가 생깁니다.

SDD는 바이브 코딩의 적이 아닙니다. 바이브 코딩의 빠른 속도를 유지하면서, 명세서라는 구조를 더해 결과물의 품질과 일관성을 높이는 접근 방식입니다. 많은 팀이 초기 탐색 단계는 바이브 코딩으로, 실제 개발 단계는 SDD로 진행합니다.

---

## SDD의 장점

- **재작업 감소**: 명세서가 방향을 사전에 정의하므로, 잘못 만든 기능을 다시 만드는 비용이 줄어듭니다.
- **AI 활용 효율 향상**: 구조화된 명세서를 입력으로 받는 AI 에이전트는 모호한 지시를 받을 때보다 훨씬 정확한 결과를 냅니다.
- **팀 간 소통 개선**: 명세서가 기술팀과 비기술팀 사이의 공통 언어 역할을 합니다.
- **변경 관리 용이**: 요구사항이 바뀌면 명세서를 먼저 업데이트하여 영향 범위를 파악합니다.
- **지식 보존**: 개발자가 팀을 떠나도 명세서가 시스템의 의도를 기록합니다.

---

## 위험 요소와 한계

- **초기 진입 비용**: 명세서 작성에 익숙해지기까지 시간이 필요합니다. 팀 전체가 명세서를 제대로 작성하는 방법을 익혀야 합니다.
- **과도한 명세화 위험**: 모든 작은 결정까지 명세서로 만들려 하면 오히려 속도가 느려집니다. 명세서는 핵심 요구사항에 집중해야 합니다.
- **명세서-구현 불일치**: 명세서를 업데이트하지 않으면 코드와 명세서가 어긋납니다. 이는 기존 문서화 문제와 동일하며, 팀의 규율이 필요합니다.
- **단기 프로젝트에서의 overhead**: 며칠 안에 끝나는 작업에 전체 SDD 워크플로우를 적용하면 과도할 수 있습니다.

---

## 주요 용어 정리

| 용어 | 설명 |
|------|------|
| **명세서 (Specification)** | 기능의 요구사항, 성공 기준, 범위를 기술한 문서. SDD의 핵심 산출물. |
| **원칙 문서 (Constitution)** | 프로젝트 전체에 적용되는 개발 원칙과 제약 조건을 담은 문서. |
| **AI 코딩 에이전트 (AI Coding Agent)** | 사람의 지시를 받아 코드를 자동으로 작성·수정하는 AI 소프트웨어 도구. |
| **바이브 코딩 (Vibe Coding)** | 형식적인 명세서 없이 AI에게 자연어로 자유롭게 지시하며 개발하는 방식. |
| **파이프라인 (Pipeline)** | SDD의 6단계(Constitution → Specify → Plan → Tasks → Implement → Review) 순서대로 진행되는 작업 흐름. |
| **TDD (테스트 주도 개발, Test-Driven Development)** | 구현 전에 실패하는 테스트를 먼저 작성하는 개발 방법론. |
| **BDD (행동 주도 개발, Behavior-Driven Development)** | 사용자 행동 시나리오를 먼저 기술하는 개발 방법론. |
| **살아있는 명세서 (Living Specification)** | 개발이 진행되면서 지속적으로 업데이트되는 명세서. |

## 관련 문서

SDD를 실제로 구현하는 도구로 [Spec Kit](../spec-kit/index.md)과 [Kiro](../kiro/index.md)가 있습니다. Spec Kit은 Claude Code 환경에서 SDD 파이프라인을 슬래시 명령으로 제공합니다. [Claude Code](../claude-code/index.md)는 Spec Kit이 동작하는 AI 코딩 에이전트 환경입니다. 프롬프트 엔지니어링의 기초는 Prompt Engineering 기초 문서(TBD — 아직 작성되지 않음)를 참고하세요.
