## 1. 시스템 요구사항 및 설치

### 요구사항

| 항목 | 최소 | 권장 |
|------|------|------|
| 운영체제 | macOS 13+, Ubuntu 22.04+, Windows WSL2 | macOS 14+, Ubuntu 24.04+ |
| Node.js | 22.0 | 22 LTS |
| RAM | 2 GB | 4 GB |
| 인터넷 연결 | 필수 | — |

### 설치

```bash
# Codex CLI 설치 (npm, 권장)
npm install -g @openai/codex

# 버전 확인
codex --version

# Homebrew 설치 (macOS, Computer Use 기능 제외)
brew install --cask codex
```

> [!warning] 패키지 이름 주의
> npm의 `codex`(스코프 없음)는 2012년의 무관한 패키지입니다. 반드시 `@openai/codex`를 사용하세요.

## 2. 인증 (Authentication)

Codex CLI는 두 가지 인증 방법을 지원한다.

### 방법 A: ChatGPT 계정 (플러스/프로/팀/엔터프라이즈)

```bash
# 대화형 모드 실행 시 브라우저가 열리고 ChatGPT 로그인 화면이 나타난다
codex
```

로그인 완료 후 토큰이 로컬에 저장되며, 이후 실행 시 자동으로 인증된다.

### 방법 B: OpenAI API 키

```bash
# 환경 변수로 설정 (권장)
export OPENAI_API_KEY="sk-..."

# 또는 ~/.codex/config.json 파일에 저장
```

```json
{
  "apiKey": "sk-..."
}
```

> [!tip] 보안 권고
> API 키를 코드나 Git 저장소에 직접 포함하지 말 것. 환경 변수 또는 비밀 관리 도구(예: 1Password, AWS Secrets Manager)를 사용하라.

### 인증 상태 확인

```bash
# CLI 세션 내부에서
/status
```

`/status` 명령은 첫 메시지 전송 이후에만 사용량 데이터를 표시한다. 초기 실행 시 비어 있는 것은 정상이다.

## 3. 승인 모드 (Approval Modes)

승인 모드는 에이전트가 파일 시스템과 셸 명령에 얼마나 자율적으로 접근할지 결정한다.

| 모드 | 설명 | 적합한 상황 |
|------|------|------------|
| `suggest` (기본값) | 모든 변경을 제안만 하고 실행하지 않음 | 처음 사용 시, 민감한 코드베이스 |
| `auto-edit` | 파일 수정은 자동, 셸 명령은 승인 필요 | 일반적인 개발 작업 |
| `full-auto` | 파일 수정 및 셸 명령 모두 자동 실행 | 신뢰할 수 있는 자동화 파이프라인 |

### 모드 변경

```bash
# 시작 시 모드 지정
codex --approval-mode auto-edit

# 세션 중 변경
/permissions
```

> [!warning] full-auto 모드 주의
> `full-auto` 모드에서는 파일 삭제, 패키지 설치, 셸 스크립트 실행이 모두 자동으로 이루어진다. 신뢰할 수 있는 환경에서만 사용하라.

## 4. 슬래시 명령 (Slash Commands)

대화 세션 중 슬래시 명령으로 에이전트 동작을 제어할 수 있다.

| 명령 | 설명 |
|------|------|
| `/new` | 현재 세션의 컨텍스트를 초기화하고 새 세션 시작 |
| `/resume` | 이전 세션을 목록에서 선택해 재개 |
| `/plan` | 현재 태스크의 실행 계획을 텍스트로 출력 |
| `/review` | 변경된 파일의 diff를 출력 |
| `/diff` | `git diff` 결과를 에이전트 컨텍스트에 포함 |
| `/permissions` | 현재 승인 모드를 확인하고 변경 |
| `/status` | 인증 상태, 모델, 사용량 표시 |
| `/help` | 전체 명령 목록 출력 |

### 사용 예시

```
> /plan
에이전트가 다음을 수행할 예정입니다:
1. src/api/auth.ts의 JWT 검증 로직 수정
2. tests/auth.test.ts에 단위 테스트 추가
3. 변경 사항을 git commit
```

## 5. AGENTS.md 작성법

`AGENTS.md`는 Codex 에이전트에게 프로젝트 규칙·컨벤션·금지 사항을 알려주는 설정 파일이다. 저장소 루트에 두면 에이전트가 자동으로 읽는다.

### 기본 구조

```markdown
# AGENTS.md

## 프로젝트 개요
이 저장소는 TypeScript 기반 REST API 서버입니다.

## 코딩 컨벤션
- 언어: TypeScript strict mode
- 포매터: Prettier (`.prettierrc` 참조)
- 린터: ESLint (`eslint.config.js` 참조)
- 테스트: Vitest, 커버리지 80% 이상 유지

## 금지 사항
- `any` 타입 사용 금지
- `console.log` 프로덕션 코드에 남기지 말 것
- `main` 브랜치에 직접 커밋 금지

## PR 규칙
- 제목 형식: `feat:` / `fix:` / `refactor:` 접두어 사용
- 본문에 변경 이유와 테스트 방법 포함

## 참고하지 말아야 할 파일
- `legacy/` 디렉토리 (마이그레이션 예정)
- `*.bak` 파일
```

### AGENTS.md 위치 우선순위

1. 현재 작업 디렉토리의 `AGENTS.md`
2. 저장소 루트의 `AGENTS.md`
3. 홈 디렉토리의 `~/.codex/AGENTS.md` (전역 설정)

> [!tip] AGENTS.md 활용 팁
> 에이전트가 자주 실수하는 패턴을 `## 금지 사항` 섹션에 명시적으로 추가하면 반복 오류를 줄일 수 있다.

## 6. 설정 파일 (Config)

`~/.codex/config.json` 또는 프로젝트 루트의 `.codex/config.json`으로 기본값을 설정한다.

```json
{
  "model": "codex-1",
  "approvalMode": "auto-edit",
  "notify": true,
  "contextWindow": {
    "compressionThreshold": 0.85
  },
  "env": {
    "EDITOR": "vim"
  }
}
```

### 주요 설정 옵션

| 옵션 | 기본값 | 설명 |
|------|--------|------|
| `model` | `codex-1` | 사용할 모델 |
| `approvalMode` | `suggest` | 기본 승인 모드 |
| `notify` | `false` | 태스크 완료 시 데스크탑 알림 |
| `contextWindow.compressionThreshold` | `0.90` | 컨텍스트 압축 시작 비율 |

## 7. MCP 통합

MCP (Model Context Protocol)를 사용하면 Codex에 외부 도구를 연결할 수 있다.

### MCP 서버 설정

`.codex/config.json` 또는 `~/.codex/config.json`에 추가:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/workspace"]
    },
    "database": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "${DATABASE_URL}"
      }
    }
  }
}
```

### 사용 가능한 MCP 서버 예시

| 서버 | 패키지 | 용도 |
|------|--------|------|
| GitHub | `@modelcontextprotocol/server-github` | 이슈·PR·코드 검색 |
| 파일시스템 | `@modelcontextprotocol/server-filesystem` | 지정 디렉토리 접근 |
| PostgreSQL | `@modelcontextprotocol/server-postgres` | DB 스키마·쿼리 |
| Brave Search | `@modelcontextprotocol/server-brave-search` | 웹 검색 |

## 8. 자동화 패턴

### Non-Interactive 모드 (CI/CD)

```bash
# 단일 태스크 자동 실행 (승인 없이)
codex exec --full-auto "변경 이력을 기반으로 CHANGELOG.md 업데이트"

# GitHub Actions 예시
# .github/workflows/codex-changelog.yml
```

```yaml
name: Codex Changelog
on:
  push:
    branches: [main]

jobs:
  update-changelog:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: openai/codex-action@v1
        with:
          task: "git log의 변경 내용을 바탕으로 CHANGELOG.md의 [Unreleased] 섹션을 업데이트해줘"
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
```

### 병렬 태스크 패턴

여러 독립적인 태스크를 동시에 실행해 처리 시간을 줄인다.

```bash
# 터미널 1: 테스트 커버리지 개선
codex "테스트 커버리지가 낮은 모듈부터 단위 테스트를 추가해줘"

# 터미널 2: 문서 업데이트
codex "API 변경 사항을 README와 OpenAPI 스펙에 반영해줘"

# 터미널 3: 의존성 업그레이드
codex "안전하게 업그레이드할 수 있는 패키지를 업데이트하고 테스트를 통과시켜줘"
```

### 반복 개선 패턴

```bash
# 단계별 접근: 먼저 계획 확인 후 실행
codex "이 PR의 코드 리뷰를 해줘"
# → /plan 으로 계획 확인
# → /permissions auto-edit 로 모드 변경
# → 실행 확인
```

## 9. 워크플로우 가이드

### 신규 기능 개발 워크플로우

```
1. AGENTS.md에 해당 기능의 요구사항 추가
2. codex "feature/login-page 브랜치를 만들고 로그인 페이지를 구현해줘"
3. /review 명령으로 변경 내용 확인
4. 승인 후 PR 생성 지시
5. 사람이 PR 검토 및 머지
```

### 버그 수정 워크플로우

```
1. 버그 증상을 구체적으로 설명
   codex "로그인 후 세션이 10분 뒤 만료되는 버그. auth.ts의 JWT 만료 설정 확인"
2. /plan 으로 수정 범위 확인
3. auto-edit 모드로 수정 적용
4. /diff 로 변경 내용 검토
5. 테스트 실행 확인 후 PR 생성
```

### 코드 리뷰 워크플로우

```bash
# PR 번호 기준으로 리뷰 요청
codex "PR #142를 리뷰해줘. 특히 SQL injection 취약점과 에러 핸들링에 집중해줘"
```

### 컨텍스트 관리 팁

- 컨텍스트 윈도우가 90% 이상 차면 자동으로 압축된다. 중요한 지시는 `/new`로 세션을 초기화한 뒤 다시 명시한다.
- 복잡한 태스크는 `/plan` 명령으로 에이전트의 계획을 미리 확인하고 범위를 조정한다.
- 긴 세션에서 이전 결정으로 돌아가려면 `/resume`을 사용한다.

## 주의사항

> [!warning] 사용량 제한
> - 5시간 롤링 윈도우와 주간 할당량이 있다. `/status` 명령으로 현재 사용량을 확인한다.
> - ChatGPT 플랜 등급에 따라 할당량이 다르다 (Pro > Plus > Free).
> - 알려진 버그: 사용하지 않아도 할당량 비율이 약 1%씩 감소할 수 있다.

> [!note] Windows 지원
> Windows 네이티브 환경은 실험적 지원 단계다. 프로덕션 환경에서는 WSL2 사용을 권장한다.
