## 1. System requirements and installation

### Requirements

| Item | Minimum | Recommended |
|------|---------|-------------|
| OS | macOS 13+, Ubuntu 22.04+, Windows WSL2 | macOS 14+, Ubuntu 24.04+ |
| Node.js | 22.0 | 22 LTS |
| RAM | 2 GB | 4 GB |
| Internet | Required | — |

### Installation

```bash
# Install Codex CLI via npm (recommended)
npm install -g @openai/codex

# Verify installation
codex --version

# Homebrew (macOS — note: Computer Use feature not available)
brew install --cask codex
```

> [!warning] Watch out for the package name
> The `codex` package on npm (without a scope) is an unrelated 2012 package. Always use `@openai/codex`.

## 2. Authentication

Codex CLI supports two authentication methods.

### Method A: ChatGPT account (Plus / Pro / Team / Enterprise)

```bash
# Running codex in interactive mode opens a browser for ChatGPT login
codex
```

After login, the token is saved locally and reused on subsequent runs.

### Method B: OpenAI API key

```bash
# Set as an environment variable (recommended)
export OPENAI_API_KEY="sk-..."

# Or store in ~/.codex/config.json
```

```json
{
  "apiKey": "sk-..."
}
```

> [!tip] Security note
> Never put API keys directly in code or Git repositories. Use environment variables or a secrets manager (e.g., 1Password, AWS Secrets Manager).

### Check authentication status

```bash
# From inside a CLI session
/status
```

The `/status` command shows usage data only after the first message is sent. An empty usage section on the first run is normal.

## 3. Approval modes

Approval modes control how much autonomy the agent has over the file system and shell commands.

| Mode | Behavior | Best for |
|------|----------|---------|
| `suggest` (default) | Proposes all changes; executes nothing | First use, sensitive codebases |
| `auto-edit` | Applies file edits automatically; asks for shell command approval | Everyday development |
| `full-auto` | Applies file edits and shell commands without asking | Trusted CI/CD pipelines |

### Changing the mode

```bash
# Set mode at startup
codex --approval-mode auto-edit

# Change during a session
/permissions
```

> [!warning] full-auto caution
> In `full-auto` mode, the agent can delete files, install packages, and run shell scripts without confirmation. Only use it in environments you fully trust.

## 4. Slash commands

Control the agent during a session with these commands.

| Command | Description |
|---------|-------------|
| `/new` | Clear the current context and start a fresh session |
| `/resume` | Pick a previous session from a list and continue it |
| `/plan` | Print the agent's current execution plan in plain text |
| `/review` | Show a diff of all changed files |
| `/diff` | Include `git diff` output in the agent's context |
| `/permissions` | View or change the current approval mode |
| `/status` | Show authentication status, model in use, and usage |
| `/help` | List all available commands |

### Example

```
> /plan
The agent plans to:
1. Fix JWT validation logic in src/api/auth.ts
2. Add unit tests in tests/auth.test.ts
3. Commit the changes
```

## 5. Writing AGENTS.md

`AGENTS.md` tells the Codex agent about your project's rules, conventions, and off-limits areas. Place it in the repository root and the agent reads it automatically.

### Basic structure

```markdown
# AGENTS.md

## Project overview
This repository is a TypeScript REST API server.

## Coding conventions
- Language: TypeScript strict mode
- Formatter: Prettier (see .prettierrc)
- Linter: ESLint (see eslint.config.js)
- Tests: Vitest, maintain 80%+ coverage

## Off-limits
- Do not use the `any` type
- Do not leave console.log in production code
- Do not commit directly to the `main` branch

## PR rules
- Title format: use `feat:` / `fix:` / `refactor:` prefix
- Body must explain the reason for the change and how to test it

## Directories to ignore
- `legacy/` (scheduled for migration)
- `*.bak` files
```

### AGENTS.md lookup priority

1. `AGENTS.md` in the current working directory
2. `AGENTS.md` in the repository root
3. `~/.codex/AGENTS.md` (global defaults)

> [!tip] Effective use of AGENTS.md
> Add patterns where the agent has made mistakes in the past to the "Off-limits" section. Explicit rules reduce repeated errors.

## 6. Config file

Use `~/.codex/config.json` (global) or `.codex/config.json` (per project) to set defaults.

```json
{
  "model": "codex-1",
  "approvalMode": "auto-edit",
  "notify": true,
  "contextWindow": {
    "compressionThreshold": 0.85
  },
  "env": {
    "EDITOR": "vim"
  }
}
```

### Key configuration options

| Option | Default | Description |
|--------|---------|-------------|
| `model` | `codex-1` | Model to use |
| `approvalMode` | `suggest` | Default approval mode |
| `notify` | `false` | Desktop notification on task completion |
| `contextWindow.compressionThreshold` | `0.90` | Fraction of context window that triggers compression |

## 7. MCP integration

MCP (Model Context Protocol) lets you connect external tools to Codex.

### MCP server configuration

Add to `.codex/config.json` or `~/.codex/config.json`:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/workspace"]
    },
    "database": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "${DATABASE_URL}"
      }
    }
  }
}
```

### Example MCP servers

| Server | Package | Purpose |
|--------|---------|---------|
| GitHub | `@modelcontextprotocol/server-github` | Issues, PRs, code search |
| Filesystem | `@modelcontextprotocol/server-filesystem` | Access a specific directory |
| PostgreSQL | `@modelcontextprotocol/server-postgres` | DB schema and queries |
| Brave Search | `@modelcontextprotocol/server-brave-search` | Web search |

## 8. Automation patterns

### Non-interactive mode (CI/CD)

```bash
# Run a single task automatically (no approval prompts)
codex exec --full-auto "Update CHANGELOG.md based on the git log since the last tag"
```

```yaml
# .github/workflows/codex-changelog.yml
name: Codex Changelog
on:
  push:
    branches: [main]

jobs:
  update-changelog:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: openai/codex-action@v1
        with:
          task: "Update the [Unreleased] section of CHANGELOG.md based on git log"
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
```

### Parallel task pattern

Run multiple independent tasks at the same time to reduce wall-clock time.

```bash
# Terminal 1: Improve test coverage
codex "Add unit tests to the modules with the lowest coverage"

# Terminal 2: Update documentation
codex "Reflect the API changes in README and the OpenAPI spec"

# Terminal 3: Upgrade dependencies
codex "Upgrade packages that can be safely updated and make tests pass"
```

### Iterative refinement pattern

```bash
# Check the plan before committing to execution
codex "Review the code in this PR"
# → /plan   — review the plan
# → /permissions auto-edit   — switch mode
# → confirm execution
```

## 9. Workflow guide

### New feature development

```
1. Add the feature requirements to AGENTS.md
2. codex "Create a branch feature/login-page and implement the login page"
3. /review to inspect changes
4. Approve, then ask the agent to open a PR
5. Human reviews and merges the PR
```

### Bug fix workflow

```
1. Describe the bug precisely
   codex "Session expires after 10 minutes post-login. Check JWT expiry config in auth.ts"
2. /plan to confirm the fix scope
3. Apply changes in auto-edit mode
4. /diff to review changes
5. Confirm tests pass, then open a PR
```

### Code review workflow

```bash
# Request a review by PR number
codex "Review PR #142 — focus on SQL injection vulnerabilities and error handling"
```

### Context management tips

- When the context window exceeds 90%, compression happens automatically. Re-state critical instructions after a `/new` session reset.
- For complex tasks, use `/plan` to preview the agent's plan and adjust scope before it starts working.
- To return to a previous decision point in a long session, use `/resume`.

## Known issues and limits

> [!warning] Usage limits
> - A 5-hour rolling window and a weekly quota apply. Check current usage with `/status`.
> - Quota varies by ChatGPT plan tier (Pro > Plus > Free).
> - Known bug: quota percentage may decrease by ~1% even without active usage.

> [!note] Windows support
> Native Windows support is experimental. WSL2 is recommended for production use.
